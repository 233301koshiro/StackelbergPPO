# 週次進捗報告 2026-07-03

## 研究の全体像（おさらい）

**やりたいこと**: 手書きスケッチ → AI が 3D メッシュ生成 → Stackelberg PPO でロボットの形態と制御を同時最適化

**Level 0 目標（今週の作業対象）**: まず「2軸の固定アームが cube（箱）を押す」という単純なタスクを Choreonoid 物理シミュレーター上で学習させる

---

## 今週やったこと

### 1. 「アームがそもそも動いていなかった」問題の解決

学習が進まない原因を探るため、アームや cube の状態を毎ステップ数値で記録する診断ログを追加して調査した。

**発見①：物理エンジン（Choreonoid）のバグ**

関節の「減衰（ブレーキ）」と「可動域の制限」が設定しても完全に無視されていた。その結果、肩関節が **400ラジアン（65回転）** まで回り続けるという異常な状態になっていた。これを修正したことで、初めてアームが正常に制御された動きをするようになった。

> このバグは最初は「ポリシーが悪い戦略を学習した」と見えるため、発見が遅れた。以降は「学習がおかしい → まず物理エンジン側のバグを疑う」という調査手順を確立した。

**発見②：初期姿勢の設定ミス（arm_safe_init）**

アームが床に沈まないよう「エピソード開始時に肩関節を 90° に初期化する」という設定が入っていた。肩の可動域は −90° 〜 +90° であり、90° はその上限に当たる。

物理的には上限から下方向（0° 方向）へ動かすことはできる。しかし強化学習のポリシーは「cube に向かってアームを伸ばす = 角度を増やす」という方向に勾配が向くため、上限に当たってブロックされ続ける。「一度引いてから逆方向に展開する」という迂回戦略は報酬シグナルがなく学習されないため、結果として毎ステップ 90° に押し当てたまま動かない状態になっていた。この設定を廃止し、中立姿勢（0°）からのスタートに変更した。

---

### 2. 「cube が飛んでいく」問題の発見と対策

学習が進むにつれて、形態最適化フェーズでアーム先端が cube にめり込んだ状態でエピソードが始まるケースが発生した。この状態では、リセット時の物理的な衝撃で cube が吹き飛び、**アームを一切動かさなくてもスコアが出る**という意図しない学習（exploit）が起きていた。

対策として、エピソード開始時にアーム先端と cube の位置関係を確認し、接触が検出された場合はそのエピソードを無効化する機能を実装した。

---

### 3. 元リポジトリとの設計ミスマッチの発見

学習が進まない根本的な原因を探るため、元の Stackelberg PPO リポジトリの Pusher タスクの実装を詳細に調べた。

**元リポジトリの前提**
- ロボット：**4本脚の移動ロボット（Crawler）** ← 体全体が自由に動ける
- cube のサイズ：**1辺 1.0m（巨大）**
- 特徴：1歩歩くだけで報酬が得られるため、探索が自然に起きる

**我々の設定との差異**

| | 元リポジトリ | 我々の設定（修正前） |
|---|---|---|
| ロボットの種類 | 体ごと移動できる | **根本が固定**されたアーム |
| cube サイズ | 1辺 1.0m | 1辺 0.3m（小さい） |
| 報酬の勾配 | 常に得られる | アームが届いて初めて発生 |

移動ロボットは「とにかく動けば報酬が得られる」ため探索が自然に起きるが、固定アームは「cube に当たらなければ報酬ゼロ」のため探索の勾配がほぼ存在しなかった。cube が小さいほどこの問題は深刻になる。

**対応**：cube を元リポジトリ準拠の **1辺 1.0m** に変更し、速度ベース報酬（Δ位置 / 時間）に切り替えた。

---

### 4. 対照実験：co-design の効果検証（現在進行中）

cube サイズと報酬の修正後、2つの実験を並列実行している。

**実験 B（形態＋制御を同時最適化）**：Stackelberg PPO 本来の設定

**実験 C（制御のみ、形態は初期値で固定）**：co-design の効果を分離するための対照

| | 実験 B（co-design あり） | 実験 C（制御のみ） |
|---|---|---|
| ep 0 | fwd_cube = 0.006 | fwd_cube = 0.001 |
| ep 50 付近 | **fwd_cube = 0.105** ✅ | fwd_cube = 0.007 |

`fwd_cube` は cube が 1 ステップあたり動く距離（速度）の指標。

B は ep50 で設定した成功基準（`fwd_cube > 0.1`）を突破。**cube が実際に動いていることが数値で確認された**。C は低空飛行のまま横ばい傾向。

> この差は「形態最適化（co-design）があることで制御学習の足場が作られる」ことを示唆している。

---

### 5. 実験管理の改善：早期打ち切り基準の整備

従来は「元リポジトリが max 1000 epoch なので 500 epoch で動かなければ打ち切り」という経験則だったが、タスクの難易度が異なるためこの基準は不適切だった。

今後は実験ごとに以下を事前に決める：
- **成功基準**（この値を超えたら目標達成）
- **打ち切り基準**（この状態が続いたら設定に問題があると判断）
- **実験の役割**（対照実験なら「横ばいを確認できた」時点で目的達成）

打ち切りの判断は「絶対値が低い」だけでなく「上昇の傾きがなくなったか」で行う。

## 追記：可視化バグの発見と修正、戦略の特定

### 可視化バグ（Bug 7）

動画でアームが全ステップ静止して見えるバグを発見・修正。

**原因**：描画コードが Choreonoid の「参照ボディ」（静的コピー）を読んでおり、シミュレーション実行中に更新されていなかった。  
**修正**：シミュレーション live データ（`env._body_xpos`）に切り替え。

### 実験 B の戦略分析（修正後の動画で確認）

修正後の動画から、B が学習した戦略を特定した：

| Step | q0 | cube_vx | 状態 |
|------|-----|---------|------|
| 0（実行直前）| 0.027 | 0 | アーム・cube 静止 |
| 1 | 0.258 | 0 | 回転開始、接触なし |
| 2 | 0.956 | 10.5 | 前腕スイング → cube に接触 |
| 3 | **1.571（π/2）** | 15.5 | **joint limit に到達 → 停止** |
| 4〜200 | 1.571 固定 | 減衰 | アーム静止、cube が惰性で飛翔 |

**「スイング押し」戦略**：肩関節を 3 ステップで可動域上限まで振り切り、前腕で cube を打撃する。接触後はアームが完全に止まる。fwd_cube > 0.1 の成功基準は達成しているが、継続的な押し動作ではない。

### co-design が最適化した形態

- 上腕リンクを 0.3m → 0.7m に伸長（スイング時の接触面積を拡大）
- 折り畳み形（上腕が後方、前腕が前方）：肩の π/2 回転で前腕が +x 方向に展開する

### 実験 C の結論

ep 100 時点で `fwd_cube ≈ 0.01〜0.02`（B の約 1/8）で横ばい継続。**形態最適化なしでは制御学習が進まない**ことを確認。早期打ち切り基準に照らし、実験終了を判断する時期。

## その後の展開（報告後に判明した内容）

### 実験 D の結果（ep66 で停止）

velocity cap（3.0 rad/s）により、バットタクティックのスピードは「3ステップで振り切り」→「10ステップかけてゆっくり倒れる」に変化した。  
しかし **「1回押して止まる」パターンは解消されなかった**。

根本原因は2つ：
1. **幾何学的制約**：1回の押しで cube が x=4.23m まで飛ぶ。アームの最大リーチ 1.41m を大きく超えるため、物理的に再接触不可能
2. **報酬勾配の問題**：速度報酬（Δx/dt）では「どこで止まるか」の勾配がなく、往復する理由が生まれない

### スケッチ生成アーム（tripo_arm）の環境適応

7 リンク・6 関節の tripo_arm（スケッチから AI 生成した実際の 3D メッシュアーム）を Stackelberg PPO 環境に適応させた。

- OBJ メッシュ → capsule 近似、URDF → MuJoCo XML 変換
- `max_body_depth=6`、`embedding_len=1500` の config 修正が必要だった
- ep5 で `fwd_cube=0.943`（rrbot_arm の初期より大幅に高い）を確認
- ただし `fwd_contact=nan`（分岐構造の先端 body の計算問題）が残存

### 目標座標報酬（target PBRS）への移行と sparse reward 問題

速度報酬からの脱却として target PBRS（φ(t+1)−φ(t)）を試みたが、**sparse reward 問題**に直面した。  
cube が target 方向に動いたときにしか報酬勾配が出ないため、探索序盤は学習がほぼ進まない。

これは RL における本質的なトレードオフで、完全には解消できない。対策として2案を設計・実装した：

**① 組み合わせ報酬**（実験 D3、現在学習中）  
速度報酬（探索確保） + target PBRS（目標誘導）の重み付き合算。スクラッチから学習。

**② カリキュラム転移**（実験 D4、現在学習中）  
D（速度報酬で「押す能力」習得）→ target PBRS のみに切り替え。  
D ep66 の checkpoint（arm 1.41m に成長済み）を転移元として利用。
