## TL;DR（現在地サマリー）

**研究目標**: Stackelberg PPO（形態・制御 co-design）で、固定根本2軸アームに「cube を押す」タスクを学習させる。

**✅ 2026-07-03 確認できた事実**:
- **実験 B (bigcube + 速度報酬 + co-design) が成功基準（fwd_cube > 0.1）を ep50 付近で達成**
- 実験 C (形態固定) は ep100 で横ばい → 対照実験として「co-design なしでは学習不可」を確認し停止
- 学習された戦略は「スイング押し（バットタクティック）」: 肩関節を 3 ステップで π/2 に振り切り cube を打撃、その後は joint limit で固定
- 可視化バグ（Choreonoid 参照ボディを読んでいたためアームが動かなく見えた）を修正、修正後の動画でスイング動作を確認

**現在動いている実験**:
- `single_run/rrbot_arm_dense_D6`（ep33〜）: dense 報酬（vel - 0.01×dist(cube, target)）、対照
- `single_run/rrbot_arm_clipped_D7`（ep0〜）: clipped vel + dense 0.01、本命

**完了済み実験**:
- B（ep139）: co-design で fwd_cube > 0.1・バットタクティック特定
- C（ep100）: 形態固定では学習不可確認
- D（ep66）: velocity cap で bat tactic 緩和確認・往復戦略出ず
- E（ep5）: tripo_arm 環境適応確認（fwd_cube=0.943 but fwd_contact=nan→修正済み）
- D2（ep4）: target PBRS のみ→sparse で立ち上がらず廃止
- E2（ep3）: tripo_arm + target PBRS→nan 問題継続・廃止
- D3（ep999）: ① 組み合わせ報酬 weight=1.0 → fwd_cube=0.20 だが **eval で final_x≈8-10m（target 1.5m を大幅超過）**。vel_reward が PBRS を完全支配していた
- D4（ep999）: ② カリキュラム転移 → fwd_cube=0.0003 で失敗（sparse 問題は転移でも解消されず）
- D5（ep39 で停止）: weight=5.0 でも vel_reward の 0.25% 以下にしかならず効果なし → 早期終了

**現在の課題**:
1. D6 vs D7: dense ペナルティ単独 vs clipped vel + dense、target 停止を実現できるか
2. Level 0 達成後: スケッチ → pipeline 統合へ

## 知見: penetration インパルス退化戦略の全容（2026-06-24）

## 知見: cube 初期速度ノイズ drift バグと修正（2026-06-25）

### 症状

arm_safe_init + prismatic damping 修正後も動画でアームが触れていないのに cube が x+ 方向にジリジリ進む現象が再発。

ゼロアクションテスト（v0=0）では delta=0.000000 → **AIST damping 修正は正しく機能している**。
初速 v0=0.05 m/s を与えると 15 ステップ後も delta=+0.001 m/step → **初速が残留 drift の真の原因**。

### 根本原因

`pusher.py` の `transit_execution()` がエピソードの execution フェーズ開始時に **eval/train 問わず** `reset_state(True)` を呼ぶ。

```python
# pusher.py:226 — 常に add_noise=True
def transit_execution(self):
    self.reset_state(True)
```

`add_noise=True` のとき `qvel = init_qvel + uniform(-0.1, 0.1)` が全 joint に適用される。
cube の slide joint（アクチュエータなし）にも ±0.1 m/s のノイズが乗る。

**damping=2.0, mass=2.7 → τ = m/b = 1.35 s**
→ 初速 0.1 m/s が 1 秒以上残留し、動画で明確に見えるほど drift する。

### なぜ前回の調査で見逃したか

前回のゼロアクションテスト（cube 初速 0 から実行）では drift ゼロを確認し「修正済み」と判断。
しかし実際の学習・eval では `transit_execution()` が `reset_state(True)` を呼ぶため常に初速ノイズが注入されていた。
eval モードでも `transit_execution` は `add_noise=True` で動くため動画でも drift が発生する。

### 修正

**Fix 1: cube slide joint の qvel を常にゼロに固定** (`design_opt/envs/pusher.py`)

```python
# reset_state() の末尾（set_state の直前）
cube_x_idx = self.model.nq - 2
qvel[cube_x_idx]     = 0.0  # cube_slide (x)
qvel[cube_x_idx + 1] = 0.0  # cube_slide2 (y)
```

cube はアクチュエータを持たないため qvel ノイズは探索に寄与しない。常に静止状態でスタートするのが物理的に正しい。

**Fix 2: cube damping 2.0 → 10.0** (`assets/mujoco_envs/rrbot_arm.xml`)

τ = 1.35s → 0.27s に短縮。アームが cube を押して離れた後の停止も速くなる。

### 検証結果

```
damping=10, v0=0   → delta=+0.000000  ✅ 完全ゼロ
damping=10, v0=0.1 → 10 step で +0.001 m/step まで急速に減衰（旧 +0.001 に 15+ step 要していた）
```

## 知見: 初期姿勢貫通（penetration impulse）と対策（2026-06-25）

### 症状

形態最適化でアームが伸びるにつれ、**エピソード開始直後（joints=initial の初期姿勢）にアームリンクが cube の中に入り込む**。
物理エンジンが分離インパルスを発射し、アームが何もしなくても cube が吹っ飛ぶ。
epoch 50 の v5 で gap=0.051m（あと 5cm で初期姿勢が cube に到達）と判明。

### 根本原因

joints=0 の初期姿勢でアームが +x 方向（cube 方向）を向いている。
アームが長くなるほど arm_tip の x 座標が増え、やがて cube 左面 (x=1.35m) に到達する。

- v2 (cube_x_offset=0): arm tip gap ≈ 0 で既に退化発生（前腕が刺さって cube 飛翔）
- v2-2 / v5 (cube_x_offset=0.5): cube が x=1.5m に移動したため猶予が増えたが epoch 50 時点で v5 のアームが 5cm まで接近

### 対策（案1採用）: 初期姿勢を安全な向きに固定

エピソードリセット時に shoulder を π/2（+y 方向）に設定する。

- cube は +x 方向 (x=1.5m) にある
- 初期姿勢を +y 向きにすれば、アームがいくら伸びても +y に伸びるだけ → cube と干渉しない
- ポリシーはエピソード中にアームを cube 方向へ swing してから押す動作を学習する

**変更ファイル:**

| ファイル | 変更内容 |
|---|---|
| `assets/mujoco_envs/rrbot_arm.xml` | shoulder joint range: `-60 60` → `-90 90`（π/2 が range 内に収まるよう拡張） |
| `design_opt/envs/pusher.py` | `reset_state` に `arm_safe_init` オプション追加。`True` のとき `qpos[0] = π/2` を設定 |

**学習コマンドへの追加:**
```
+env_specs.arm_safe_init=true
```

### 採用しなかった案

| 案 | 理由 |
|---|---|
| アーム長の上限設定（案3） | 最適化の探索空間を制約してしまう |
| penetration チェック + ペナルティ | 毎ステップのチェックが重く、学習シグナルとしても弱い |

## 知見: cube drift 物理バグと修正（2026-06-25）

### 症状

v2-2・v5 どちらも「ロボットが静止したまま cube がジリジリ x+ 方向に進む」動画が確認された（epoch 50・317 の両方）。

ゼロアクションテスト結果:
- **delta = +0.0009 m/step（一定）**で 200 ステップ連続 → 定速 drift
- v2-2・v5 で完全に同一パターン → 報酬や学習と無関係な**物理バグ**
- v5（目標報酬）は epoch 317 時点で毎エポック **train_R ≈ 0.8 に収束** → drift exploitation で何もしない戦略に完全収束

### 原因1: Choreonoid AIST が prismatic joint の damping を無視

`.body` ファイルへの `joint_damping: 2` の書き出しは正しく行われていたが、
**Choreonoid の AIST シミュレータが prismatic（slide）ジョイントの `joint_damping` を dynamics 計算に適用しない**。

ゼロ減衰 → cube に初期速度があると永続的に drift する。

### 原因2: cube 質量の計算ミス

`mujoco_env_choreonoid.py` の `make_shape` 関数が `default_density=5.0` を全 geom に使用し、
cube の geom 属性 `density="100"` を無視していた。

- 実際: `5.0 * 8 * 0.15^3 = 0.135 kg`
- 正しい値: `100 * 8 * 0.15^3 = 2.7 kg`（約 20 倍の差）

### 修正 (`khrylib/rl/envs/common/mujoco_env_choreonoid.py`)

**Fix 1: step() で明示的に粘性減衰を適用（キャッシュ版）**

内側ループで Choreonoid Python/C++ API を呼ぶと **19× スローダウン**（T_sample 16s → 306s）が発生した。
そのため `_build_sim_body_entries` で `(joint_obj, decay_factor)` ペアをキャッシュし、step() 内は純粋な積算のみ。

```python
# _build_sim_body_entries: 事前計算
decay = []
for entry_idx, (sb, _) in enumerate(entries[1:], start=1):
    body_key = f'extra_{entry_idx - 1}'
    eb = sb.body()
    total_m = max(1e-6, sum(eb.link(k).m for k in range(eb.numLinks)))
    for j_idx in range(eb.numJoints):
        j = eb.joint(j_idx)
        dq = self._extra_joint_damping.get((body_key, j.jointName), 0.0)
        if dq > 0:
            decay.append((j, math.exp(-dq / total_m * self._timestep)))
self._extra_joint_decay = decay

# step(): 内側ループは j.dq *= factor のみ（API 呼び出しなし）
for _ in range(n_frames):
    self.sim_item.tickRequest(True)
    IU.processEvent()
    for j, factor in self._extra_joint_decay:
        j.dq *= factor
```

> **注意**: Choreonoid 2.3 Python バインディングは `j.Dq`（減衰係数取得）を公開していない（AttributeError）。
> そのため fallback は `j.Dq` ではなく `0.0` とし、YAML から読んだ値のみを使う（Fix 2）。

**Fix 2: damping 値を YAML からパースして保存**

`_load_body_defs` で extra body の YAML から `joint_name` + `joint_damping` を regex 抽出し
`self._extra_joint_damping[(body_key, jname)] = dq` に保存。

**Fix 3: per-geom density を読む**

```python
geom_density = float(geom_el.get('density', default_density))
```

capsule / sphere / box すべてで geom 属性の density を優先。cube は正しく 2.7 kg に。

### 影響範囲

- robot body (extra でない first body) は影響なし。explicit damping は `_sim_body_entries[1:]` のみ。
- robot geom の density は `<default><geom density="5.0"/>` → per-geom density なし → 変更なし。

## 知見: 野球バット戦術と報酬関数の問題（2026-06-23）

### 何がわかったか

`eval_cnoid_visual.py` でアーム動作を可視化したところ、rrbot アームが**野球バット型**の戦術を学習していることが判明した。

**観察**: アームを振り上げて cube に一撃を与え、cube が高速で飛んでいく。「持続的に押す」動作ではない。

**根本原因 (現在の報酬関数)**:
```python
reward_fwd_cube = Δx / dt  # cube の x 方向速度に比例
```
速度ベースの報酬は「cube を最大速度で吹き飛ばすこと」を最適化する。固定根本 2 関節アームでは、先端速度を最大化する戦略は「エルボー ≈ 0°（直線伸ばし）でショルダーを高速回転させる」こと。これは野球バットのスイングと同じ運動学。

**さらに、形態最適化も絡む**:  
`reward_fwd_contact = 1/(1 + ||cube - arm_tip||)` により、アームリンク長が cube に届くよう伸長される。結果として「長くなったアームをバットのように振る」挙動が強化される。

**元のリポジトリでバット戦術が出なかった理由**:  
元実装はモバイルベース（自由関節付き）ロボット。アクションペナルティ + ベース移動コストにより「ゆっくり前進して押し続ける」戦術がバット戦術より高報酬になる。固定根本アームにはその誘因がない。

### どう対処するか

`design_opt/envs/pusher.py` に **目標座標ベース報酬**を追加（`use_target_reward: true` で有効化）:

```python
# 新報酬: cube が target_pos に近いほど高報酬
reward_fwd_cube = 1.0 / (1.0 + dist(cube, [target_x, target_y]))
```

- バット戦術で cube を目標を超えて飛ばすと距離が大きくなり報酬が下がる
- 目標手前で止まっても報酬が下がる → 目標付近に制御して維持する動作が最適
- `reward_specs.target_x`, `target_y` で目標座標を設定可能（デフォルト: x=1.5, y=0）
- `use_target_reward: false`（デフォルト）で従来の速度ベース報酬を維持（後方互換）

## 知見: 形態最適化による「アームを cube に向けるだけ」報酬ハッキングと PBRS 修正（2026-06-26）

### 症状（発見）

v2-2・v5 ともに動画では cube もアームも「微動だにしない」ように見えるが、`exec_R_eps` は上昇し続けた（v2-2 epoch 287 で 587、v5 epoch 170 で 1240）。

### 原因

`reward_fwd_contact = 1/(1 + dist(arm_tip, cube))` が **毎ステップ定数として蓄積**されるため、
Leader（形態最適化）が bone_offset を調整して初期姿勢で arm_tip を cube 付近に常駐させることで、
Follower が何も動かなくても高報酬を得られる第 3 の報酬ハッキングが発生。

静止ベースライン約 0.39/step → 1000 step で 390/ep がタダで入る構造。

### 修正済み（2026-06-26）: ポテンシャルベース報酬シェーピング（PBRS）

`reward_fwd_contact` の積算形式を**差分形式（Δφ）**に変更した。

```python
# 旧: 毎ステップ静止でも蓄積
reward_fwd_contact = 1.0 / (1.0 + dist(arm_tip, cube))

# 新: φ(t+1) - φ(t) のみ（φ は同じ関数）
reward_fwd_contact = contact_weight * (φ_curr - φ_prev)
```

- 静止 → Δφ = 0（搾取不可）
- arm_tip が cube に近づく → Δφ > 0（探索シグナル維持）
- PBRS は最適ポリシーを変えないことが数学的に証明済み（Ng et al. 1999）

### 適用後の実験結果（v2-3・v5、epoch ~728 で確認）

v2-3・v5 を PBRS(contact) 適用版で epoch 0 から再起動し動画で確認。

| run | exec_R_eps 推移 | 観察 |
|---|---|---|
| v2-3（速度報酬 + PBRS contact） | ≈ 0 で横ばい（epoch 0〜728） | arm が cube と逆方向（-x/-y）に成長。cube 変位なし |
| v5（目標座標 + PBRS contact） | ≈ 667 で横ばい（epoch 0〜728） | arm が joints=0 で最大限 cube 方向に伸長。cube 変位なし |

### PBRS は reward_fwd_contact にしか適用していない

`reward_fwd_cube`（v5 の目標座標報酬）は **変更していない**。

```python
# reward_fwd_cube (v5): 静止でも 0.667/step ← PBRS 未適用
reward_fwd_cube = 1.0 / (1.0 + dist_to_target)   # cube at 1.5m, target 2.0m → 0.667

# reward_fwd_contact: こちらだけ PBRS 適用済み
reward_fwd_contact = contact_weight * (φ_curr - φ_prev)   # 静止 = 0
```

v5 の exec_R_eps≈667 がゼロから改善しない直接原因は **`reward_fwd_cube` の静止ベースライン（0.667/step × 1000 = 667/ep）** が残っているため。v2-3 は速度報酬（Δx/dt）のため静止 = 0 だが、探索シグナルも完全ゼロで arm が誤方向に育った。

### 残存問題と今後の対策

| 問題 | 対象 | 対策候補 |
|---|---|---|
| `reward_fwd_cube` の静止ベースライン | v5 | `reward_fwd_cube` にも PBRS を適用（Δφ_cube） |
| 探索シグナル完全ゼロ | v2-3 | `alive_bonus` 追加、または初期 arm 方向を cube 向きに誘導 |
| arm_safe_init と形態方向の不一致 | v5 | joints=0 で cube 方向に育ったアームが init で +y に強制回転される |

詳細: `docs/研究応用/タスク設計と報酬関数.md` セクション 6

## 知見: PBRS(cube) 適用後も箱を押せていない — 形態が cube に到達できていない（2026-06-29）

### 何をしたか

`reward_fwd_cube` にも PBRS（Δφ_cube）を適用した修正版で v5-2 を epoch 0 から起動し、v2-3（PBRS contact のみ）と並走させて epoch 999 まで完走させた。完走後、両 run について `eval_cnoid_visual.py` で実行動画（200 step）を生成し、フレーム画像と `cube x` の数値表示で押せているか確認した。

### 結果: 両 run とも cube は完全に静止

| run | exec_R_eps (epoch 0→999) | 実行動画での cube x | 観察 |
|---|---|---|---|
| v2-3（速度報酬 + PBRS contact） | -0.13 〜 +0.02 で横ばい | step1: 1.52 → step200: 1.52（変位ゼロ） | arm は cube から離れた -x 側に位置し、届いていない |
| v5-2（目標座標 + PBRS contact + PBRS cube） | -0.23 〜 +0.08 で横ばい | step1: 1.52 → step200: 1.52（変位ゼロ） | arm tip が cube と逆方向（-x, +y）に伸びており、届いていない |

→ **PBRS 自体は静止搾取（reward_fwd_contact/cube の偽の高報酬）を正しく解消した**（v5_pbrs_contact_only の exec_R_eps≈667 のような偽の高値は再現していない）。しかし副作用として、**探索シグナルがほぼゼロになり、Leader（形態最適化）が cube に到達する方向にアームを育てられなかった**。

### 根本原因（仮説）

1. **PBRS は静止時の報酬を 0 にする** → エピソード開始時に arm が cube から離れていると、Δφ もほぼ 0 のまま変化せず、Leader 側に「cube 方向に伸ばせ」という勾配がほとんど伝わらない。
2. これは `docs/進捗.md` 2026-06-26 知見の「残存問題」で予告した「探索シグナル完全ゼロ」（v2-3）が、cube 側にも PBRS をかけたことで v5-2 にも伝播した形。
3. `arm_safe_init`（貫通 exploit 対策で導入、初期姿勢を+y方向に強制）が、形態最適化が育てた方向と噛み合っていない可能性も残る（2026-06-26 知見の「arm_safe_init と形態方向の不一致」も未解消）。

### まだ検証していないこと

- PBRS の Δφ が実際に epoch を通してどう推移したか（ログに reward_fwd_cube / reward_fwd_contact の生値が出ていないため、現状は exec_R_eps と動画からの逆算のみ）
- Leader 側の勾度がそもそも morphology 更新に乗っているか（bone_offset の学習曲線を見ていない）
