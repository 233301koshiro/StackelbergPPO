# デバッグ戦記

**目的**: 調査中に発見したバグ・不具合を記録し、次に行き詰まったときに「報酬設計や学習アルゴリズムのせい」と誤認しないための再現性ある方法論をまとめる。

---

## 方法論：原因を特定するための3ステップ

次に「armが動かない」「報酬が上がらない」で行き詰まったとき、**すぐ報酬やハイパラに手を入れる前に**この順序で確認する。

### Step 1: まず物理・環境層が正しく動いているか疑う

学習の停滞は、アルゴリズムや報酬より**物理エンジンや環境実装のバグ**が原因のことが多い。「ポリシーが悪い戦略に収束した」という説明は、物理バグとほぼ区別がつかない外見をしている。

**確認すべき仮定（チェックリスト）**:
- [ ] 物理エンジンが XML/YAML に書かれた damping を実際に適用しているか
- [ ] joint_range（可動域制限）が実際に動力学計算に反映されているか
- [ ] qpos（状態変数）が実際のリンク位置（body_xpos/body_xmat）と一致しているか
- [ ] 観測値（observation）に極端な値やNaN混入がないか（`normalize_obs` 警告を確認）
- [ ] 報酬値のログ（fwd_cube/fwd_contact）が物理的に意味のある値を示しているか

### Step 2: 2つの独立した測定値を並べて矛盾を探す

同じ物理状態を2通りの方法で計算し、**矛盾がないか**を確認する。矛盾があればどちらかが嘘をついている = バグ。

例:
- `qpos`（MuJoCo互換の状態ベクトル）と `body_xmat`（Choreonoidが計算したリンク回転行列）
- `ctrl_norm`（制御出力）と `tip_dist`（アーム先端の実際の軌跡）
- YAMLに書かれた `joint_range` と 実際のqposの取りうる値

### Step 3: 制御実験で「1変数固定」して物理応答を直接確認する

- ランダムポリシーや複雑な報酬ではなく、**自分で決めた単純な入力**（例：既知のトルクパルスを数ステップだけ与えて、その後ゼロにする）を与える
- 物理的な**予測**（減衰があれば速度は指数的に減るはず、可動域があれば90°で止まるはず）と**観測**を比較する
- 予測と観測が一致しなければ実装バグ

### Bug 1: Choreonoid AIST が prismatic joint の damping を無視する

**発見日**: 2026-06-25  
**影響**: cube が damping=2.0 と設定されているにも関わらず、ゼロ減衰でドリフトし続ける。動画で「アームが触れていないのに cube がジリジリ進む」現象として顕在化。

#### 症状
- ゼロアクションテスト（ctrl=0）でも cube が `delta = +0.0009 m/step` で一定速度 drift し続ける
- v2-2・v5 でまったく同一パターン → 報酬や学習と無関係な物理バグと判断

#### 根本原因
Choreonoid の AIST シミュレータが `.body` ファイルの `joint_damping` を **prismatic（slide）ジョイントに対して dynamics 計算に適用しない**。設定値は読み込まれるが無視される。加えて cube の質量を計算する `make_shape` 関数が per-geom の `density` 属性を無視しており、`0.135 kg`（正しくは `2.7 kg`）になっていた。

#### 発見方法
- `eval_cnoid_visual.py` で動画を生成し「アームが静止しているのに cube が動く」を目視確認
- ゼロアクションテストスクリプトで ctrl=0 のまま cube が drift することを数値で確認 → 学習結果でなく物理バグと確定

#### 修正内容
`khrylib/rl/envs/common/mujoco_env_choreonoid.py` の `step()` 内で、毎物理ティックごとに **明示的に粘性減衰を `j.dq *= decay_factor` として適用**するキャッシュ方式を導入（`_extra_joint_decay` リスト）。API 呼び出しコストを避けるため事前にキャッシュ。cube damping を 2.0 → 10.0 に変更。

#### 教訓
「設定ファイルに書いた値が物理エンジンに反映されている」という前提を疑わなかった。**YAML/XMLに値を書くだけでは不十分で、エンジンが実際に使っているか確認が必要。**

### Bug 2: 同修正が「extra body（cube）のみ」に適用され、ロボット本体の hinge joint が対象外だった

**発見日**: 2026-07-01  
**影響**: ロボット自身の関節（肩・肘）が damping・joint_range ともに無視され、**一度でもトルクをかけると関節が永久に無制限回転し続ける**。これまでの全実験（v2〜v5-4）がこの壊れた物理の上で行われていた。`[normalize_obs] extreme-value rows clamped` 警告がずっと出続けていた原因でもある。

#### 症状
- epoch387まで学習しても cube が一度も動かない（全 run で変位ゼロ）
- 「ポリシーが止まる戦略を学習した」という解釈をしていたが、物理バグだった可能性が高い
- バグ修正後、初めてアームが制御された動き（ctrl 非ゼロが200ステップ継続、tip_dist が単調に減少）をするようになった

#### 根本原因

**3層の問題が重なっていた**:

1. **`mujoco_xml_to_body()` で hinge joint に damping が渡されていなかった**  
   prismatic 分岐は `damping=` を明示的に渡していたが、hinge 分岐は省略していた。そのため YAML に `joint_damping` フィールドが書き出されなかった。
   ```python
   # 修正前（hinge）
   add_link(..., axis, rng_deg, ..., shape)  # damping 省略
   # 修正後
   damping = float(j_el.get('damping', default_damping))
   add_link(..., axis, rng_deg, ..., shape, damping=damping)
   ```

2. **Bug 1 の修正が `entries[1:]`（extra body = cube のみ）に限定されていた**  
   `_build_sim_body_entries()` の減衰キャッシュ構築ループが `entries[1:]` だったため、`entries[0]`（ロボット本体）のアーム関節には一切適用されていなかった。

3. **`joint_range` をクランプする処理が存在しなかった**  
   YAML に `joint_range: [-90, 90]` は書き出されていたが、それを読んでシミュレーション中に強制する処理がなかった。AIST は revolute joint の range を自力では強制しない。

#### 発見方法

**起点：ユーザーの一言「動いているようには見えないけど動いているってこと？」**

3フレームサンプリング（step1/100/200）では視覚的に静止して見えていたが、この疑問から細かく再調査した。

**Step 1（粒度を上げる）**: 毎5ステップのdense diagnosticを実施
```
step  20: qpos1=0.209  tip_dist=0.83633  ctrl1≈0.00000
step 100: qpos1=1.630  tip_dist=0.83633  ctrl1≈0.00001  ← 矛盾！
step 195: qpos1=3.317  tip_dist=0.83633
```
`qpos1` は 3ラジアン以上変化しているのに `tip_dist` が小数点5桁以下まで完全に固定。**2つの独立測定値が矛盾**している。

**Step 2（控制実験）**: 環境を直接構築し、肩関節に8ステップだけ既知のトルクパルスを与えて zero ctrl に戻す制御実験を実施
```
step  7: ctrl=1.00  qpos0=7.96  qvel0=47.86  real_angle=1.678
step  8: ctrl=0.00  qpos0=9.88  qvel0=47.86  ← qvel が一切減らない！
step215: ctrl=0.00  qpos0=406.2 qvel0=47.86  ← 65回転以上
```
- `qvel` が ctrl=0 にしても 207ステップ全く減衰しない → **damping 無効の直接証明**
- `qpos0` が 406 rad（65 回転以上）→ **joint_range 無効の直接証明**
- `real_angle`（body_xmat から直接計算）が `qpos0` と完全一致 → 表示バグではなく実際に物理的に回転し続けている

#### 修正内容
`mujoco_env_choreonoid.py` の3箇所を修正:
1. hinge joint にも damping を渡すよう `process_one_joint()` を修正
2. `_load_body_defs()` の解析ループから `if i > 0:` 制限を撤廃し、ロボット本体も含む全 body の `joint_damping` と `joint_range` を解析
3. `step()` 内で毎ティック、全 joint に対して減衰（`j.dq *= factor`）と range クランプ（`j.q` を `[lower, upper]` に制限、超えた場合は `j.dq=0`、`calcForwardKinematics()` で実リンク姿勢を同期）を適用

詳細: タスク設計と報酬関数.md セクション11

#### 教訓
1. **Bug 1 の修正を入れたとき、「同じ問題が別の箇所にないか」を確認しなかった**。今後同種の修正を加えるときは、同じパターンが他の場所でも発生していないかを必ずスコープ確認する
2. **「ポリシーが最適戦略として止まることを学習した」はデフォルト仮説にしない**。Stop は報酬設計問題にも見えるし物理バグにも見える
3. **制御実験（1変数固定）が最も確実**。ランダムポリシーや複雑な報酬ではなく、自分で決めた単純な入力を与えて物理応答を直接確認する

### Bug 3: Choreonoid exit 後もプロセスが残り続ける（choreonoid Qt イベントループ問題）

**発見日**: 2026-06-30  
**影響**: `eval_cnoid_visual.py` / `choreonoid_train.py` など `choreonoid --python` 経由で動かすスクリプトが、スクリプト終了後も Qt のイベントループが残りプロセスが終了しない。`timeout` コマンドの SIGTERM も効かず、最大3日間プロセスが生き続けてメモリをリークしていた。

#### 症状
- 動画生成後 `[visual] Done.` が出てもプロセスが終了しない
- `timeout 300` を設定しても Kill されない
- 複数のゾンビ的プロセスが蓄積し、メモリが枯渇

#### 発見方法
定期的にメモリをチェックしていて `free -h` で空き容量が急減 → `ps` で漏れているプロセスを特定 → `eval_cross_env.py` に同問題が既に存在しており `os._exit(0)` で解決済みだったことを確認

#### 修正内容
`eval_cnoid_visual.py` / `eval_cnoid_numerical.py` / `eval_cnoid_viewer.py` / `choreonoid_train.py` の末尾に `os._exit(0)` を追加

```python
# スクリプト末尾に追加
os._exit(0)  # choreonoid では sys.exit が TypeError になるため強制終了
```

#### 教訓
`choreonoid --python` 経由で動かすスクリプトは、すべて末尾に `os._exit(0)` が必要。**新しい choreonoid 用スクリプトを作るときは最初から入れる**。

### Bug 4: `get_body_com()` がサブツリー全体の重心を返す（個別リンクの重心ではない）

**発見日**: 2026-06-29  
**影響**: `compute_reach_bonus()` でリンク長を `||get_body_com(bodies[1]) - get_body_com(bodies[0])||` で計算しようとしたところ、`bodies[0]`（初期形態ではほぼ質量ゼロのルートリンク）と `bodies[1]` のサブツリーCOMがほぼ一致し、`L1` が常に 0 になった。

#### 発見方法
デバッグ出力で `L1=0.000` が全サンプルで一致 → 計算式の前提が間違いであることに気づく

#### 修正内容
リンク長は `body.bone_offset`（設計パラメータのベクトル、ノルムが長さ）を直接読む。ショルダーのワールド座標は `data.body_xpos[body_name]`（フレーム原点、サブツリーCOMではない）を使う。

#### 教訓
MuJoCo スタイルの `get_body_com` は「そのボディを根とするサブツリー全体の重心」を返す。個別リンクの情報が欲しいときは `body.bone_offset` または `data.body_xpos`（フレーム原点）を使う。

### Bug 5: config.py の `reward_specs` / `env_specs` が Hydra FLAG を無視する

**発見日**: 2026-06-24  
**影響**: `+reward_specs.use_target_reward=true` などコマンドライン引数で渡した設定が学習中に無視され、デフォルト値で動いていた。`+env_specs.cube_x_offset=0.5` も効いておらず、v2-2/v5 が意図しない設定で数十エポック学習していた。

#### 発見方法
`.hydra/config.yaml` に `cube_x_offset: 0.5` が記録されているのに、`check_strategy.py` で `cube 初期: x=1.000m (offset=0.0)` と表示 → 設定と実際の挙動が一致しないことで発覚

#### 修正内容
`design_opt/utils/config.py` に `self.env_specs.update(FLAG.get('env_specs', dict()) or {})` を追加（`reward_specs` は既に修正済みだったが `env_specs` が漏れていた）

#### 教訓
`.hydra/config.yaml` に記録されていても、コード側で読んでいなければ意味がない。**設定が実際に使われているかは `check_strategy.py` 等で実機確認が必要**。

### Bug 6: 学習済み形態が cube に初期接触してインパルスで cube を飛ばす（initial contact exploit）

**発見日**: 2026-07-01
**影響**: curriculum_v1（cube_x_offset=0.1）で epoch11 に fwd_cube が初めて非ゼロになった。しかし動画でアームが動いていない、ステップごとのデータでアームの link.p が全く変化しないことを確認。ゼロアクション×学習済み形態でも cube が動くことが判明。

#### 症状
- `fwd_cube > 0` にもかかわらずアームが視覚的に静止
- ステップ0から `cube_vx = 0.266 m/s` が発生（アクションより先に速度がある）
- `qpos0 = 1.571 = π/2` が全ステップで不変（shoulder が joint_range 上限に張り付き）
- `elbow_x = -0.4934`, `tip_x = 0.0047` が全ステップで不変（アームは1mmも動いていない）

#### 根本原因
Leader（形態設計）が reach_bonus の最大化として「arm_safe_init 補正後の先端が cube の左面（x≈0.97m）にぴったり触れる形態」を学習。エピソード開始直後に物理エンジンが接触解消インパルスを発生 → cube が飛ぶ。アームは能動的に何もしなくていい。

arm_safe_init は「joints=0 の時のアーム方向が cube を向かないよう +90° 回転」で初期貫通を防ぐ設計だったが、**形態の成長（bone_offset の方向・長さが変わる）によって +90° 回転後の先端位置が再び cube に当たる形態を学習できてしまった**。

#### 発見方法（デバッグ戦記の方法論に従い）

**Step 1（矛盾を探す）**: fwd_cube 非ゼロなのに動画でアームが静止 → qpos と link.p の両方を記録する dense diagnostic → elbow_x, tip_x が全ステップ不変（qpos1 は変化しているのに物理的には動いていない）

**Step 2（対照実験）**: ゼロアクション × デフォルト形態 → cube 静止（Bug ではない）。ゼロアクション × 学習済み形態 → `cube_vx = 0.266 m/s` 発生（形態依存の initial contact と確定）

#### 修正内容
`transit_execution()` の `reset_state()` 直後に `_check_initial_contact()` を呼び出し。arm tip の world 座標（arm_safe_init +90° 補正後）が cube のバウンディングボックス内に入る場合はエピソードを即終了（`return False`）。Leader は再設計を余儀なくされ、このexploit形態を学習できなくなる。`+env_specs.check_init_contact=false` で無効化可能。

#### 教訓
- **「cube が動いた」は「アームが動いた」ではない** — step-by-step の dense diagnostic でアームの link.p（世界座標）を確認しないと騙される
- **reach_bonus の最大化と initial contact exploit は表裏一体** — Leader に「arm をcube に近づけよ」というインセンティブを与えると、「そもそもエピソード開始時点で触れている」という搾取解が最適になりうる
- **ゼロアクション対照実験の対象は「デフォルト形態」だけでなく「学習済み形態」でも行うべき**

## 早期発見のためのシグナル一覧

以下が観測されたとき、報酬設計より先に物理・環境実装のバグを疑う：

| 観測シグナル | 疑うべき原因 |
|---|---|
| `[normalize_obs] extreme-value rows clamped` が多発 | qpos などの状態変数が異常値になっている |
| cube や arm が「触れていないのに動く」「触れているのに動かない」 | 物理エンジンの damping/接触計算がおかしい |
| ctrl=0 にしても関節速度が全く変わらない | joint damping が適用されていない |
| qpos と body_xpos/body_xmat から計算した値が一致しない | 状態変数と実際の物理状態がデシンクしている |
| 設定ファイルの値と `check_strategy.py` 等の表示値が一致しない | config の読み込みバグ |
| `timeout N` が効かずプロセスが生き続ける | choreonoid Qt ループ問題（`os._exit(0)` 不足） |
| 学習曲線がある時点から全く動かなくなる（ゼロ横ばい） | 上記いずれかのバグが原因で観測値が破損している可能性 |

## 対照実験ルート（cube が動かない原因調査、2026-07-01）

### 背景

物理バグ（Bug 2）修正後も cube 変位ゼロが続いていた。「報酬設計の問題か、別のバグか」を切り分けるため、3つの対照実験ルートを設計した。

**原則**: 確定している事実まで戻り、一変数ずつ検証する。

---

### Route C: ゼロアクション健全性チェック（最小コスト、即実施）

**目的**: ポリシー・報酬と完全に切り離して物理層のみを確認する。ctrl=0 で 200ステップ流し、アームと cube が静止するかを見る。

**手順**:
1. `PusherEnv` を直接構築（fix_skeleton=true, cube_x_offset=0.5, arm_safe_init=true）
2. 設計フェーズをゼロアクションで通過して execution ステージへ
3. 200ステップ ctrl=0 で実行し、qpos/qvel/cube_x/cube_vx を毎ステップ記録

**2026-07-01 実施結果（Bug 2 修正後）**:

```
step  0: qvel1 = -0.006722  （リセット時の初期ノイズ）
step  1: qvel1 = -0.002650  → 指数減衰
step  4: qvel1 = -0.000162
step 20: qvel1 = -0.000000  ← 完全ゼロ。以後200stepまで静止
cube_vx = 0.000000 （全ステップ通してゼロ）
```

**判定: ✅ PASS（物理層は正常）**

肘関節の初期 qvel が非ゼロに見えたが、これはリセット時の速度ノイズが正しく指数減衰している正常な動作（damping 修正が機能している証拠）。cube は完全静止。

**この結果が意味すること**: 物理エンジンに残存バグはない。cube が動かない原因は**報酬設計か学習ダイナミクス**（物理エンジンではない）に絞られた。

---

### Route A: 現状継続（フル装備 + 物理修正）【結論済み: 失敗】

**設定**: v5-5 / rrbot_arm2_cnoid_v2（2026-07-01 起動、同日停止）

**結果**: 探索ギャップ問題（arm_safe_init 構造的欠陥）で fwd_contact=0 のまま停滞 → Route B に移行。

---

### Route B: 速度ベース報酬に戻す【結論済み: bat tactic 確認 → B/C/D 実験へ】

**設定**: `use_target_reward=false`（速度ベース `Δx/dt`）+ 大 cube（1m）

**結果**: バットタクティック（スイング押し）が出現。fwd_cube > 0.1 の成功基準を ep50 で達成（実験 B）。ただし joint limit で 3 ステップ後に固定 → 継続押し戦略を学習させるため joint range 拡張 + 速度制限（実験 D）へ。

---

### ルート選択フローチャート（2026-07-03 時点で全ルート結論済み）

```
Route C（ゼロアクション）✅ 完了: 物理は正常
         ↓
Route A ✅ 完了: 探索ギャップ → 失敗（arm_safe_init 廃止へ）
         ↓
```

### Bug 8: pgrep/pkill の自己マッチによる監視デッドロックと誤 kill（2026-07-09、ops）

**症状**: K1/K2 完走（13:22）を検知して L2 を自動起動するはずの `L2_launcher.sh` が 2.7 時間動かず、完走監視ループも永遠に「実行中」のまま通知が来ない。

**原因**: `pgrep -f "hydra.run.dir=...K"` の `-f` はプロセスの**コマンドライン全文**にマッチする。監視ループを `bash -c '<パターンを含むスクリプト>'` として起動すると、**監視シェル自身のコマンドラインがパターンを含む**ため pgrep が自分にマッチし続け、「K がまだ生きている」と誤認し続けた。さらに別プロセス（L2_launcher）の pgrep も**この監視シェルに**マッチするため、相互に偽陽性を生む。後始末の `pkill -f "L2_launcher.sh"` も同様に**発行元シェル自身を kill**し（exit 144）、後続コマンドが実行されないまま静かに死んだ。

**確認方法**: `ps aux | grep <キーワード> | grep -v grep` でコマンドライン全文を目視し、「マッチしているのは誰か」を確認（pgrep の数字だけ見ていると自分自身と気づけない）。

**教訓**:
1. 監視・後始末で `pgrep -f` / `pkill -f` を使うときは、**パターンが自分のコマンドラインに含まれない**ことを保証する（スクリプトファイル化して `bash file.sh` で起動する／`pgrep -f "^完全な先頭一致"` を使う／`ps | grep -v grep | grep -v $$` で除外する）
2. pgrep は「pgrep 自身」は除外するが、**呼び出し元シェルは除外しない**
3. 完走検知は プロセス監視より **`training done` のログ grep** の方が自己マッチの余地がなく安全

### Bug 9: 初期接触棄却の全滅 →「All episodes are filtered」assert 死（F4・L2・TP1 で3回再現）

**症状**: 学習が序盤（ep13〜18）で突然死。stdout 末尾に `AssertionError: All episodes are filtered`（genesis_agent.py:446）と `[EP-FILTER] dropped 8336 episodes`。直前の T_sample が異常に長い（365s、通常 20s — 棄却→再サンプルの空回り）。**main 死亡後もワーカー4個/run が孤児として残り、計 ~15GB の RAM を占有し続ける**二次被害つき。

**根本原因（死のスパイラル）**:
1. `_check_initial_contact()` は初期接触形態のエピソードを「exec 0 ステップで打ち切り」にする
2. EP-FILTER は follower=0 のエピソードを**丸ごと捨てる** → **Leader にも勾配が渡らない**
3. Leader の設計分布が棄却領域（tip が cube に触れる形態）に入っても、「そこが悪い」という学習信号が存在しない → 分布が留まる/深入りする
4. 全エピソード棄却 → assert 死

**トリガー条件**: `ctrl_cost_coeff` が大きい（F4=0.1、L2/TP1=0.2）と Follower の初期学習が遅く、Leader が実タスク報酬の勾配を得る前にランダムドリフトで棄却領域に到達しやすい（ctrl コスト 1e-4 の I2 は 683ep 無事故 = 同条件で Follower が ep13 時点で既に押せていた）。

**修正（2026-07-10）**:
1. **pusher.py**: 初期接触時、棄却ではなく「物理を進めず 1 exec ステップ + `init_contact_penalty`（デフォルト 1.0）の負報酬」でエピソードを成立させる → EP-FILTER を通過し **Leader が棄却領域の悪さを勾配として学習できる**。exploit（接触インパルス）は物理を進めないため不可能のまま
2. **genesis_agent.py**: 全滅エポックは assert 死でなく更新スキップ + 警告に変更（10連続で初めて異常終了）— 多重防御

**教訓**:
1. **「データから捨てる」ガードは、捨てた領域から抜け出す勾配も捨てる**。エピソード棄却型のガードを入れるときは「Leader（またはポリシー）がその領域を避ける学習信号は何か」を必ず問うこと。ペナルティ化が原則
2. assert による即死はワーカー孤児（RAM リーク）を生む。学習ループの防御は「スキップ + 上限つき」で
3. 同じ死に方が2回目に出たら（F4→L2）偶然ではなく構造。3回目を待たない

**追記（2026-07-10 午後）: 修正第1版（penalty=1.0）自体に欠陥があった**。
ctrl_cost=0.2 下では正直に動くエピソードの序盤リターンが ≈-16 のため、**「接触して1ステップで終わる（-1）」方が正直に動くより得** — Leader は正しく最適化して全設計を接触形態に収束させた（L2/TP1 再走 ep8-10 で train_R_eps≈-1.05 に張り付き・eval=-1.00 を実測）。しかも全エピソードが同じ -1 になると advantage の対比が消えて脱出勾配も出ない。
→ **ペナルティは「正直な1エピソードの最悪リターンより確実に悪い」こと**が設計原則（懸念3「早期終了が得」の変種）。init_contact_penalty デフォルトを 50（≈ ctrl コスト満額 + マージン）に変更し再々起動。
**メタ教訓: ペナルティ・報酬の定数は「比較対象となる正直な行動の実測リターン」と並べて桁を決める**（-1 は直感的には十分な罰に見えるが、報酬系全体の中では報酬だった）。

**追記（2026-07-13 夕）: 最悪条件での設計検証に成功 — penalty=50 は「接触圏スタートの seed」でも脱出勾配として機能した**。
TP2_s2（seed=1）は初期形態が接触ペナルティ圏から開始し、ep0-2 は eval=-50.00 に張り付いた（= penalty=1 世代なら死んでいた条件）。しかし **ep3 で -17.45 → ep4 以降 -0.3〜-0.7 の正直な値に離陸**し、train_R_eps も -49.4→-30.7 と改善継続。棄却（Bug 9 原型: 勾配ゼロで assert 死）→ penalty=1（張り付き）→ penalty=50（3エポックで脱出）という3世代の対照が偶然にも同一タスクで揃い、**「棄却でなくペナルティ・大きさは正直な最悪リターン未満」の設計原則が最悪条件込みで実証された**。修論の考察 5.2.4 の実証例として使える。

### Bug 10: checkpoint 再評価時の転用フィルタ事故 — TP/K 系の形態抽出が全てアーティファクトだった（2026-07-15 発覚）

**症状**: TP2_s2 の best checkpoint を決定論再評価すると全エピソードが初期接触ペナルティ(-50)になるのに、学習ログの eval は +0.48。矛盾を追うと、プローブが再現する形態（tip=(0.84,0.24)）と学習中の実形態（tip=(0.55,1.00)）が全く別物だった。

**根因**: `load_checkpoint()` は `cfg.control_prior=true` のとき **(a) policy/value を control_ プレフィクスだけに絞って読み込み（Leader = skel_/attr_ ネットはランダム初期化のまま）、(b) obs_norm も読み込まない**（転用時の仕様としては正しい）。再評価ツールは保存済み `.hydra/config.yaml` から cfg を作るため、**転用で起動した run（TP1/TP2/TP2_s2/K1/K2）の再評価では control_prior=true が残り、この転用フィルタが自分自身の checkpoint に適用されていた**。形態はランダム Leader の出力 = アーティファクト。

**なぜ気づけたか**: 「学習ログの eval」と「checkpoint 再評価」の矛盾を放置せず突き合わせた（-50 vs +0.48）。**学習ログと再評価の値が合わないときは、まず重みの読み込みを疑う**。

**修正**: 全再評価ツール（visualize_morph_changes / compare_morphology / eval_reach_hover / probe_cube_trace / probe_L0_intervention）で cfg 構築後に `control_prior=False; morph_prior=False` を強制（コミット fabff75）。完走 run 自身の checkpoint には全重み + obs_norm が保存されているため、これが正しい読み込みである。

**無効化された測定（要再測定・再測定済みは順次訂正）**: TP1「短腕×低gear の第3クラスタ」形態・TP1 持続押しトレース・「ペナルティ境界拘束」知見・TP2/TP2_s2 中間形態・K1/K2 の形態（K2 異常収束 L11=0.868/gear371 を含む）。**無効でないもの**: 学習ログ由来の数値（exec_R_eps・離陸判定・仮説④の判定値）、prior なしで起動した run（B/C/G/H/I/J/L 系）の全形態抽出・G1 判定・J1 介入実験。

**メタ教訓**: 再評価ツールは「学習を再現する」のではなく「学習済み状態を復元する」もの。**訓練時と評価時で意味が変わる設定フラグ（転用・正規化リセット等）は、再評価パスでは明示的に無効化する**。ツールを書いた時点で prior 付き run を評価する予定がなかったため、フィルタの存在自体を忘れていた。

### 既知の挙動メモ（2026-07-10）

- **eval スクリプトの例外パスで Qt が残留する**: `--python` スクリプトが例外を投げると Choreonoid が例外を握りつぶし（"Failed to run the python script"）、`os._exit(0)` に到達しないため Qt イベントループが残ってプロセスが生き続ける（2026-06-30 の os._exit 修正は正常終了パスのみ）。`sys.excepthook` の差し替えも**効かない**（Choreonoid が先に捕まえる）。対策: eval 実行は常に `timeout -s KILL` でラップする（SIGTERM は効かない）
- **pgrep の自己マッチ回避の定石**: パターンに `[ ]` を入れる（例: `pgrep -f "eval_reach_[h]over"`）— パターン文字列自身がリテラルとして自分のコマンドラインに現れなくなる（Bug 8 の追加対策）

### 発見: バットタクティック（スイング押し）

Bug 7 修正後の動画で実際の挙動を確認。

**挙動**:
```
Step 0 (before sim): q0=0.027, cube_vx=0.000  ← アーム静止、cube 静止
Step 1:              q0=0.258, cube_vx=0.000  ← アームが動き始め、まだ接触なし
Step 2:              q0=0.956, cube_vx=10.5   ← 前腕が cube 方向にスイング → 接触
Step 3:              q0=1.571, cube_vx=15.5   ← q0 が π/2（joint limit）に到達
Step 4〜200:         q0=1.571 固定            ← joint limit で停止、cube が惰性で飛翔
```

**メカニズム**: Leader が設計した形態は上腕が左後方（bone_offset≈[-0.49,-0.50]）、前腕が右前方（[+0.50,+0.50]）という「折り畳み形」。実行フェーズで肩関節を 0 → π/2 まで一気に回転させると前腕が +x 方向にスイングして cube に当たる。その後は joint limit で止まる。

**これは exploit か？**: 論文が想定する「持続的な制御でタスクをこなす」からは外れているが、物理的に正当な相互作用（アームが動いて cube を押した）ではある。fwd_cube > 0.1 の成功基準は達成。

**次のステップ**: joint range の拡張（-90〜+90° → より広い範囲）や報酬の見直しで「joint limit に当たってから引いて再度押す」継続的な戦略を学習させる。
