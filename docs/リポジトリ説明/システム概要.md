# StackelbergPPO システム概要（大局的理解）

---

## 何をしているシステムか

**「ロボットの形を自動で進化させながら、同時に動き方も学習する」強化学習システム**。

普通のロボット制御 AI は「形は固定」で「どう動くか」だけを学ぶ。このシステムは形（骨格・リンク長・関節）も AI が自分で変えながら、最も性能の高いロボットを探す。

---

## 2つの意思決定者

システムの核心は、**2種類の「意思決定」が同時に走っている**こと。

```
┌──────────────────────────────────────────┐
│  リーダー（設計者）                         │
│  「どんな形のロボットを作るか」              │
│  → 骨格の追加/削除、リンク長・関節の調整    │
└──────────────────┬───────────────────────┘
                   │ この形を渡す
┌──────────────────▼───────────────────────┐
│  フォロワー（コントローラー）               │
│  「渡された形でどう動くか」                 │
│  → 各関節のトルク（力）を出力              │
└──────────────────────────────────────────┘
```

---

## Stackelberg（シュタッケルベルク）戦略とは

リーダーは「自分がこの形を選んだら、フォロワーはどう動くか」を**予測したうえで**形を決める。

```
普通の共設計:  形を決める → 動きを学ぶ → 形を評価する（後から）
Stackelberg:  形を決める時点で「この形なら動きはこうなる」を先読みして形を選ぶ
```

ゲーム理論でいう「リーダーがフォロワーの反応関数を知っている」状態。これにより、見た目は良いが実際には動かしにくい形を避け、コントローラーが活かせる形に収束しやすくなる。

## 1エピソードの流れ

1エピソードは**3つのフェーズ**に分かれる。

```
エピソード開始
   │
   ▼
【フェーズ1: 骨格変換】 5ステップ
   ポリシーが「ボディを追加」「削除」「そのまま」を選択
   → ロボットのツリー構造（グラフ）が変化
   → シミュレーターにリロード

   ▼
【フェーズ2: 属性変換】 1ステップ
   ポリシーがリンク長・関節軸・関節可動域を微調整
   → シミュレーターにリロード

   ▼
【フェーズ3: 実行】 最大1000ステップ（異常挙動で早期終了あり）
   ポリシーが関節トルクを出力して実際に動く
   → タスクの報酬（どれだけ遠くまで押せたか等）を獲得

   ┌─ 1ステップの内訳 ──────────────────────────────────┐
   │ ① 状態観測（関節角度・速度・キューブ位置など）         │
   │ ② ポリシーが各関節のトルクを決定                     │
   │ ③ 物理シミュを 4 tick 進める（= 仮想時間 0.04 秒）    │
   │    ※ 1 tick = 0.01 秒（pusher.xml timestep設定）     │
   │    ※ 現実の経過時間（wall clock）とは無関係           │
   │ ④ 報酬を計算して次の観測へ                           │
   └────────────────────────────────────────────────────┘

   1000ステップ = 仮想時間 40 秒分ぶっ続けてロボットが動く。
   毎ステップ（0.04秒ごと）に「今どのくらいトルクをかけるか」を再決定し続ける。

エピソード終了
```

フェーズ1・2がリーダー（形態設計）、フェーズ3がフォロワー（制御）に対応する。

## 学習ログの指標

ログ（`log/log_train.txt`）とグラフ（`scripts/plot_rewards.py`）に出てくる主な指標の意味。

| 指標 | フェーズ | 意味 |
|------|---------|------|
| `exec_R_eps` | 実行（フェーズ3）のみ | 完成したロボットが実際にタスクをこなしたときの1エピソード報酬。**本当に解けているかの直接評価**。 |
| `train_R_eps` | 全フェーズ（1〜3）平均 | 形態探索中の試行錯誤を含む学習全体の平均報酬。`exec_R_eps` より低い値になる。**学習の安定性モニタリング用**。 |
| `exec_R` | 実行フェーズのみ | `exec_R_eps` のステップ単位版（エピソード長で割る前）。 |
| `train_R` | 全フェーズ | `train_R_eps` のステップ単位版。 |

`exec_R_eps` が重要な理由: フェーズ1・2（骨格探索）の報酬は設計途中のランダムな状態を含むため、タスク達成度の指標としては雑音が多い。`exec_R_eps` は設計が確定した後の純粋な実行報酬のみを集計するため、「このエポックでロボットがどれだけ課題を解けたか」を正確に反映する。

## 学習ループの全体像

```
┌─────────────────────────────────────────────────────┐
│                    1エポック                         │
│                                                     │
│  [1] サンプリング                                    │
│      N個のワーカーが並列でエピソードを走らせる         │
│      合計 min_batch_size ステップ分のデータを集める    │
│                                                     │
│  [2] ネットワーク更新                                │
│      フォロワー（制御）: 通常の PPO で更新            │
│      リーダー（形態）: Stackelberg 勾配で更新         │
│         ↑「フォロワーの反応を予測した勾配」を使う      │
│                                                     │
└─────────────────────────────────────────────────────┘
          ↓ 1000エポック繰り返す
最適な形態 × 制御を持つロボットが完成
```

## ニューラルネットワークの構造

同じ1つのネットワーク（ポリシー）が3種類の行動を出し分ける。

```
観測（状態）
  ├─ ボディの固定情報（親子関係・深さ等）
  ├─ シミュレーション状態（関節角度・速度・位置等）
  └─ 現在の設計パラメータ（リンク長・関節軸等）
       │
       ▼
  Transformer（各ボディをトークンとして処理）
       │
       ├─→ 骨格変換ヘッド → 「追加/削除/維持」を選択
       ├─→ 属性変換ヘッド → 「パラメータ調整量」を出力
       └─→ 制御ヘッド    → 「関節トルク」を出力
```

**Transformer を使う理由**: ロボットのボディ数はエピソード中に変わる（追加・削除）。可変長の入力を扱えるアーキテクチャが必要なため。

---

## ロボットの表現方法

ロボットは**ツリー構造のグラフ**として表現される。

```
根ボディ（胴体）
  ├─ 腕ボディ ─ 手首ボディ
  ├─ 脚ボディ ─ 足ボディ
  └─ ...
```

各ボディは「質量・慣性・形状・関節タイプ・リンク長...」のパラメータを持つ。このグラフが MuJoCo XML / URDF 形式に変換されてシミュレーターに渡される。

---

## タスク（pusher 環境ファミリー）

同じ pusher 環境コードが reward_specs の切り替えで3タスクを提供する（本研究の拡張）。

- ロボットは地面から伸びた「腕」のような形から始まる
- 骨格変換で腕の形・関節数を自由に変化させられる（本研究では fix_skeleton=true が本線）
- **Pusher**（デフォルト）: 報酬 = cube の x 方向速度 `Δcube_x/dt`
- **Reach**（`use_reach=true`）: 報酬 = `-dist(arm_tip, target)`
- **Target-Pusher**（`use_target_reward=true`）: 報酬 = target への PBRS（cube を目標位置に止める）

---

## まとめ: なぜこれが難しいか

| 課題 | 対処法 |
|------|--------|
| 形と動きを同時に最適化するのは難しい | Stackelberg: 設計時に動きの影響を予測 |
| ロボットの形が変わると観測の次元が変わる | Transformer: 可変長トークン列を処理 |
| 形の空間は巨大で探索が難しい | PPO: 確率的ポリシーで探索 |

## 補足: シミュレーターバックエンドについて

このシステムは `USE_CHOREONOID=1` 環境変数で MuJoCo と Choreonoid を切り替えられる。

```
USE_CHOREONOID=0（デフォルト）  → mujoco_env_gym.py    → MuJoCo で物理演算
USE_CHOREONOID=1               → mujoco_env_choreonoid.py → Choreonoid で物理演算
```

**現在の推奨**: Choreonoid バックエンド（`akita_sp` Docker イメージ）。
MuJoCo バイナリは不要。

**起動方法**（Choreonoidバックエンド）:
```bash
USE_CHOREONOID=1 OMP_NUM_THREADS=1 \
  choreonoid --no-window --python scripts/choreonoid_train.py \
  cfg=pusher num_threads=1
```

**なぜ `choreonoid --no-window --python` が必要か**:
`mujoco_env_choreonoid.py` は内部で `WorldItem`・`AISTSimulatorItem` を作成する。
これらは Choreonoid の Qt アプリケーションコンテキストが必要なため、
`choreonoid` プロセスの中で実行しなければならない。
通常の `python3` では segfault になる。
