# 研究方針

---

## 理想システム（長期ビジョン）

ユーザーが「やりたいタスク」と「使いたいロボットのスケッチ」を入力すると、
そのアイデアの妥当性（「この形態でこのタスクはできそうか？」）を素早く検証できるシステムを目指す。

```
スケッチ（手描き or AI生成メッシュ）
    ↓
形態パラメータ（リンク長・関節配置）
    ↓
Stackelberg PPO で形態 × 制御を co-design
    ↓
「この形態でこのタスクは妥当か」を数エポックで判定
```

ユーザーはロボット設計の専門知識がなくても、スケッチレベルのアイデアを素早く検証できる。

---

## 基盤ゴール（現在のスコープ）

**アームロボット（固定根本・操作タスク）** に限定して、上記システムの研究基盤を構築する。

### スコープの根拠

| ロボット種 | スコープ | 理由 |
|---|---|---|
| 固定根本アーム | ✅ **現在のスコープ** | 関節数が少ない・倒れない・タスクが先端位置に集約・スケッチ→パラメータ変換が容易 |
| 四脚ロボット | 🔮 将来の拡張候補 | 歩容設計・接地判定が絡み、シンプルな報酬で扱いにくい |
| ヒューマノイド | 🔮 将来の拡張候補 | 30+ DOF・バランス制御が必須・「スケッチ→形態パラメータ」の定義自体が難しい |

元論文（Stackelberg PPO）の locomotion タスク（ant・hopper 等）も 4〜8 関節のスケールで設計されており、
アームロボットはこのフレームワークのスケール前提と整合する。

## 研究の問い

> **シンプルな報酬でも、Stackelberg PPO の co-design はタスクに適した形態を正しく識別できるか？**

### なぜ「シンプルな報酬」を問うのか

- 複雑な報酬は特定の動作を誘導することはできるが、形態最適化のシグナルを消す副作用がある
- 「スケッチ → 妥当性検証」の用途では、報酬を毎タスク手作業でチューニングすることは現実的でない
- シンプルな報酬がうまく機能するなら、タスク記述から報酬を半自動的に導出する方向に拡張できる
- ボスの設計原則：「報酬はシンプルであるほど良い」

### 複雑化した報酬が失敗した教訓

pusher タスクで bat tactic（バット型スイング）を排除しようとして PBRS・reach_bonus・clipped vel 等を追加した結果、
co-design の探索シグナルが消えてむしろ学習が止まった。
bat tactic は「固定根本アームが速度報酬に対して取る正当な最適戦略」であり、排除すべき hacking ではなかった。
詳細: [タスク設計と報酬関数.md](タスク設計と報酬関数.md)

## タスクロードマップ

### 対象タスクと報酬設計方針

| タスク | シンプルな報酬 | bat tactic | 検証ポイント |
|---|---|---|---|
| **Pusher**（cube 押し） | `Δcube_x / dt`（速度） | 最適戦略として出現する | co-design が「長くて速い形態」を選ぶか（Phase 0 で確認済み） |
| **Reach**（先端到達） | `-dist(arm_tip, target)` | 発生しない | co-design が「リーチの長い形態」を選ぶか。最も reward hacking の余地が少ないクリーンなタスク |
| **Target-Pusher**（cube を目標に止める） | `Δφ`（PBRS: φ=1/(1+dist(cube,target))、オーバーシュートで負が内在） | 抑制される（最大速度でなく制御性が正解） | Pusher と正反対の形態要求（制御性 vs 速度）。形態差が最も出やすい第3タスク |

### フェーズ

```
Phase 0（完了）
  Pusher × co-design の動作確認
  ✅ co-design あり → fwd_cube > 0.1（実験 B）
  ✅ co-design なし → fwd_cube ≈ 0.01（実験 C）
  ✅ bat tactic が最適戦略として出現

Phase 1（✅ 完了・2026-07-06）
  Reach タスクで co-design が機能するか確認
  ✅ F7（co-design あり）: exec_R=-0.014・L1 が 0.3→0.839m に伸長
  ✅ G2（形態ほぼ固定）: -0.067 → co-design の効果を Reach でも実証
  - 過程で計6バグを発見・修正（docs/進捗.md 知見セクション）

Phase 2（進行中・2026-07-09）
  Pusher vs Reach の最適形態を比較
  - 中間シグナル: 前腕長（Reach 上限張り付き vs Pusher 内部解、2vs2 一貫）
    と gear（20 vs 400）の2軸で分岐を確認（docs/研究応用/形態比較.md）
  - 本判定: L1 vs L2（target 内側化 + ctrl コスト論文水準・差はタスクのみ）
    ※比較設計の交絡是正の経緯は方針レビュー_2026-07-08.md

Phase 3（パイロット起動済み・2026-07-09）
  Target-Pusher タスク追加・3タスク比較
  TP1（contact_weight=0・arm_safe_init=false・control_prior 転用）で挙動確認中
  ※当初案の Strike（final_cube_x）は telescoping により Pusher の速度報酬と
    ほぼ等価（割引と y ペナルティ分の差のみ）と判明したため target_pusher に変更
    （2026-07-09 決定。詳細: 方針レビュー_2026-07-08.md 懸念4）
```

## 論文との関係・独自 contribution の位置づけ

### 元論文（Stackelberg PPO, ICLR 2026）からの変更点

| 要素 | 元論文 | 本研究 |
|---|---|---|
| ロボット | mobile base（自由関節・追従可能） | **固定根本アーム**（追従不可） |
| manipulation タスク | pusher 1種のみ | **複数タスク（Reach・Target-Pusher）に拡張** |
| 報酬設計 | 速度報酬（シンプル・そのまま使用） | 速度報酬をベースに、固定根本アーム特有の問題（penetration exploit 等）への最小限の対処を追加 |
| Leader 専用報酬 | なし（論文 Appendix E.7 で「あり」とされている） | compute_reach_bonus()（開発・検証済みだが現方針では採用しない） |

### 独自 contribution

1. **固定根本アームへの適用と失敗の記録**: mobile base 前提の報酬・設計をアームに移植したときに何が起きるかを実証（bat tactic・penetration exploit 等）
2. **シンプルな報酬 × 複数タスク比較**: 同じ co-design フレームワークで複数のシンプルな報酬を使い、タスクごとの最適形態の差を定量比較
3. **スケッチ → 形態パラメータ → co-design パイプライン**: 理想システムへの基盤実装

## ネットワークアーキテクチャと将来拡張の方針

### 方策ネットワークの実体

Stackelberg PPO の方策は **Transformer ベース**（`TransformerSimple`）であり、GNN ではない。

- ロボットの各リンクを**シーケンスのトークン**として処理する
- padding マスクで可変長に対応しているため、2リンクでも6リンクでも同じネットワーク重みが原理上は適用できる
- self-attention は「全リンクの関係を一度に見て判断する」

グラフ構造（親子エッジ・ラプラシアンPE・距離行列）を計算するコードは `bodygen_policy.py` 内に実装されているが、**訓練ループで一度も呼ばれておらず**、リンクを単純な順序付き配列として処理している。「無効化されている」というより「実装済みだが未使用」の状態。

### 現スコープの制約と将来への道

現在は「2DOF・全関節 Y軸・グラフ構造無効」という単純な設定で研究を進めている。これは将来の拡張を閉じているわけではなく、段階的に解放できる：

| 段階 | 解放する制約 | 期待される効果 |
|---|---|---|
| 今 | — | 2DOF Y軸アームで原理確認 |
| 近い将来 | グラフ構造（エッジ・ラプラシアンPE）を有効化 | Transformer がトポロジーを構造的に理解できる |
| 中期 | 軸の種類（Y/Z）の多様化・DOF のカリキュラム拡張 | 6DOF 実機アームへの転用が見えてくる |
| 遠い将来 | スケッチ → 任意DOF・任意軸 → co-design | 理想システムの完成 |

### 6DOF 転用が現実的になる条件

以下の2つがそろって初めて高 DOF への転用が現実的になる：

1. **グラフ構造を有効化して訓練する** — 現状は全リンクを配列として見るだけなので、「このリンクはあのリンクの子」という情報が学習に活きていない
2. **軸の多様性を訓練データに混ぜる** — Y軸のみで訓練すると、Z軸関節（旋回）に対する注意パターンが分布外になる

### 関節軸の扱いに関する方針

スケルトン抽出は関節軸を90°誤るケースがある（詳細: [メッシュXMLパイプライン.md](メッシュXMLパイプライン.md) §7）。対策の優先度：

1. **今すぐ**: タスク別デフォルト軸（固定根本アーム × pusher/reach → 全関節 Y軸）
2. **Tripo3D 統合時**: PCA 平面法線による自動軸推定
3. **安定性確認後**: `joint_params` への限定的 PPO 軸最適化（±30° 範囲）を追加

なお `joint_params: axis` の最適化はコードに実装済みだが元論文でも未検証。単独では使わず、良い初期値（案1/2）と組み合わせて使うこと。
