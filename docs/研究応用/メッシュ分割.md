# メッシュ分割手法の比較と StackelbergPPO への統合方針

## 概要

Tripo3D 等で生成した単一メッシュをロボットの「パーツ」に分割する方法の比較。
分割粒度（パーツ数）の制御しやすさと、StackelbergPPO への接続しやすさを主軸に評価する。

---

## 1. ヒューリスティック・幾何学的アプローチ

プログラム的なルールで分割するため、**計算が軽く、過分割の制御が容易**。

### 1-c. 色ベースセグメンテーション ← **現在の推奨アプローチ（2026-07 採用）**

Gemini でロボット画像を生成し（またはユーザスケッチを整形し）、Tripo3D で 3D モデル化すると、
入力画像の色がテクスチャとして GLB に焼き込まれる。
この色情報を直接使って分割するのが最も信頼性が高い。

#### パイプライン

```
ユーザのスケッチ（または今回用の画像生成プロンプト）
  ↓ Gemini（色付き CAD スタイルに整形 / 生成）
色分け済み画像（各リンク = 単色、関節球 = マゼンタ）
  ↓ Tripo3D image-to-3D
color-textured GLB
  ↓ skeleton_extract.py --mode color --n-links N
topology.json + per-link GLB files
  ↓ mesh_to_params.py / topology_to_xml.py
MuJoCo XML → StackelbergPPO 学習
```

#### 色の設計規則

| パーツ | 色 | HEX |
|---|---|---|
| リンク 1（根元） | 鮮明な赤 | `#FF2222` |
| リンク 2 | 鮮明な青 | `#2222FF` |
| リンク 3（あれば） | 鮮明な緑 | `#22CC22` |
| リンク 4（あれば） | 鮮明なオレンジ | `#FF8800` |
| 関節球（接続点マーカー） | 純マゼンタ | `#FF00FF` |
| ベース台座 | 濃いグレー | `#333333` |

- **単色・高コントラスト** に塗ること（グラデーション禁止）
- **関節球** はリンクより太い球形の突起として描き、前後リンクとの色境界をくっきりさせること

#### `skeleton_extract.py --mode color` の動作

```bash
python mine/skeleton_extract.py robot.glb \
  --mode color \
  --n-links 3 \
  --joint-color 255 0 255 \
  --joint-tol 40 \
  --task-axis pusher \
  --segment-out mine/robot_segments/ \
  -o mine/robot_topology.json
```

アルゴリズム:
1. `mesh.visual.to_color()` → per-face RGB 抽出
2. マゼンタ（`#FF00FF`、tolerance=40）に近い面 → 関節球として除外
3. 残り面を K-means（`--n-links N` クラスタ）で色分類
4. クラスタを Z 座標でソート（高い方 = 根元 = link_0）
5. 関節球面の重心 → 関節位置（マゼンタ面が不足する場合は色境界から推定）
6. kinematic chain topology を構築

幾何学ベース（kink 検出）より単純・高精度。kink 検出は形状依存の誤差があるが、色は形状に依存しない。

#### 既知の制限

| リスク | 対策 |
|---|---|
| Tripo3D が色を混合/褪色させる | テスト時は GLB を Blender で開いて色を目視確認してから実行 |
| マゼンタ面が少なすぎる（Tripo3D で潰れる） | `--joint-tol` を 40→80 に緩める、またはスクリプトが境界検出にフォールバック |
| **金色は使わない**（2026-07 教訓：金色がアーム本体色と同化した） | マゼンタ #FF00FF は自然物には現れない → 誤検出しにくい |
| K-means が収束しない（色が似すぎる） | リンク色のコントラストを上げる（赤・青・緑を原色に近づける） |

### 1-a. スケルトン抽出ベース（Reeb Graph / Medial Axis）

メッシュの内部に「骨格（カーブスケルトン）」を抽出し、骨格の分岐点（ノード）を境界として分割する。

```
メッシュ
  ↓ Medial Axis Transform / Reeb Graph
骨格グラフ（ノード = 分岐点、エッジ = リンク）
  ↓ 枝刈り（Pruning）で細かすぎる枝を削除
ノード数 = パーツ数 に対応した分割
```

**メリット:**
- 骨格のエッジが直接「リンク」、ノードが「関節」に対応 → キネマティックチェーンと直結
- 枝刈りの閾値 1 つでロボットの関節数を直接コントロールできる
- 抽出した骨格がそのままジョイント軸の初期推定にも使える

**デメリット:**
- 球体・箱形など骨格が不定形なメッシュでは骨格が不安定になる
- 実装ライブラリが少ない（`scikit-tda`, `vedo` の `skeletonize()` など）

**ツール例:**
```python
import vedo
mesh = vedo.load("robot.glb")
skel = mesh.tomesh().skeletonize(tol=0.01)  # tol で枝刈り閾値を調整
```

### 1-b. 曲率・凹面ベースの分割（Concavity-based）

メッシュ表面の「凹んでいる部分（谷）」を関節の継ぎ目とみなし、そこを境界として分割する。

```
メッシュ
  ↓ 各頂点の mean curvature / Gaussian curvature を計算
  ↓ 曲率が閾値を超える「谷」領域を検出
  ↓ Fast Marching / Geodesic Distance で領域を成長
パーツ境界 = くびれ部分
```

**メリット:**
- 物理演算で干渉しやすい「くびれ」で自然に分割される → 爆発問題回避に直結
- 曲率の閾値だけで分割の粗さを制御できる
- `trimesh` で曲率を計算できる

**デメリット:**
- 曲率が均一なメッシュ（円柱状の胴体など）では境界が曖昧になる
- 表面ノイズに弱い（スムージング前処理が必要）

**ツール例:**
```python
import trimesh
import numpy as np

mesh = trimesh.load("robot.glb")
# 頂点曲率の近似: 各頂点の法線変化量
# trimesh には直接の曲率 API はないので法線変動を代替指標として使う
vertex_normals = mesh.vertex_normals
adjacency = mesh.vertex_adjacency_graph
curvature = np.zeros(len(mesh.vertices))
for v, neighbors in adjacency.items():
    neighbor_normals = vertex_normals[list(neighbors)]
    curvature[v] = np.mean(1.0 - neighbor_normals @ vertex_normals[v])

# 高曲率領域（閾値以上）= 分割候補境界
threshold = 0.3
boundary_vertices = np.where(curvature > threshold)[0]
```

**より本格的な実装:** [VHACD（Approximate Convex Decomposition）](https://github.com/mikedh/trimesh/blob/main/trimesh/decomposition.py)を使うと、凹面分解として直接パーツに切り出せる。

```python
# trimesh の VHACD ラッパー
parts = trimesh.decomposition.convex_decomposition(mesh, maxhulls=8)
# maxhulls でパーツ数の上限を指定
```

## 2. ディープラーニング・データドリブンアプローチ

セマンティクス（意味）を理解して分割するため、**スケッチの意図を汲み取りやすい**。

### 2-a. 特徴量クラスタリング（PointNet++ + K-Means）

メッシュを点群に変換し、PointNet++ で局所特徴量を抽出した後、K-Means でクラスタリングする。

```
メッシュ → 点群サンプリング（FPS: Farthest Point Sampling）
  ↓ PointNet++ エンコーダ
各点の局所特徴量ベクトル（128〜512次元）
  ↓ K-Means（K = 目標パーツ数）
K 個のセグメント
```

**メリット:**
- `K`（分割数）を直接指定できる → 「大まかに 5 パーツ」のようなユーザーフレンドリーな指定が可能
- 形状の幾何学的特徴（面の向き・曲率・局所構造）を自動的に学習した特徴量で分割されるため、意味的にまとまりやすい
- 事前学習済みモデルを使えばファインチューニング不要

**デメリット:**
- PointNet++ の推論が必要（GPU or 数秒の CPU 推論）
- クラスタリング結果が毎回異なる（初期値依存）

**ツール例:**
```python
# torch-points3d や open3d の PointNet++ を使う場合
import open3d as o3d
import numpy as np
from sklearn.cluster import KMeans

mesh = o3d.io.read_triangle_mesh("robot.glb")
pcd = mesh.sample_points_uniformly(number_of_points=4096)
points = np.asarray(pcd.points)

# PointNet++ の特徴量抽出（事前学習済みモデルを利用）
# ここでは簡略化として座標 + 法線を特徴量として代替
pcd.estimate_normals()
normals = np.asarray(pcd.normals)
features = np.hstack([points, normals])  # 6次元

K = 6  # 目標パーツ数
kmeans = KMeans(n_clusters=K, random_state=0).fit(features)
labels = kmeans.labels_
```

### 2-b. 3D-MLLM / VLM を用いた言語主導の分割（Open-vocabulary Segmentation）

画像または 3D モデルに対してテキストプロンプトを与え、ゼロショットで分割する。

```
メッシュ → マルチビューレンダリング（6〜12 方向からの 2D 画像）
  ↓ SAM2（Segment Anything Model 2）+ CLIP / DINO
「左腕」「右脚」「胴体」などのテキストプロンプト
  ↓ 各 2D マスクをメッシュ頂点に逆投影（バックプロジェクション）
3D セグメント
```

**メリット:**
- スケッチを描いた人が「ここは腕、ここは車輪」と指定した意図をそのままプロンプトとして流し込める
- 意味的なまとまりで剛体を切り出せる（胴体・脚など形状によらずセマンティクスで分割）
- ゼロショットのため追加学習不要

**デメリット:**
- マルチビューから 3D への逆投影で誤差が生じる（特に陰影になる部分）
- パーツ境界が 2D の視点依存になりやすい
- 推論コストが高い（SAM2 + CLIP の並列実行）

**ツール例（概念）:**
```python
# PartSLIP2 / PartDistill などを利用するアプローチ
# または SAM2 + テキストプロンプトで 2D セグメンテーション → 3D 逆投影
from segment_anything import SamPredictor
# ... マルチビューレンダリング → SAM2 でセグメント → 点群への逆投影
```

**参考実装:** [PartSLIP2](https://github.com/zyc00/PartSLIP2)（テキストプロンプトによる 3D パーツセグメンテーション）

## 3. 手法の比較

| 手法 | パーツ数の制御 | 意図の反映 | 計算コスト | StackelbergPPO との相性 |
|------|--------------|-----------|-----------|------------------------|
| **色ベース（★推奨）** | `--n-links` で直指定 ◎ | ◎（スケッチの色＝意図） | 軽い | ◎（Tripo3D テクスチャをそのまま利用）|
| スケルトン抽出 | 枝刈り閾値で○ | △（形状依存） | 軽い | ◎（骨格=キネマティックチェーン）|
| 凹面ベース（VHACD）| maxhulls で○ | △（物理的に自然） | 軽〜中 | ○（くびれ=関節で綺麗に切れる）|
| PointNet++ + K-Means | K を直指定 ◎ | △（意味は弱い） | 中 | ○（パーツ数指定が直感的）|
| 3D-MLLM / VLM | プロンプトで○ | ◎（意図を直接反映） | 重い | ○（意味的なパーツ定義）|

**推奨構成（2026-07 更新）:** Gemini → Tripo3D で **色テクスチャ付き GLB** を生成し、`--mode color` で直接分割する。Tripo3D が色を保持しなかった場合のフォールバックとして `--mode chain` + `--task-axis pusher` を使う。

## 4. StackelbergPPO への統合方針

### 分割結果を初期状態 $s_0^L$ として設定する

分割済みメッシュを `topology.json`（[メッシュXMLパイプライン.md](メッシュXMLパイプライン.md) 参照）に変換した上で、Leader の初期状態を固定する。

```
分割済みメッシュ × N パーツ
  ↓ FK 解決・カプセルフィット（メッシュXMLパイプライン.md）
topology.json + assets/mujoco_envs/my_robot.xml
  ↓
StackelbergPPO（fix_skeleton=true）
```

### Leader のアクション空間を制限する

| 制限事項 | 設定方法 | 意図 |
|---------|---------|------|
| トポロジーを固定（腕の追加・削除を禁止） | `fix_skeleton=true` or `max_nchild=0` | スケッチの意図を壊さない |
| リンク長の微調整幅を制限（±20%程度） | `body_params.offset` の `lb/ub` を現在値近傍に設定 | 「スケールの小さな補正」のみ許可 |
| モーターギア比の探索範囲を絞る | `actuator_params.gear` の `lb/ub` を実機仕様近傍に設定 | 物理的に非現実的なトルクを防ぐ |
| 関節可動域を解剖学的制限内に収める | `joint_params.range` の `lb/ub` を人体・動物の可動域に設定 | 自然な動作に制限 |

**設定例（スケッチ由来の 4 脚ロボット用 yml）:**

```yaml
# design_opt/cfg/my_robot.yml
env_name: ant  # または pusher
fix_skeleton: true

robot:
  param_mapping: sin
  no_root_offset: true

  body_params:
    offset:
      type: 'xy'
      lb: [-0.6, -0.6]   # デフォルト ±0.5 より少し広げる
      ub: [0.6, 0.6]

  geom_params:
    size:
      lb: 0.03
      ub: 0.12            # スケッチのリンク太さに合わせて調整

  actuator_params:
    gear:
      lb: 50              # 実機の最小トルク想定
      ub: 300             # 実機の最大トルク想定

  joint_params:
    range:
      lb: -1.04           # -60度（ラジアン）
      ub:  1.04           # +60度
```

### 分割手法ごとの統合特性

| 分割手法 | topology.json 生成の容易さ | ジョイント位置の自動推定 |
|---------|--------------------------|------------------------|
| スケルトン抽出 | ◎（骨格ノード → ジョイント座標が直接得られる）| ◎ |
| 凹面ベース | ○（パーツ境界の重心 → ジョイント候補）| △ |
| PointNet++ + K-Means | △（クラスタ境界からジョイント位置を推定）| △ |
| VLM プロンプト | ○（セマンティクスからジョイント名を推定可）| △ |

**スケルトン抽出が最も StackelbergPPO との相性が良い理由:**
骨格抽出の結果がそのまま `topology.json` の `offset_from_parent` に使えるため、パイプライン全体の人手介入が最小になる。

## 5. 実装ロードマップ

```
✅ Phase 1: 手動 topology.json で動作確認（完了）
  └─ パイプライン後半（FK → XML → 学習）の動作確認済み

✅ Phase 2: スケルトン抽出（chain/branch モード）実装（完了）
  └─ skeleton_extract.py: voxel → MST → kink 検出 → 最近傍ボーン割り当て
  └─ rrbot 79.2%、UR3 chain ~40%（水平姿勢が苦手）

✅ Phase 3: 色ベースセグメンテーション（--mode color）実装（2026-07 完了）
  └─ Gemini → Tripo3D の color-textured GLB を入力
  └─ per-face RGB 抽出 → 関節色マスク（マゼンタ、当初は金色） → K-means → Z ソート → topology 構築
  └─ 幾何学依存をなくした「色が意味を持つ」パイプラインを確立

⬜ Phase 4: Tripo3D 実メッシュでエンド to エンド検証（次のタスク）
  └─ 空間分割ルート（glb_to_links.py、関節 Z 手動指定）では RL 学習まで実証済み（2026-07-08、メッシュXMLパイプライン.md §8）
  └─ --mode color の E2E はマゼンタ版 GLB の再生成待ち（金色版はマーカーが本体色と同化し検出不可だった）
  └─ Tripo3D が色を保持しない場合は --mode chain へ自動フォールバック追加

⬜ Phase 5: VLM プロンプト統合（オプション、将来）
  └─ PartSLIP2 等で意味的パーツ名を topology.json の name フィールドに自動入力
```

## 6. rrbot での実装・検証結果

### 対象ロボット: rrbot

ROS の練習用ロボット。3リンクのシンプルな垂直アーム。

```
世界座標での構成（URDF 正解）:
  link1  : 0.1×0.1×2.0m の直方体  中心 z=1.0m
  link2  : 0.1×0.1×1.0m の直方体  中心 z=2.4m  y=0.10m
  link3  : 0.1×0.1×1.0m の直方体  中心 z=3.3m  y=0.20m
  camera : 0.05×0.05×0.05m の箱   中心 z=3.75m y=0.20m

  joint1 : [0.0, 0.1, 1.95]  軸=Y
  joint2 : [0.0, 0.2, 2.85]  軸=Y
```

メッシュ: 結合後 **48面・32頂点**（各リンクが直方体 12 面）

### 実装した手法: 最近傍ボーン分割（スケルトン抽出ベース）

`khrylib/utils/skeleton_extract.py` に実装。パイプライン:

```
1. ボクセル化 (pitch=0.02m)
2. スケルトン点群の抽出・クラスタリング (r=0.04m)
3. 最小全域木 (MST) でグラフ化
4. チェーン順序付け（最高Z点をルートに設定）
5. 折れ角（kink）検出で関節位置を推定

#### 抽出スケルトン結果

#### 関節位置精度

| GT関節 | GT位置 | 推定位置 | 誤差 |
|---|---|---|---|
| joint1 | [0.0, 0.1, 1.95] | [-0.04, -0.04, 1.89] | **0.158m** |
| joint2 | [0.0, 0.2, 2.85] | [-0.03, 0.107, 2.833] | **0.099m** |

- 20cm以内に収まった関節: **2/2 (100%)**
- 10cm以内に収まった関節: **1/2 (50%)**

```
GT軸:    Y軸 [0, 1, 0]

rrbot の関節軸（Y軸）はリンクの延伸方向（Z軸）と直交するため一致しない。

#### メッシュ分割精度（混同行列）

```
                  link1(GT)  link2(GT)  link3+cam(GT)

  link_1 (推定)       0          4          20       → 主に link3+cam を担当
  link_2 (推定)       4          8           2       → 主に link2 を担当

| GT リンク | 総面数 | 最良マッチ | リコール |
|---|---|---|---|

| link3+cam (GT) | 22面 | link_1 | **90.9%** (20/22) |
| **全体** | **48面** | — | **79.2%** (38/48) |

### 精度が落ちる原因

1. **rrbot は非常に単純な形状**
   凹凸がなく、細長い直方体が並んでいる。スケルトン抽出は局所的な形状変化（くびれ・分岐）を手がかりにするため、均質な形状では kink 検出の精度が落ちる。

2. **link_0 に面が割り当てられない**
   最近傍ボーン方式では、「ボーン（親→子の線分）」への距離で各面を割り当てる。ルートボディには親がないためボーンが定義できず、常に面数ゼロになる。

3. **link1/link2 の境界付近で混乱**
   joint1 の推定位置が GT から 0.158m ずれているため、link1 と link2 の境界が誤る（confusion matrix の中央ブロックに誤分類が集中）。

### 他手法との比較（推定）

| 手法 | rrbot 精度（推定） | 理由 |
|---|---|---|
| **最近傍ボーン（実装済み）** | **79.2%** | rrbot の単純形状には kink が曖昧 |
| 凹面ベース（VHACD） | ~60-70% | 凹面が少ないため分割点が見つかりにくい |
| PointNet++ + K-Means | ~70-85% | 局所特徴量で区別できるが K の指定次第 |
| VLM（SAM2 + CLIP） | ~80-90% | 色・テクスチャがない箱形では弱い |

> ※ VHACD・PointNet++・VLM は未実装。推定値。

### 実用性の評価

**rrbot は「最悪ケース」に近い**:
- 凹凸なし・同質材料・単純形状 → スケルトン手法が苦手とする対象
- 本来の想定ターゲット（Tripo3D 生成の実際のロボット形状）は脚・アーム・関節部分に凹凸があり、精度は改善すると考えられる

**79.2% という精度について**:
- StackelbergPPO の枠組みでは、メッシュ分割の結果はあくまで「初期形態の提案」であり、Leader エージェントが学習中にリンク長・太さ・関節位置を最適化する
- 分割が多少ずれていても、**トポロジー（何本脚か・腕が何段か）が正しければ許容範囲**
- rrbot のトポロジー（3段チェーン）は正しく抽出できているため、初期値としては十分

## 7. UR3 での検証：ロボットの問題 vs アルゴリズムの問題

rrbot（単純な箱形・垂直チェーン）との対比として、より現実的な **UR3（6自由度アーム・STLメッシュ）** で同じアルゴリズムを検証した。

### 対象ロボット: UR3

Universal Robots の産業用アーム。6関節のシリアルチェーン。

```
FK (q=0) での世界座標:
  base_link      [0,     0,     0    ]  ← 根元ベース
  shoulder_link  [0,     0,     0.152]  ← 肩（Z軸回転）
  upper_arm_link [0,     0.120, 0.152]  ← 上腕（Y軸回転）
  forearm_link   [0.244, 0.027, 0.152]  ← 前腕（Y軸回転）

|---|---|---|

### 精度サマリ（2つの指標）

評価に 2 種類の指標を使う:

**① per-GT-link recall**（rrbot との比較向き）
- 各 GT リンクの面が「どこか 1 つの抽出ボディ」に集まっているかを測る
- 複数の GT リンクが同一抽出ボディに混入していても加点される（甘い指標）

**② semantic accuracy**（実際の分割品質）
- 抽出ボディと GT リンクを 1 対 1 で対応付け（最大二部マッチング）
- 正しく分離されていない面は不正解とする

| ロボット | モード | ① per-GT recall | ② semantic accuracy |
|---|---|---|---|
| rrbot | chain | **79.2%** | ~75% (rrbot では①≈②) |
| UR3 | chain | 91.9%（見かけ） | **~40%** |
| UR3 | branch | 86.6%（見かけ） | **~54%** |

UR3 の chain モードでは①と②の乖離が大きい（91.9% vs 40%）。これは「base+shoulder+upper_arm の 3 リンクが 1 つの抽出ボディにまとめられた」ことで per-GT recall が高止まりしているため。

### 結論：アルゴリズムの問題か、ロボットの問題か

**主要因はアルゴリズム（Z方向オーダリングの仮定）**

| 失敗要因 | 詳細 |
|---|---|
| **Z方向チェーン仮定** | chain モードは「根元が高Z、先端が低Z」を前提にする。UR3 は q=0 で水平伸展し、上腕〜手首が Z≈0.152 に揃うため kink が縦方向に現れない |
| **多関節同一平面問題** | 6 関節のうち 4 関節（shoulder_lift, elbow, wrist_1, wrist_2）がほぼ同一 Z 平面上にある → 横方向 XY ジャンプで検出可能なはずが、チェーン順序が X 方向になるためスコアが低くなる |
| メッシュ形状の複雑さ | 副次的。高密度 STL は skeleton 抽出の品質に **好影響**（rrbot の単純箱より細かい形状情報がある） |
| ロボット形状そのもの | 影響なし。UR3 は幾何学的に明確なくびれ・段差を持つ実際の産業用ロボット形状 |

**branch モードは縦横不問だが** 「serial chain の分岐点がない」という問題がある。branch モードは degree-3+ のノードを関節とみなすが、serial chain のスケルトンは degree-2 のみ → 全ノードが縮約されて 4 ボディになる。

### 対策の方向性

| 対策 | 効果 | 難易度 |
|---|---|---|
| **ロボットを垂直姿勢でサンプリング** | chain モードの Z 仮定を活かせる | ○ 姿勢選択のみ |
| **XY+Z 複合 kink 検出** | 水平アームの kink も検出可能 | △ アルゴリズム改修 |
| **スケルトンを 3D 距離で分割** | chain 順序に依存しない | △ chain/branch 統合 |
| **q=π/2 等で再サンプル** | 全関節を垂直展開してから処理 | ○ 前処理のみ |

→ 最も実装コストが低いのは「**垂直に近い姿勢（全関節 0 ではなく π/2 等）で結合メッシュを作る**」前処理。

### 姿勢正規化（前処理）の追加と効果

`compare_ur3.py` に `pose_normalize()` を追加した。PCA でボディ位置の主成分方向を求め、それを +Z に合わせる回転を適用する。

```python
positions = np.array([world_T[name][:3, 3] for name in link_order])
_, _, Vt  = np.linalg.svd(positions - positions.mean(axis=0), full_matrices=False)
main_axis = Vt[0]          # 最大分散方向
# Rodrigues 回転で main_axis → [0,0,1]
```

UR3 では主軸 = [0.994, 0.10, 0.054]（ほぼ X 方向）→ **87° 回転**して Z 方向に揃えた。

#### 正規化前後の比較

| 指標 | 正規化前（q=0 水平） | 正規化後 |
|---|---|---|
| kink 検出数 | 6（正解と一致） | 10（過検出） |
| 抽出ボディ数 | 8 | 12 |
| 5cm以内の関節 | **2/6** | **4/6** |
| 10cm以内の関節 | 5/6 | **6/6** |
| per-GT recall | 91.9%（見かけ） | 82.2% |
| 実質的分離 | base+shoulder+upper_arm が混在 | shoulder・upper_arm・forearm が分離 ✓ |

**正規化でうまくいったこと：**
- shoulder_lift_joint 位置誤差: 0.118m → **0.009m**（13倍改善）
- shoulder・upper_arm・forearm が別々のボディに分離
- 全関節が 10cm 以内に収まった

**正規化で新たに発生した問題：**
- base_link が 4 ボディに過剰分割（ベースが「チェーンの先端」として底に来るため、円筒形状の内部に kink が生まれる）
- wrist_1+2+3 は引き続き 1 ボディに混在（コンパクトな手首が分離不能）
- kink が 6 → 10 に増え、`min_segment_len` の再調整が必要

**結論：** 正規化は関節位置推定を大幅に改善するが、kink 閾値の再チューニングがセットで必要。

## 8. スケール比較：StackelbergPPO ロボット vs 実ロボット URDF

### ロボットスケール一覧

| ロボット | 代表サイズ | StackelbergPPO比 |
|---|---|---|
| **ant**（StackelbergPPO） | 胴 r=0.25m、脚 0.57m カプセル | 基準 |
| **hopper**（StackelbergPPO） | リンク 0.4m + 0.45m = 高さ ~1.25m | 基準 |
| **UR3** | リーチ 0.47m、高さ 0.21m | **≈ 同スケール** |
| **rrbot** | 高さ **4m**（link1=2m, link2=1m, link3=1m） | **8倍大きい** |

UR3 は StackelbergPPO ロボットとほぼ同スケール（0.25〜1m）。rrbot はあくまで ROS 練習用の非現実的なサイズ。

### pitch パラメータのスケール依存性

skeleton_extract.py の主要パラメータはすべて物理距離（メートル）。リンクの最小径（細さ）から決定する。

| ロボット | 最小リンク径 | 適切な pitch | 使用値 |
|---|---|---|---|
| rrbot | 0.10m（辺長） | 0.01〜0.02m | 0.02m ✓ |
| UR3 | 0.04m（手首） | 0.008m | 0.008m ✓ |
| ant/hopper | 0.05〜0.08m（カプセル半径） | 0.01m | — |
| Tripo3D 生成 | 不定 | **スケール正規化後に設定** | — |

### Tripo3D 生成メッシュへの適用上の注意

**スケールが不定**（1 unit = 1m のツールと 1cm のツールが混在）。
適用前に AABB 対角長などでスケール正規化する前処理が必要:

```python
bbox = mesh.bounds           # (2, 3)
diag = np.linalg.norm(bbox[1] - bbox[0])
TARGET_DIAG = 0.8            # 想定ロボット対角長 [m]
mesh.apply_scale(TARGET_DIAG / diag)
```

この正規化と姿勢正規化（主軸 → Z）をセットで前処理とすれば、pitchや閾値パラメータを固定値で運用できる。

#   mine/ur3_topology_after_norm.json    ← 正規化後スケルトン
#   mine/ur3_segments/before_norm/       ← 正規化前 分割 GLB
