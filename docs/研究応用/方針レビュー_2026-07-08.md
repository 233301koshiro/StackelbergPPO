## 懸念1（最重要）: Phase 2 の比較が「同じ角への張り付き」で無効になるリスク

「Pusher と Reach で異なる形態が出るか」が研究の核心だが、**両タスクとも設計空間の同じ角が最適になる構造**に見える。

| 事実 | 出典 |
|---|---|
| G3（Reach）: L11 が上限 0.901 に張り付き（ep20 で bounds の角に到達、以降不動） | 進捗.md 形態解析 |
| Pusher の bat tactic: 先端速度 = ω×(L1+L11) → **長いほど有利** | タスク設計と報酬関数.md §2 |
| Reach の target_x=1.5 は実効リーチのほぼ限界 → **Reach も「最大限伸ばす」が最適** | F6 の設計空間不足の知見 |

両方の答えが「bounds の許す限り長く」なら、形態が一致しても「co-design がタスクを識別できない」証拠にはならない——**設計空間の天井を測っているだけ**。

**対策案**:
1. **Reach の target を作業空間の十分内側に置く**（例: x=0.8）。到達ギリギリでなければ「必要十分な長さ + 精密停止できる gear」が最適になり、Pusher の「最大長・最大 gear」と原理的に分かれる
2. bone_offset の bounds を広げ、少なくとも一方のタスクの最適が**内部解**になるようにする
3. gear を比較軸にする計画（タスク計画.md fallback）は妥当だが、懸念2 と連動する

## 懸念2: ctrl コストが論文実装より約3桁弱い

`pusher.py` のデフォルトは `ctrl_cost_coeff=1e-4` × `mean(a²)`、元論文は `0.1 × sum(a²)`（§1 に自ら記録あり）。2関節なら実効で**約2000倍トルクが安い**。cfg/pusher.yml は reward_specs を設定していないためこのデフォルトが効いている。

- 「元のシンプル報酬をそのまま使用」という研究方針.md の記述と実態がずれている
- トルクがほぼ無料なら **gear は常に上限に張り付くのが合理的** → 「gear を形態差の比較軸にする」計画と正面衝突
- gear で差を見たいなら ctrl コストを論文水準に戻すか、少なくとも明示的に選択して文書化すべき

## 懸念3: Reach 報酬の構造は「早期終了が得」のまま

累積 `-dist` 報酬はエピソードが短いほど罰の総量が減る。F/F2/F3 の NaN 偽収束（ep_len 317→23 崩壊）は**この構造の必然的な帰結**で、NaN ペナルティ -1000 は症状を塞いだだけ。別の早期終了経路（max_ang 等）が見つかれば再発する。

**構造的な解**: エピソード長の固定、または progress 形式 `d(t-1) − d(t)`（telescoping で長さ不変）。現状 J1/G 系は安定しているので優先度は中だが、環境をいじる際に再発しやすい地雷。

## 懸念4: Strike は Pusher とほぼ同値のタスク

Pusher の速度報酬の総和は `Σ Δx/dt = (final_x − initial_x)/dt`。**割引を無視すれば Strike の `final_cube_x` と定数倍しか違わない**（γ=0.995 と y ペナルティ分の差のみ）。3タスク比較の3本目としての情報量は薄い。

形態要求が本当に異なる第3タスクは **target_pusher（cube を目標に止める）**——最大速度ではなく制御性を要求し、bat tactic では解けない。詳細は後述の target_pusher 評価を参照。

## target_pusher（design_opt/cfg/target_pusher.yml）の評価

> 問い: シンプルな報酬設計と、論文の「形態を決めた時点で Follower の出来を先読み・継承して評価する」思想（Stackelberg、eq.5）を遵守しているか？

### 遵守している点 ✅

1. **Stackelberg の報酬結合は無傷**。`use_target_reward` は R^F（実行フェーズ報酬）の中身を変えるだけで、「Leader が Follower の実リターンをそのまま継承する」仕組み（eq.5）には触れていない。§8 で NG とされた「分離」に該当しない。
2. **Leader 専用加算 `compute_reach_bonus()` は無効**（`reach_bonus_scale` デフォルト 0.0、yml でも未設定）。「現方針では採用しない」（研究方針.md）と一致。
3. **報酬は実質2項**（target PBRS + 0.5×contact PBRS）+ 微小 ctrl コスト。D6/D7 や reach_bonus 時代のスタックと比べ十分シンプル。
4. **target PBRS は Leader の形態比較を歪めない**。エピソード合計は telescoping で `φ_T − φ_0` になるが、φ_0 = 1/(1+|cube_0−target|) は cube 初期位置（固定）だけで決まり**形態非依存**。「オーバーシュートで負」という bat tactic 抑制も追加項なしで報酬定義に内在しており、筋が良い。

### 遵守できていない・危うい点 ⚠️

1. **contact PBRS（contact_weight=0.5）が Leader の形態比較を歪める（最重要）**。
   PBRS の方策不変性定理は「固定 MDP 内」の話であり、**Leader は形態間でリターンを比較する**。contact 項のエピソード合計は `φ_T − φ_0`、φ_0 = 1/(1+dist(cube, tip_0)) は**形態依存** → **cube から遠くに生まれる形態ほど contact 報酬の「伸び代」が大きい**。これは v2-3/v5-2 で観測された「arm が cube 逆方向に成長し到達不能」の力学的説明そのもの。Follower には無害でも、Leader には「Follower の出来と無関係な、初期配置で決まる項」が混入する——論文思想への违反はここ。
   **対策**: (a) `contact_weight: 0` にして探索の立ち上げは `control_prior`（K1 で検証中の仕組み）による転用で賄う、または (b) `prev_contact_potential` の初期化を形態非依存の定数（例: 0）にする。(b) なら合計が `φ_T − 定数` になり歪みが消える。初回ステップに一度だけ φ 相当が出るが、per-step ではないため静止搾取は復活しない。
2. **純 target PBRS の bootstrap 失敗の前歴**。D2（物理バグ修正後・大 cube でも ep4 で立ち上がらず）と同系統。smoke（2ep、fwd_cube=0.0001）では判定不能。スクラッチ起動なら同じ失敗を繰り返す公算が高く、**I1/I2 の押し方策からの warm-start（control_prior）を前提に設計すべき**。
3. **`arm_safe_init: true` は §15 の教訓と矛盾**。「arm_safe_init 廃止が本質的ブレークスルー（初めて qpos0 が全ステップで動いた）」（2026-07-03）と結論済み。penetration exploit 防止が目的なら `check_init_contact: true` で足りるか要検討。
4. （軽微）PBRS が厳密形 `γφ(s')−φ(s)` でなく `φ(s')−φ(s)`。γ=0.995 なので実害はほぼないが、脚注として認識しておく。
5. （軽微）「止める」タスクでは cube を置いた後の余計な動きを抑える項が ctrl コストのみ。1e-4 では target 付近の jitter で cube を蹴り出すリスクがある。

### 方針文書上の位置づけ

研究方針.md のタスクロードマップ（Pusher/Reach/Strike）に target 系タスクは存在しない。進めるなら **Strike の代替として** ロードマップに正式に載せるのが一貫する（懸念4: Strike は Pusher とほぼ同値、target_pusher は形態要求が真に異なる）。「シンプル報酬原則」は PBRS 1項（contact_weight=0 の場合）なら維持できる。
