## 1. 序論

### 1.1 背景

- 形態と制御の同時最適化（co-design）研究の現状: Stackelberg PPO（Dai et al., ICLR 2026）は Leader（形態）が Follower（制御）の適応を織り込んで形態を更新する枠組みを提案（eq.5 の return 継承）
- しかし既存 co-design 研究の入力は**人手のパラメトリックな初期形態**であり、設計案を「絵」の段階から評価する経路がない
- 原論文はモバイルベースのロコモーション系9環境で評価しているが、**タスク間で最適形態がどう分かれるかは比較していない**（arXiv:2603.15388 本文・付録確認済み、2026-07-13）— 「co-design がタスクを識別する」こと自体が検証されていない空白

### 1.2 ビジョンと想定ユーザー（2026-07-13 言語化）

- **ロボット設計者の試作前検証**: ラフ案の段階で「この形はこのタスクに筋が良いか」を製作前に判定する
- **非専門家の設計参加**: 絵が描ければロボットの形を提案できる、設計参加の間口の拡大
- 対象を**固定根本アーム**とする動機（2026-07-13 言語化）: (i) co-design 検証の**最小構成**（2関節・平面・固定ベース）であること、(ii) **スケッチ/画像生成との相性**（据え置き型アームは生成 AI で構造が安定して描け、関節マーカーの自動抽出が成立しやすい）

### 1.3 課題

1. 原論文の前提（モバイルベース・1m cube）から固定根本アームへ移すと何が壊れるか（→ 4章）
2. co-design の判定は信用できるか = タスクの要求を形態に正しく反映するか（→ 5章）
3. スケッチ1枚ごとに 1000ep 学習は非現実的 → 判定の高速化（→ 6章）

### 1.4 貢献

- **C1（システム）**: スケッチ → 関節構造自動抽出 → XML → co-design 学習 → 判定、の人手ゼロ E2E パイプライン（マゼンタマーカー方式の自動関節検出を含む）
- **C2（実証）**: シンプル報酬 × 3タスク × 2 seed の統制実験による「co-design はタスクに適した形態を識別する」の定量実証 — 原論文が扱っていないタスク間形態比較（識別軸: gear プロファイル・リーチ帯・行動クラスタ）
- **C3（高速化）**: 転用粒度の分解実験（制御のみ/形態のみ/全部/なし）による warm-start 設計則と、判定所要エポックの較正（順位確定は ep100、最下位除外は ep75）
- **C4（設計知見）**: 固定根本移植で顕在化した構造的問題群（bat tactic の正当性・PBRS の φ_0 形態依存・bounds 張り付きと比較可能性・ペナルティ境界が形態を拘束する現象）とその是正の記録

## 4. 固定根本アームへの移植と環境是正（C4 前半）

### 4.1 何が壊れたか
- bat tactic の出現と再解釈: 排除すべき hacking ではなく「固定根本 × 速度報酬」の正当な最適解（tip 速度 = ω×ΣL）。**ctrl コスト論文水準でも bat は生存**し、省エネ化（打撃後 ctrl 2.7→0.002）で応答することを行動実測で確認（形態比較.md ④）
- 報酬複雑化の系統的失敗（v2〜v5/D 系）: 目標座標→penetration exploit、PBRS→探索シグナル消滅、reach_bonus→形態専用チューニング化。**教訓: 報酬を複雑にするほど co-design の探索シグナルが消える**
- PBRS の φ_0 形態依存: contact PBRS の合計は telescoping で φ_T−φ_0 に潰れ、初期状態を Leader 自身が決める co-design では「遠くに生まれる形態ほど伸び代が大きい」歪みが構造的に入る（Ng の方策不変性は形態固定の Follower にしか効かない）
- 環境ミスマッチの特定: 原論文はモバイルベース + 1辺 1m cube。固定根本ではこの前提が崩れ、環境側の是正（cube 拡大 + シンプル速度報酬回帰）で学習が成立

### 4.2 比較設計の是正
- 幾何床（target 1.5 > 最大リーチ 1.440、hover 実測 6.02cm）→ target 内側化 0.8
- ctrl コスト論文水準（0.2×mean ≡ 0.1×sum、2関節）
- bounds 張り付きの診断則: 「角に張り付いた形態はタスクの選好ではなく箱のサイズを測っている」

## 5. 実験1: co-design はタスクを識別するか（C2 = システムの妥当性根拠）

### 5.1 セットアップ
- 3タスク: Pusher（速度）/ Reach（-dist、target 0.8）/ Target-Pusher（target PBRS、φ_0 形態非依存）
- 統制: L 系 + TP 系はタスク以外の全条件同一（ctrl 0.2・scale 1・fix_skeleton・1000ep）、各 2 seed

### 5.2 co-design の必要性（前提実験）
- B vs C: 形態最適化あり/凍結で fwd_cube 8〜10倍差（0.10-0.12 vs 0.01-0.02）

### 5.3 形態の分離（主結果）
- 旧世代 2×2 final（確定済み）: Reach = L11 上限 0.901・gear 20/20 × 2 run / Pusher = L11 内部解 0.64-0.67・gear 400/400 × 2 run
- L 系（交絡なし）中間: Reach 肩 gear=20（2/2）・リーチ帯 0.9-1.1 / Pusher 肘 gear=400（2/2）・リーチ帯 1.2-1.7 / TP1 = 短腕 0.81・gear (66,20) の第3クラスタ
- **識別軸の定式化: 個別パラメータではなく機能量（gear プロファイル・リーチ帯）。タスク → 単一形態ではなく戦略ファミリー**（タスク内 seed 分散は大きい: 長さ配分・肩 gear・L11 の bounds 接触が seed で揺れる）

### 5.4 行動の分離（形態の裏付け）
- 実測 3 クラスタ: bat（Pusher、cube 4.1〜6.4m 飛翔・ピーク 9.7〜17 m/s）/ 持続押し（TP、0.3m を 1100 步・ピーク 0.19 m/s）/ hover（Reach、サブミリ精度静止）
- 機構の傍証: 低 gear の利得は「ノイズ下の着地失敗の抑制」（確率的評価で L1 は 1/3 誤着地 vs L1_s2 は 3/3 成功。n=3 傾向）

- scale 不変性（H1 ≈ I2、G2 ≈ G3/G4）

## 8. 考察

### 8.1 設計論としての知見（C4 後半）
- ペナルティ設計の原則: 「正直な最悪エピソードより確実に悪い」こと（penalty=1 が報酬になった実測）+ 棄却でなくペナルティ（棄却は Leader の脱出勾配を消し死のスパイラルに入る）
- **ペナルティ境界が形態を拘束する現象**: TP1 の「短い腕」は init_contact 幾何境界の 1.2cm 内側まで伸びた形態 — 形態の解釈にはタスク報酬だけでなくペナルティ幾何の考慮が必要
- 長期学習の崩壊は方策側で起こる（J1: 消去法で gear ✗・L0 obs ✗・方策重み ✓）→ 打ち切りは傾きの消滅で判断し best.p を採用
