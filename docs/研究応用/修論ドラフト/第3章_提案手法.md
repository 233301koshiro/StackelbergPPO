## 3.1 システム全体像

提案システムは、手描きスケッチを入力とし、その形態の対象タスクへの適性判定を出力する。処理は次の6モジュールをこの順に通過する一方向のデータフローであり、モジュール間の受け渡しに人手を要しない。

```
手描きスケッチ
  → [M1] 画像整形（関節マーカー規約の適用）        … 生成 AI（プロンプト固定）
  → [M2] 三次元形状生成（image-to-3D）             … 既存技術（Tripo3D）
  → [M3] 関節検出とリンク分割                      … 色検出 + 幾何処理（決定的）
  → [M4] 形態パラメータ抽出                        … OBB 近似（決定的）
  → [M5] シミュレーションモデル生成                … テンプレート変換（決定的）
  → [M6] 形態-制御 co-design による評価・補正      … Stackelberg PPO
  → 判定（タスク適性スコアと補正後形態）
```

設計方針は3点である。第一に、可動構造の獲得を学習ベースの推定に委ねず、**上流（M1）で規約を課し、下流（M3）を決定的な検出に置き換える**。生成 AI の出力の不確かさをパイプライン内部で吸収するのではなく、生成 AI への入力を統制することで問題自体を消す。第二に、M3〜M5 を**乱数や学習を含まない決定的処理**とし、失敗時の原因特定と再現を容易にする。第三に、M6 は既存の co-design フレームワークをモデル生成側が「話せる」形式（設計パラメータ付きの MuJoCo 互換 XML）に合わせることで、co-design 側のコード変更なしに接続する。

以下、データの流れに沿って各モジュールを述べる。

## 3.2 M1: 画像整形と関節マーカー規約

入力スケッチはまず画像生成 AI（Gemini）により、後段の三次元化・検出に適した「機械設計図スタイル」の画像へ整形される。このとき本システムの中核的な工夫である**関節マーカー規約**を、固定プロンプトとして課す。規約の要点は次の3つである。

1. **すべての関節接続部に純マゼンタ（#FF00FF）の球を1個ずつ描くこと**。マゼンタは自然画像・工業製品に稀な色相であり、後段の色検出の分離性を最大化する
2. **各リンクを彩度の高い単色（赤・青・緑等）で塗り分け、グラデーションを禁止すること**。またマゼンタをリンク本体色に使うことを禁止する
3. **関節の位置を場所として列挙し（「台座と上腕の間」等）、描画後にマゼンタ球の個数を自己検査させること**。位置の列挙を欠くと、生成 AI は先端側の関節をヒンジ金具等の機械要素として描き分けてしまい、マーカーが欠落することが実測で確認されている

この規約は「生成 AI に構造推定をさせる」のではなく「生成 AI に構造の**宣言**をさせる」設計であり、M3 の関節検出問題を、精度保証のない推定問題から閾値つき色マッチングという決定的問題に変換する。

## 3.3 M2: 三次元形状生成

整形済み画像は image-to-3D サービス（Tripo3D）により三次元メッシュ（GLB 形式、Y-up 座標系）へ変換される。このモジュールは既存技術をそのまま用いるが、2点の実測知見が後段の設計に反映されている。第一に、**テクスチャの色変換**である。入力画像の純マゼンタ #FF00FF = (255, 0, 255) は、生成過程で概ね RGB (211, 75, 169) へ暗色化する。ただしリンク本体色（実測: 赤 (216,61,54)・青 (63,46,184)）とは明確に分離した独立クラスタとして保存されるため、検出目標色を実測値に合わせることで検出は成立する。第二に、出力メッシュのスケールは概ねメートル単位であり、スケール補正を要しなかった。

## 3.4 M3: 関節検出とリンク分割

M3 は GLB メッシュから関節位置を検出し、メッシュをリンク単位に分割する（`glb_to_links.py`）。処理は次の手順による。

**(1) 頂点色の復元。** Tripo3D の GLB は色情報を UV テクスチャとして持つため、まずテクスチャを頂点色に焼き込む（`to_color()` 変換）。この工程を欠くと頂点色が常に空となり、検出が無条件に失敗する。

**(2) マーカー頂点の抽出。** 目標色 $c_0 = (r_0, g_0, b_0)$ と許容幅 $\tau$（既定 60）に対し、頂点 $i$ の色 $c_i$ が各チャネル独立に

$$|c_i^{(k)} - c_0^{(k)}| \le \tau \quad (k \in \{R, G, B\})$$

を満たすものをマーカー頂点とする。座標系を Y-up から Z-up へ変換（$(x,y,z) \mapsto (x,-z,y)$）した上で処理する。

**(3) 関節位置のクラスタリング。** マーカー頂点の Z 座標を昇順に走査し、隣接値の差が閾値（0.05 m）を超えた位置で区切る 1 次元クラスタリングを行う。各クラスタの Z 平均を関節高さとする。このとき、頂点数がマーカー頂点総数の 2% に満たないクラスタは棄却する — テクスチャの色滲みが生む少数の迷い頂点（実測: 1 頂点）が偽の関節を作ることを防ぐためである。関節の XY 位置は、当該 Z 近傍のマーカー頂点の重心として求める。実証実験では 3 関節すべてが自動検出された（Z = −0.298, 0.001, 0.298）。

**(4) リンク分割。** 検出した関節 Z を境界とし、各フェースをその重心の Z 値で区分してメッシュを分割する。境界間にフェースが存在しない場合は関節位置の異常としてエラーを返す（フェイルファスト）。分割片は各リンクのローカル座標に平行移動され、STL として出力される。最下段の分割片は固定台座として扱い、可動リンクから除外できる。

**(5) 姿勢の補正（オプション）。** エンドエフェクタの面の向きが運動方向と不整合な場合（実測例: ヘラの面法線がスイング平面と垂直で、薄い側面が押し方向を向く「チョップ」姿勢）、指定リンクをボーン軸まわりに回転させる補正を適用できる。

## 3.5 M4: 形態パラメータ抽出

分割された各リンク STL から、co-design の設計変数の初期値を抽出する（`mesh_to_params.py`）。リンクごとに OBB（Oriented Bounding Box、有向外接直方体）を計算し、

- **リンク長** $L$: OBB の最長辺の長さ。ボーンベクトル bone_offset の大きさとなる
- **カプセル半径** $r$: 残る2辺の長さの平均の半分、$r = (e_2 + e_3)/4$

を得る。OBB を用いることでメッシュの置かれた向きに依存せずに主軸長が取れる。実証実験では、マーカー球がリンク本体と分離して抽出されるため OBB が球に引きずられて太ることがなく、妥当な半径（0.033〜0.045 m）が得られた。抽出された bone_offset はタスク座標の X 軸方向へマッピングされ、設計空間の探索範囲（bone_offset は初期値相対 ±0.5 m、ギア比は [20, 400]）が付与された topology.json として出力される。関節可動域とギア比の初期値は現状パイプラインの既定値を用いる。[要チェック: 可動域・ギアを「スケッチからは決まらない量」として限界節で明示するか、プロンプト規約に将来組み込むと書くか]

## 3.6 M5: シミュレーションモデル生成

topology.json は MuJoCo 互換 XML へ変換される（`topology_to_xml.py`）。各リンクはカプセルとして近似され、カプセルの両端は bone_offset から計算される（始端 = 原点近傍、終端 = bone_offset）。子リンクの取り付け位置は親リンクの bone_offset 終端に一致させ、関節（回転軸・可動域）とアクチュエータ（ギア比）を付与する。生成された XML は、2関節の手書きモデル（rrbot_arm）と同一の書式であり、co-design フレームワークからは通常のロボットモデルとして扱われる。実証実験では、3 関節メッシュ由来のモデルが**コード変更ゼロ**で学習系に受理された — 形態を可変長のグラフとして扱う Transformer ベースの方策ネットワークが、関節数の変化を追加ノードとして透過的に処理するためである。

## 3.7 M6: co-design による評価・補正と判定

最後に、生成されたモデルを初期形態として Stackelberg PPO による形態-制御 co-design を実行する。エピソードは3フェーズ — 骨格変換（本システムではスキップ: トポロジーはスケッチが決めるため固定）、属性変換（Leader が bone_offset・ギア比を編集）、実行（Follower が制御を実行しタスク報酬を得る）— から成り、Leader の目的関数には Follower の実行リターンが継承される。したがって学習の進行に伴い、(i) 実行リターンの推移が**その形態系列のタスク達成度の評価**を与え、(ii) Leader による設計パラメータの更新が**スケッチ由来の初期値の補正**を与える。

判定の高速化のため、学習は初期状態からではなく、同一タスクの学習済み run から**制御方策ネットワークのみを転用**（control_prior）して開始する。転用成分の選択は第6章の分解実験に基づく — 全成分の転用は形態ネットワークの持ち込みが探索を歪めるため短期ではかえって有害であり、制御のみの転用が最良である。判定の読み取りは学習曲線の順位に基づき、複数候補の順位確定には約 100 エポック、明らかに筋の悪い候補の除外には約 75 エポックを要する（第6章）。（空欄: 前向き検証の結果 — 良い/悪い形態を意図的に用意した判定実験）

なお、パイプライン全体（M3〜M5）は単一のシェルスクリプトで一括実行でき、実証実験では画像 → GLB → XML → 学習開始までを人手の介在なしに通している（第7章相当の実証は骨子 7.2〜7.3 の実験計画に従う）。（空欄: 生成メッシュでの本格 co-design 学習・手描きスケッチでの E2E 実走の結果）
