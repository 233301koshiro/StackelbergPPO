# タスク設計と報酬関数：元リポジトリとの比較・改変記録

## 概要

元の StackelbergPPO の pusher タスクは「箱を前方に押す」という同じ目標でも、**報酬関数の設計思想が形態 co-design 向けに特化している**。  
本ページでは元設計の意図を考察した上で、固定根本アームへの移植で生じた問題と、独自改変（目標座標報酬）の設計根拠を記録する。

---

## TL;DR（このページの流れ・現在地）

1. **1〜2節**: 元の報酬（速度ベース・mobile base 前提）を固定根本アームに移植したら「野球バット戦術」（接触一撃で吹き飛ばす）が出現
2. **3〜4節**: 目標座標ベース報酬を導入するも、penetration（貫通）退化戦略が発覚・対策
3. **5〜6節**: 形態最適化が「アームをcubeに向けて静止させるだけ」で報酬を稼ぐ第3の reward hacking を発見 → PBRS（Δφ形式）で解消
4. **7〜9節**: PBRS後も探索シグナル不足でcubeが動かない問題に対し、Leader専用の到達可能性ボーナス`compute_reach_bonus()`を追加（案A→案B＝半径×角度→`arm_safe_init`の+90°回転補正）。**ここが現在の最新実装**
5. **10節**: 報酬の汎化性への懸念（このアーム専用にチューニングしていないか）を受け、リンク比率の異なる`rrbot_arm2`での並行実験を開始（現在進行中）

**現在地**: 9節の修正版（v5-4）と、10節の汎化テスト（rrbot_arm2_cnoid_v1）が並走中。**cubeはまだどちらの run でも動いていない**。次の判断ポイントは両 run の比較結果待ち。

## 1. 元リポジトリの報酬設計と意図

### 報酬式

```python
# 元の pusher.py
reward_fwd_cube = (xposafter - xposbefore) / self.dt          # cube の x 方向速度
reward_fwd_cube -= 0.1 * abs(yposafter - yposbefore) / self.dt  # y 方向逸脱ペナルティ
reward_ctrl = -0.1 * np.sum(action ** 2)                       # アクションコスト
```

「**cube を x 方向に速く動かせ**」という速度ベースの報酬。到達目標座標は指定しない。

### なぜ速度報酬なのか

**（1）形態最適化との相性：どんな形態でも成立するスケール**

morphology co-design では学習中にアームのリンク長・構造が変化する。  
距離報酬 `1/(1 + dist(cube, target))` は事前に target を固定する必要があるが、

- 短い形態期：cube がせいぜい x=0.8 まで → 報酬の天井 `1/(1+0.7) ≈ 0.59` で頭打ち → 形態を伸ばす勾配が弱い  
- 速度報酬：押せた量に比例して報酬が出る → 形態の能力成長に常に勾配が生まれる

速度報酬は「どれだけ押せる形態か」という能力そのものを測定しており、**暗黙的なカリキュラム**として機能する。

**（2）Dense かつ常に有意なシグナル**

速度報酬はすべてのタイムステップで有意な値を返す。  
距離報酬では cube が遠い初期状態ほど勾配が薄く、学習初期のシグナルが乏しい。

**（3）タスク定義が「能力」であって「特定到達点」ではない**

研究目的は「より良い形態を co-design できるか」の実証。  
`target_x=1.5 に正確に置く` という特定の到達点よりも、「cube に大きな運動量を与える能力」を測定する方が形態比較として汎用的。

**（4）モバイルベースとの組み合わせ前提**

元実装のロボットは**自由関節付きモバイルベース**（位置が動ける）。

- ロボットが cube を少し押す → cube が転がる → ロボットが追従して再び押す、という**持続的押し**が成立する
- アクションコスト `−0.1*|ctrl|²` のため、一撃で大きな力を出すより**小さい力で何度も押し続ける**方が効率的
- 結果として「バット戦術」より「継続押し」が高報酬になる設計

## 2. 固定根本アームへの移植で生じた問題（野球バット戦術）

### rrbot_arm の構成

本プロジェクトでは元のモバイルベースロボットの代わりに **2 関節・固定根本アーム**を使用する。

```
world → body_0 (固定ベース, z=0.15) → body_1 (肩, ±60°) → body_11 (肘, ±90°)
```

### 何が起きたか

`eval_cnoid_visual.py` による可視化で、アームが**野球バット型スイング**を学習していることが判明した。

- 肘をほぼ 0°（直線伸ばし）に固定し、肩を高速回転させて cube を一撃する
- cube が x 方向に高速で吹き飛び、速度報酬が最大化される
- 持続的接触は行わない

### 根本原因

固定根本アームでは **cube を追従できない**。  
バット戦術で一撃したあと cube は転がっていくが、アームは届かない。  
→「持続押し」が物理的に不可能なため、速度を最大化する唯一の戦略がバットスイングになる。

さらに形態最適化も絡む：
- 接触報酬 `1/(1 + dist(cube, arm_tip))` によりアームリンクが cube に届くよう伸長される
- 長くなったアームをバットのように振る挙動が強化される → リンク長 × 角速度 = 先端速度を最大化

```
tip_velocity = ω × (L_shoulder + L_elbow)  # リンクを直線展開したときの先端速度
```

**元リポジトリではアクションコスト + ベース追従コストにより batting が不利だったが、固定根本アームにはその抑制がない。**

## 3. 独自改変：目標座標ベース報酬

### 設計方針

バット戦術を抑制するため、「cube を特定座標の近くに維持する」報酬に変更した。

```python
# design_opt/envs/pusher.py（use_target_reward: true のとき）
dist_to_target = np.linalg.norm(
    np.array([xposafter, yposafter]) - np.array([target_x, target_y]))
reward_fwd_cube = 1.0 / (1.0 + dist_to_target)
```

| 性質 | 説明 |
|------|------|
| 値域 | `(0, 1]` — 常に有界 |
| 最大 | 1.0（cube が target に一致） |
| バット抑制 | cube を target を超えて飛ばすと dist が大きくなり報酬が下がる |
| 手前止まり抑制 | target より手前で止まっても dist が大きく報酬が下がる |

### target 座標の設定根拠と変遷

`rrbot_arm.xml` の実際のキューブ幾何：

| パラメータ | 値 |
|---|---|
| cube body pos | x=1.0m（XML `pos="1.0 0 0.20"`） |
| cube half_size | 0.15m → **左面 x=0.85m**（qpos=0 時） |
| cube_x_offset=0.5 適用後 | cube 中心 x=1.5m、左面 x=1.35m |
| アーム bone_offset 上限 | x 成分最大 0.5m/段 → 2段で x 方向最大 1.0m |
| デフォルト先端リーチ | body_1(0.30m) + body_11(0.25m) = 0.55m（x 方向） |
| cube damping | 2.0（cube_fixed スライド関節の減衰） |

**v3/v4 の target_x=1.5 の問題**：`cube_x_offset=0.5` を加えると cube 中心が x=1.5m になり、`target_x=1.5` と一致する。cube はエピソード開始時点から目標位置にあるため、アームが何もしなくても最大報酬を得てしまった（v4 epoch 50 で確認）。

**v5 での修正**：`target_x=2.0` に変更。cube が x=1.5（中心）から x=2.0 まで 0.5m 押す必要がある。

### 速度報酬との比較

| 観点 | 速度報酬（元） | 目標座標報酬（改変） |
|------|-------------|-------------------|
| 形態汎用性 | 高：どんなサイズでも成立 | 低：target がリーチ内か依存 |
| バット抑制 | なし | あり（overshoot = 低報酬） |
| 学習スケール | 無制限（固定根本では 10000+/ep） | 有界（最大 ~1000/ep） |
| タスク定義 | 押す能力の最大化 | 特定座標への制御 |
| 固定根本アームとの相性 | 悪（batting を誘発） | 良（持続的押し・目標維持を誘発） |

### 設定方法

`choreonoid_train.py` 実行時に Hydra フラグで指定する（`pusher.yml` への記載は不要）：

```bash
choreonoid --no-window --python scripts/choreonoid_train.py \
    cfg=pusher xml_name=rrbot_arm \
    +reward_specs.use_target_reward=true \
    +reward_specs.target_x=1.5 \
    +reward_specs.target_y=0.0
```

デフォルト（`use_target_reward` 未指定）は元の速度報酬のまま動作する（後方互換）。

> **注意**: `design_opt/utils/config.py` の `reward_specs` は FLAG オーバーライドを明示的にマージする必要がある。  
> 修正済み（2026-06-24）: `self.reward_specs.update(FLAG.get('reward_specs', dict()) or {})`  
> 修正前は FLAG フラグが無視され、速度報酬のまま動いていた（→ `docs/進捗.md` 参照）。

## 4. penetration インパルス問題と cube 初期位置の修正（2026-06-24）

### 発見した問題：複数 run での退化戦略

v2・v3・v4 のすべてで物理エンジンの挙動を悪用する **Reward Hacking** が発生した。

#### v2（速度報酬・前腕 penetration）

v2 eval 動画で「アーム静止・cube が飛ぶ」現象を確認。骨格デバッグ print により以下が判明：

```
cube 幾何: center x=1.0m（body pos）, half_size=0.15m → 左面 x=0.85m, 右面 x=1.15m, y:[-0.15,+0.15]

v2 骨格最適化後:
  body_1  bone_offset: x=+0.497, y=−0.498  （斜め -45°）
  body_11 bone_offset: x=+0.494, y=−0.498

joints=0 の初期状態でのリンク位置:
  elbow: (0.830, −0.018)   ← cube 左面 x=0.85 の直前
  tip:   (1.323, −0.517)

前腕（elbow→tip 線分）が cube を横切る:
  t=0.04 → (0.851, −0.038)  ← cube 内部に入る
  t=0.27 → (0.963, −0.150)  ← cube 下端で出る
```

→ エピソード開始直後、物理エンジンが侵入解消インパルスを発生 → **cube 飛翔**。
先端ではなく**前腕中間部**が刺さっており、動画では step 1 時点でアームが跳ね返った後の状態が映っていた。

収束理由：
- `reward_fwd_contact = 1/(1+dist(cube, arm_tip))` がアームを cube 方向へ誘導
- 形態最適化が「初期姿勢で前腕が cube 内部を横切るリンク方向・長さ」に収束
- 毎エピソード確実に impulse → 高い cube 速度 → 速度報酬を回収

#### v3（目標座標報酬・類似の penetration 疑惑）

v3 eval 動画で「アーム静止・cube がジリジリ進む」を確認。v3 のボーン詳細データはないが、v2 と類似の penetration インパルス退化が起きた可能性が高い。

#### v4（cube が最初から target にある）

`cube_x_offset=0.5` で cube 中心を x=1.5m に移動したが、`target_x=1.5` と偶然一致した。  
アームは何もしなくても `reward_target = 1/(1+0) = 1.0/step` を得られるため、アームが縮んで静止する退化戦略に収束（epoch 50 で確認、exec_R_eps≈785 が高止まり）。

### 対処：cube_x_offset + target_x の修正

`design_opt/envs/pusher.py` の `reset_state` に `cube_x_offset` / `cube_x_noise` を追加。

```python
# env_specs で設定可能:
#   cube_x_offset=0.5  → cube 左面 = 0.85 + 0.5 = 1.35m（デフォルト arm reach 0.55m を大幅に超える）
#   cube_x_noise=0.2   → ±0.2m ランダム化 → 固定位置戦略を防止
cube_x_idx = self.model.nq - 2   # [j1, j11, cube_x, cube_y] の cube_x
qpos[cube_x_idx] = init_cube_x + cube_x_offset + noise
```

**offset=0.5 の根拠**：
- cube 中心 = 1.0 + 0.5 = **1.5m**、左面 = **1.35m**
- デフォルト arm 先端 = 0.55m → **0.80m のギャップ → 初期 penetration 不可能**
- arm を積極的に伸ばして動かさないと cube に接触できない

**v4→v5 の追加修正**：`target_x` を cube 初期位置（1.5m）より遠い **2.0m** に設定。cube を 0.5m 積極的に押す必要がある。

## 5. 第3の報酬ハッキング：形態最適化による静止近接搾取とポテンシャルシェーピング修正（2026-06-26）

### 発見した問題

v2-2・v5 の学習中（epoch 460 前後）、動画では cube もアームも静止しているのに `exec_R_eps` が上昇し続けた。

**原因**: `reward_fwd_contact = 1/(1 + dist(arm_tip, cube))` は、**arm_tip が cube に近ければ毎ステップ定数として蓄積される**。
形態最適化（Leader）が bone_offset を調整して初期姿勢で arm_tip を cube 付近に常駐させることで、制御ポリシー（Follower）が何もしなくても高報酬を得られる。

```
初期状態（arm_safe_init, shoulder=π/2, cube at x=1.5m）:
  arm_tip ≈ (0, 0.45) → dist ≈ 1.57m → reward_fwd_contact ≈ 0.39/step
  → 1000 step episode = 390/ep の静止ベースライン

v2-2 epoch 287: exec_R_eps=587, arm-cube 距離 ≈ 0.70m（縮んでいる証拠）
v5 epoch 170:   exec_R_eps=1240, cube-target reward 0.667 + contact 0.573
```

Leader が「arm_tip を cube に向けたまま固定」できる骨格を学習 → cube 変位はほぼゼロ。

### 元の設計思想との整合

元の `reward_fwd_contact` の意図は：
- **Follower 誘導**: arm_tip を cube に近づける探索シグナル
- **Leader 誘導**: arm が cube に届く形態を育てる

これは「形態が cube に届く能力を持つか」を測定するという観点では正しい。  
問題は「**静止状態でも毎ステップ蓄積される**」ことであり、報酬の方向性ではなく**積算形式**が搾取の原因。

### 修正：ポテンシャルベース報酬シェーピング（PBRS）

Ng et al. (1999) の理論に基づき、ポテンシャル関数 φ(s) を定義して**その差分のみを報酬とする**。

```python
# 旧（静止搾取される）
reward_fwd_contact = 1.0 / (1.0 + dist(arm_tip, cube))   # ∈ [0, 1]

# 新（potential shaping: φ(t+1) - φ(t)）
φ(s) = 1.0 / (1.0 + dist(arm_tip, cube))                 # ポテンシャル関数（形は同じ）
reward_fwd_contact = φ(s_{t+1}) - φ(s_t)                 # ∈ [-1, +1]
```

| ケース | 旧報酬 | 新報酬 |
|---|---|---|
| arm_tip が cube 付近で静止 | +0.39〜/step（累積） | **0** |
| arm_tip が cube に近づく | 小さな正の変化 | **+**（近づいた分だけ） |
| arm_tip が cube から遠ざかる | 小さな負の変化 | **−** |
| 形態変化で arm_tip が一歩近い位置に | 静止報酬として蓄積 | **その 1 歩分のみ** |

### 設計思想が維持される理由

Stackelberg 構造で考えると：
- **Follower** の最適行動が「arm を swing して cube に近づける」になる → 動かさないと reward = 0
- **Leader** は「Follower が速く大きく swing できる骨格」で高報酬 → **到達・接触能力の高い形態**を育てる

PBRS は最適ポリシーを変えないことが数学的に証明されており（Ng 1999）、報酬の方向性は元設計と同一。**「近づける能力」への評価はそのまま残り、「静止近接の搾取」だけが塞がれる**。

### 実装（`design_opt/envs/pusher.py`）

```python
# __init__: ポテンシャルキャッシュを初期化
self.prev_contact_potential = None

# transit_execution(): エピソード開始時の φ を記録
arm_ref_body = self.robot.bodies[-1].name if self.is_fixed_base else "0"
dist0 = np.linalg.norm(self.get_body_com("cube") - self.get_body_com(arm_ref_body))
self.prev_contact_potential = 1.0 / (1.0 + dist0)

# step() 実行フェーズ: Δφ を報酬として使う
curr_dist = np.linalg.norm(self.get_body_com("cube") - self.get_body_com(arm_ref_body))
curr_contact_potential = 1.0 / (1.0 + curr_dist)
contact_weight = self.cfg.reward_specs.get('contact_weight', 1.0)  # デフォルト 1.0
reward_fwd_contact = contact_weight * (curr_contact_potential - self.prev_contact_potential)
self.prev_contact_potential = curr_contact_potential
```

`contact_weight` を `reward_specs` から設定可能（デフォルト 1.0、変更不要）。

## 6. PBRS 適用後の実験結果と残存問題（2026-06-27 確認）

### v2-3・v5 の学習結果（epoch ~728 時点）

| run | reward_fwd_cube | reward_fwd_contact | exec_R_eps 推移 | 動作 |
|---|---|---|---|---|
| v2-3（速度） | `Δx/dt`（静止=0） | PBRS Δφ（静止=0） | ≈ 0 で横ばい | arm が cube を向いていない。cube 変位なし |
| v5（目標座標） | `1/(1+dist_target)`（**静止=0.667/step**） | PBRS Δφ（静止=0） | ≈ 667 で横ばい | cube 変位なし。静止ベースラインで頭打ち |

### v2-3 の形態（epoch best）

```
body_1  bone_length: 0.300 → 0.330m (+0.030)
body_11 bone_length: 0.250 → 0.622m (+0.372)
rest arm tip: (-0.176, -0.511)  ← cube (1.5, 0) と反対方向
```

arm が -x/-y 方向（cube と逆）に伸びている。PBRS 適用で静止ベースラインがゼロになった結果、**探索シグナルが完全にゼロ**のまま学習が進まない。

### v5 の形態（epoch best）

```
body_1  bone_length: 0.300 → 0.603m (+0.303)  ← 上限近く
body_11 bone_length: 0.250 → 0.700m (+0.450)  ← 上限近く
rest arm tip: (1.309, -0.603)  ← joints=0 でほぼ cube 方向（cube 左面 x=1.35m）
```

arm は joints=0 で最大限 cube 方向に伸びている。しかし **arm_safe_init（shoulder=π/2）によりエピソード開始時は +y 向き**に回転されるため、実際の初期 tip 位置は cube から離れる。

### PBRS 適用の誤解と実際の状況

「PBRS を適用した」のは `reward_fwd_contact` のみ。`reward_fwd_cube` は変更していない。

```python
# reward_fwd_cube (v5) ← PBRS 未適用、静止でも 0.667/step
reward_fwd_cube = 1.0 / (1.0 + dist_to_target)           # ← 静止ベースライン残存

# reward_fwd_contact ← こちらだけ PBRS 適用済み
reward_fwd_contact = contact_weight * (curr_contact_potential - self.prev_contact_potential)
```

これが v5 が exec_R_eps≈667 で全く改善しない直接原因。cube が動かなくても `1/(1+0.5)×1000=667` が毎エピソード保証される。

### 残存する問題まとめ

| 問題 | 対象 | 影響 |
|---|---|---|
| `reward_fwd_cube` の静止ベースライン | v5 | exec_R_eps=667 で収束・cube 動かず |
| 探索シグナル不足 | v2-3 | exec_R≈0 が続き、学習が進まない |
| arm_safe_init と形態最適化の方向不一致 | v5 | joints=0 で cube 方向に育ったアームが、init 時には +y 向きに強制回転される |

### 対策の方向性

`reward_fwd_cube` にも PBRS を適用することで、cube が動いたときだけ報酬が入る設計にする：

```python
# v5: cube-target 距離にも PBRS 適用
if use_target:
    curr_cube_potential = 1.0 / (1.0 + dist_to_target)
    if self.prev_cube_potential is None:
        self.prev_cube_potential = curr_cube_potential
    reward_fwd_cube = curr_cube_potential - self.prev_cube_potential
    self.prev_cube_potential = curr_cube_potential
```

ただし両報酬とも PBRS になると静止ベースラインが完全にゼロになる（v2-3 と同じ状況）。探索シグナルを補うために `alive_bonus` や `reward_approach_cube` の追加を検討する。

## 8. Leader/Follower の報酬「分離」 vs 「役割別の追加報酬」— 元論文との整合性検討（2026-06-29）

### 背景

7節までの対策（PBRS）で探索シグナルが消えた問題への対案として、「世間一般の2軸pusher報酬（dense distance, PBRSなし）をFollowerに使い、Leaderには形態だけから計算する到達可能性スコアを別途与える」という案を検討した。これは元論文（Dai et al., "Efficient Morphology-Control Co-Design via Stackelberg Proximal Policy Optimization", ICLR 2026, https://openreview.net/pdf?id=sJ0vOOkclw）の設計思想に反しないか確認した。

### 1. そもそも Leader と Follower は何を最適化しているか

- **Leader**（形態エージェント）: エピソード最初の `T` ステップだけ動き、bone_offset（リンク長・伸びる方向）を編集する。
- **Follower**（制御方策）: `T` ステップ後、確定した形態のもとで実際にアームを動かして cube を押す。

両者は別々のネットワークだが、評価される「成果」の一部は共有されている。これが論文の式（eq.5, eq.6）の意味。

### 2. Leader の成績はどう計算されるか（核心）

論文 eq.5 を日本語で言うと:

```
Leaderの成績 = (自分の設計フェーズ中の報酬 R^L) + (Followerがその後実際に稼ぐ報酬 R^F を、そのままコピーしたもの)
```

```
J^L = E[ Σ_{t=0}^{T-1} γ^t R^L(s_t^L, a_t^L)              ← Leader 自身の設計フェーズ報酬
        + Σ_{t=T}^∞    γ^{t-T} R^F(s_t^F, a_t^F; s_T^L) ]  ← Follower の長期リターンをそのまま継承

J^F = E[ Σ_{t=T}^∞ γ^{t-T} R^F(s_t^F, a_t^F; s_T^L) ]       ← Follower は後者のみ
```

Leader は自分では何もしていない実行フェーズについても、「自分が作った形態で Follower がどれだけうまくやれたか」で評価される。比喩で言うと「設計者の評価は、自分の設計コストだけでなく、その設計図を使って職人が実際にどれだけうまく作れたかにも連動する」。これにより Leader は「Follower がこの形態をどう使うか」を**先読み**して形態を決めるようになる——これが論文タイトルの "Stackelberg"（先導者が追従者の最適反応を見越して動く）の核心。

### 3. なぜ「完全分離」はNGか

検討した対案（「Leader には軌跡を一切見ない幾何的 reachability score だけ与える」）は、上の式の**後半項（Follower の報酬の継承）をまるごと削除して別物に置き換える**ことに相当する。そうすると Leader は「Follower が実際にどれだけ上手く押せたか」を一切見なくなり、ただ「理屈上届く形をしているか」だけで評価される。これは Follower の実際の挙動と無関係に Leader を評価することになり、「Follower の応答を見越して設計する」という前提（= Stackelberg である意味そのもの）を壊す。

### 4. なぜ「追加で上乗せ」はOKか

論文付録 E.7（Multi-objective and role-specific reward design）に明記あり:

> "the leader–follower decomposition allows reward terms to be **assigned selectively** to the structure-design or control-learning stages. For example, complexity or material-cost penalties can be applied **only to the leader** (structure) updates, **without interfering with controller learning**. This role-specific reward routing provides a high degree of flexibility for real-world design requirements."

つまり論文も「Leader 専用の追加報酬項」を **Follower の `R^F` 継承に上乗せする形で** 使うことを想定済み（例: 材料コスト・複雑さペナルティ）。式で見ると違いがはっきりする:

```
NG（分離）:  Leader成績 = Leader自身の報酬 ＋ 幾何reachability score（Followerの成果を見ない）
OK（上乗せ）: Leader成績 = Leader自身の報酬 ＋ 到達可能性ボーナス ＋ Followerの報酬（そのまま継承・変更なし）
```

「上乗せ」版では Leader は依然として「Follower が実際に上手くやれたか」を見ており、そこに「形態自体の特性（届きそうかどうか）」という別の評価軸を**追加情報として足す**だけ。Follower の報酬設計（押せたかどうかを測る部分）には一切触れないので、Follower の学習・探索勾配を壊さない。

### 5. 結論・採用方針

今回困っている現象（PBRS 適用後に cube へ届く勾配が消えた）は **Follower の報酬設計（reward_fwd_contact/cube）側の問題**。これを Leader 用の到達可能性ボーナスで置き換えるのではなく:

- Follower の報酬（R^F）は触らずそのまま残す（or 別途改善する場合も Leader 側の変更とは独立に検討する）
- **Leader の設計フェーズ報酬（R^L）に「到達可能性ボーナス」を追加項として乗せる**

という形にすれば、論文の枠組み内（role-specific reward routing）で「設計段階からどれくらい動けそうかを考慮する」という発想を、Stackelberg の前提（Leader が Follower の応答を見越す）を壊さずに実装できる。

### 次にやること（具体化が必要な項目）

1. 到達可能性ボーナスの具体式を決める（リンク長・`arm_safe_init` の初期回転を考慮した cube までの幾何的距離、もしくは到達可能半径との比較）
2. このボーナスを `R^L`（Leader の設計フェーズ報酬）のどこに・どう加算するか、コード上の挿入点を特定する（`design_opt/envs/pusher.py` の形態編集フェーズ報酬計算部分）
3. Follower 側の探索シグナル不足（5節・進捗.md 知見）への対策は本件と独立に検討を続ける

## 9. 到達可能性ボーナス `compute_reach_bonus()` の実装（2026-06-29）

8節の方針（「Follower の `R^F` 継承はそのまま、Leader の設計フェーズ報酬 `R^L` に到達可能性ボーナスを追加項として乗せる」）を実装した。**これは報酬の「分離」ではなく「追加」であり、元論文の Stackelberg 構造（Leader が Follower の実際の応答を見越して評価される、という前提）を壊さない設計変更である**ことを改めて明記しておく。

### 9.1 採用した式（案A: 半径のみ）

ユーザーと協議の上、まず簡易版（リンクの向き・関節可動域 ±90° を無視し、距離だけで届くかを判定する「半径のみ」案）から始めることにした。角度まで考慮する案B（より正確だが逆キネマティクス計算が必要）は次の拡張候補として保留。

```python
reach_min, reach_max = abs(L1 - L2), L1 + L2          # 2リンクアームの到達可能半径帯
excess = max(0.0, reach_min - d, d - reach_max)        # 帯の中なら 0
reach_bonus = reach_bonus_scale * exp(-reach_bonus_k * excess)
```

- `d` が `[|L1-L2|, L1+L2]` の中に入っていれば `excess=0` → ボーナス満額（`reach_bonus_scale`）
- 範囲外に出るほど指数的に滑らかに減衰（`reach_bonus_k` が減衰の鋭さ）
- 連続関数なので学習を阻害する不連続なジャンプがない

### 9.2 挿入箇所

`design_opt/envs/pusher.py` の `step()` 内、`attribute_transform` ステージが終わり `execution` に遷移する直前（旧: `reward = 0.0` 固定だった箇所）。論文 eq.5 の `R^L`（Leader 自身の設計フェーズ報酬、このリポジトリでは従来常に 0 だった）に相当する場所。

```python
elif self.stage == 'attribute_transform':
    ...
    reward = 0.0
    if self.cur_t == self.cfg.skel_transform_nsteps + 1:
        succ = self.transit_execution()
        ...
        reward = self.compute_reach_bonus()   # ← ここだけ変更。execution フェーズの reward 計算には一切触れていない
    ...
```

`execution` ステージ（Follower の `reward_fwd_cube` / `reward_fwd_contact` 計算）は **1行も変更していない**。

### 9.3 実装中に見つけたバグ: `get_body_com` は「サブツリー全体のCOM」を返す

最初の実装は `get_body_com(body名)` の差分でリンク長を求めようとした（`L1 = ||get_body_com(bodies[1]) − get_body_com(bodies[0])||` 等）。デバッグ出力を仕込んで1エポックのスモークテストを行ったところ、**全サンプルで `L1` が厳密に `0.000`** になるという不自然な結果が出た。

原因を調査した結果、`get_body_com()` は対象ボディ単体の重心ではなく **そのボディを根とする部分木（自分＋すべての子孫）の重心**を返す実装だった（既存コードの `arm_ref_body` 系のロジックもこれを前提にしている）。`bodies[0]`（root, 非アクチュエータの固定マウント）はこのリポジトリの初期形態で `bone_offset=0`・質量もほぼ無いため、`get_body_com(bodies[0])`（＝部分木{0,1,11}全体の重心）と `get_body_com(bodies[1])`（＝部分木{1,11}の重心）がほぼ一致してしまい、差分が常に0になっていた。

**修正**: リンク長は `body.bone_offset`（設計パラメータそのもの、`khrylib/robot/xml_robot.py` で `bone_end - bone_start` として保持される値）を直接読む方式に変更。肩のピボット位置（ワールド座標）は `get_body_com` ではなく `self.data.body_xpos[self.model._body_name2id[name]]`（ボディの**フレーム原点**、部分木重心ではない）を使用。既存コードの `get_sim_obs()` 内 (`design_opt/envs/pusher.py` の `root_offset` 計算箇所) で同じ `body_xpos` パターンが使われており、これに倣った。

```python
def compute_reach_bonus(self):
    scale = self.cfg.reward_specs.get('reach_bonus_scale', 0.0)
    if scale == 0.0 or not self.is_fixed_base or len(self.robot.bodies) < 3:
        return 0.0
    bodies = self.robot.bodies
    L1 = float(np.linalg.norm(np.asarray(bodies[1].bone_offset, dtype=float)))
    L2 = float(np.linalg.norm(np.asarray(bodies[-1].bone_offset, dtype=float)))
    shoulder_xy = self.data.body_xpos[self.model._body_name2id[bodies[1].name]][:2]
    cube_xy = self.get_body_com("cube")[:2]
    d = np.linalg.norm(cube_xy - shoulder_xy)
    reach_min, reach_max = abs(L1 - L2), L1 + L2
    excess = max(0.0, reach_min - d, d - reach_max)
    k = self.cfg.reward_specs.get('reach_bonus_k', 3.0)
    return scale * np.exp(-k * excess)
```

修正後、同じスモークテストで `L1`/`L2` がエピソードごとに 0.1〜0.6m 程度で妥当に変動し、ボーナスも 0.07〜0.43 の範囲で滑らかに変化することを確認した。

### 9.4 設定キーと後方互換性

| キー | デフォルト | 説明 |
|---|---|---|
| `reward_specs.reach_bonus_scale` | `0.0`（無効） | ボーナス満額時の値。明示的にオプトインしないと既存runの挙動は変わらない |
| `reward_specs.reach_bonus_k` | `3.0` | 半径項: 範囲外での減衰の鋭さ |
| `reward_specs.reach_bonus_angle_power` | `2.0` | 角度項: 9.7節で追加。大きいほど方向条件が厳しくなる |

`is_fixed_base=False`（移動ベース型ロボット）または `robot.bodies` が3未満の場合は常に `0.0` を返し、pusher 以外の環境・形態には影響しない。

### 9.5 既知の限界（次の拡張候補）

- 関節可動域（±90°×2関節）による角度方向の制約を無視している（9.1 の案B）。「半径的には届くが逆方向を向いている」ケース（v2-3・v5-2 で実際に発生した問題）を直接は検出できない可能性がある
- `cube_x_noise` によるエピソードごとのランダム化は考慮しているが（実際にサンプリングされた cube 位置を使用）、複数 noise サンプルにわたる「ロバストな到達可能性」の平均は取っていない
- `reach_bonus_scale` / `reach_bonus_k` の値は未チューニング（v2-4・v5-3 で動かしながら調整する）

### 9.6 v2-4・v5-3 中間結果（epoch139、2026-06-29）

`reach_bonus_scale=2.0, reach_bonus_k=3.0` で v2-3/v5-2 の後継として起動（`docs/進捗.md` 参照）。epoch139 時点で `eval_cnoid_visual.py` の動画・rest arm tip 座標を確認した。

| run | 報酬構成 | rest arm tip（epoch139） | train_R_eps の挙動 | 観察 |
|---|---|---|---|---|
| v2-4 | 速度ベース + PBRS(contact) + reach_bonus | x=-0.80, y=-1.19（cube 逆方向） | 0.15〜0.41 の間で不安定、時々大きな負スパイク（-3.49, -12.73, -23.91 等） | v2-3 と同じ問題（cube と逆方向に成長）が reach_bonus 追加後も継続 |
| v5-3 | 目標座標 + PBRS(contact+cube) + reach_bonus | x=1.02, y=0.17（cube 方向 = +x、**v5-2 から方向が反転して改善**） | 0.3 前後→epoch139 で 1.19〜1.36 に上昇・安定 | 方向は正しくなったが arm の総リーチ（L1+L2）がまだ cube までの距離に対して不足。cube 変位は両 run ともまだゼロ |

**速度ベース報酬 (v2-4) と目標座標報酬 (v5-3) で reach_bonus への応答が明確に分かれた**: v5-3 は Leader の学習信号（train_R_eps）が時間とともに滑らかに上昇し、形態が正しい方向に成長したのに対し、v2-4 は信号が不安定なまま方向の改善が見られない。原因はまだ特定していないが、仮説としては:

- v2-3/v2-4 系の速度ベース報酬（`reward_fwd_cube = Δx/dt`）は、cube に触れてもいないと一切信号が出ない（接触するまで完全に 0）。reach_bonus だけでは Follower 側の探索シグナル不足（5節・進捗.md 既知の問題）を補えていない可能性がある
- v5-2/v5-3 系の目標座標報酬は PBRS 化されていてもなお `reward_fwd_cube` の構造上、reach_bonus と相性が良い可能性がある（要検証）

**次の確認ポイント**: epoch300〜500 で再度動画確認。v5-3 の arm が実際に cube に届く長さまで伸びるか、v2-4 が逆方向のまま停滞し続けるかを見る。v2-4 が改善しない場合、速度ベース報酬系列そのものの探索シグナル不足を別途対策する必要がある。

### 9.7 v2-4 停止と案B（半径×角度）の実装（2026-06-30）

epoch~407 まで進めた v2-4 は `best.p` が約12時間（6/29 13:55 以降）更新されず、動画でも arm が依然 cube 逆方向（-x）のまま停滞していたため停止した。v5-3 も `best.p` が約7時間（6/29 18:29 以降）停滞していたが、rest tip 座標は x=1.024→1.258 と伸び続けており、形態側は前進していたため継続中（v2-4 のみキャンセル）。

**診断**: 9.1 で「まず案Aから」としていた reach_bonus（半径のみ）は、**`L1`/`L2` を `np.linalg.norm()` で長さだけ見ており、方向には一切インセンティブを与えていなかった**。v2-4 が方向を直せなかったのは、9.5 で明記していた既知の限界（「半径的には届くが逆方向を向いているケースを直接は検出できない」）がそのまま現実化したもの。v5-3 の方向改善は reach_bonus 自体の効果ではなく、目標座標報酬の PBRS（`reward_fwd_cube`）が間接的に方向学習を後押しした可能性が高い（v2-4 の速度ベース報酬 `Δx/dt` は接触までシグナルが完全に 0 のため、この間接効果が働かなかったと推測）。n=1 の比較のため因果は確定的ではないが、症状（v2-4 だけが完全に停滞）と整合する。

**対策（案B採用）**: `compute_reach_bonus()` を「半径項 × 角度項」に拡張。

```python
# 角度項: rest ポーズ（関節角0）の方向が cube を向いているか
rest_tip_xy = shoulder_xy + bo1_vec + bo11_vec   # bone_offset をベクトルのまま使う（長さだけでなく方向も）
to_cube = cube_xy - shoulder_xy
to_tip  = rest_tip_xy - shoulder_xy
cos_angle = dot(to_cube, to_tip) / (|to_cube| * |to_tip|)
angle_term = max(0, cos_angle) ** reach_bonus_angle_power   # 逆方向(cos<0)なら 0

reach_bonus = scale * radius_term * angle_term   # 半径と方向の両方を満たして初めて満額
```

- `rest_tip_xy` は eval_cnoid_visual.py の `build_rest_arm()` と同じ定義（回転行列を使わないベクトル単純加算）。`arm_safe_init` は実行時に**アクチュエータ付き関節だけ**を回転させるため、無関節の body "0" の位置（= shoulder pivot）には影響しない。よってこの「rest 方向」は `arm_safe_init` の回転と独立に、Leader の設計ネットワーク自身が制御する方向を表す
- `cos_angle < 0`（完全に逆方向）なら `angle_term=0` となり、半径条件をいくら満たしても bonus がゼロになる設計（v2-4 のようなケースを直接弾く）
- 厳密な逆キネマティクス（関節可動域 ±90° の正確な判定）は採用しなかった。コサイン角度による近似は実装が簡単な一方、「ほぼ届くが微妙に角度が足りない」ケースを過剰に落とす可能性がある（次の課題）
- 新設定キー: `reward_specs.reach_bonus_angle_power`（デフォルト `2.0`）。大きいほど方向条件が厳しくなる

**検証**: スモークテストで、意図的に「v2-4 の実際の rest tip 座標」を模した逆方向ケースを手計算したところ `cos_angle=-0.71` → `angle_term=0` となることを確認。1エポックの実機テストでも `cos_angle` が 0.6〜0.99 の範囲で変動し、エラーなく完走することを確認済み（デバッグ出力は最終実装からは削除）。

**v2-4・v5-3 への適用**: v5-3 は起動済みプロセスのため、この変更は反映されない（Python は起動時に一度だけモジュールを読み込むため）。新しい案B入りで再起動する場合は v2-5・v5-4 等として立ち上げる。

### 9.8 `arm_safe_init` 回転を角度項に反映する修正（2026-06-30）

9.7 の案Bは「design rest 姿勢（関節角0）」の方向を使って角度を評価していたが、`arm_safe_init=true`（v2-4・v5-3 とも有効）の場合、実際に Follower が対峙する初期姿勢は **`reset_state()` が常に `qpos[0] = π/2` を強制する**ため、design rest 方向からさらに +90° 回転した状態でスタートする（`reset_state()` 内のコメント参照: "Prevents penetration-impulse exploit... at qpos[0]=π/2 the arm always starts pointing away from cube, so no initial overlap regardless of arm length"）。

9.7 の実装はこの回転を無視していたため、**Leaderが「design空間で正しい方向」に最適化しても、Followerが実際に対峙する方向とは無関係というズレ**があった。これが v5-3 で「rest tip は cube に近づいているのに実行動画では届いていない」という観察の一因と考えられる。

**修正**: `compute_reach_bonus()` の角度項で、`arm_safe_init=true` の場合のみ `bo1+bo11` を +90°（`qpos[0]` と同じ回転）だけ回転させてから cube 方向との角度を比較する。

```python
link_vec = bo1 + bo11
if self.env_specs.get('arm_safe_init', False):
    theta = np.pi / 2
    rot = np.array([[np.cos(theta), -np.sin(theta)], [np.sin(theta), np.cos(theta)]])
    link_vec = rot @ link_vec
rest_tip_xy = shoulder_xy + link_vec
```

`body "0"`（無関節）は回転しないため shoulder pivot 自体は影響を受けない。回転対象は `bo1+bo11`（関節1で回転させられる部分）のみ。

**検証**: `transit_execution()` の `reset_state(True)` 直後に、実際の `body_xpos`（シミュレータが計算した本当のelbow位置）と、修正後の予測値・修正前の予測値を並べて出力する一時デバッグコードを仕込み、1エポックのスモークテストで比較した。

```
real_elbow=[0.498315 0.269472] pred_corrected(+90)=[0.49831538 0.26947198] pred_uncorrected=[ 0.26947198 -0.49831538]
```

**修正後の予測値が実際の `body_xpos` と小数点5桁以上で一致**し、回転方向・符号が正しいことを実機検証で確認した（手計算だけでなく、シミュレータの実値と直接比較したことで符号ミスのリスクを排除）。デバッグコードは検証後に削除済み。

**意義**: これで `compute_reach_bonus()` の角度項は、Leaderの設計空間ではなく「Followerが実際に対峙する初期姿勢」を正しく評価するようになった。v2 系（速度ベース戦術）は現在保留中のため、この修正は実質的に v5 系（目標座標戦略）のみに適用される。次に v5-4 として再起動する際にこの修正版を使う。

### 報酬の汎化性への懸念（ユーザー提起、2026-06-30）

ここまでの作業（v2→v5、PBRS、reach_bonus、角度補正…）は、特定の1体（固定根本2リンクアーム `rrbot_arm`）に対する**手作業の報酬チューニングの積み重ね**になっており、「この2軸アーム専用の報酬」が見つかっても、ロボットの形態が変わるたびに報酬を再設計する必要があるなら、Stackelberg PPO（co-design）の意義自体が薄れるのではないか、という懸念が提起された。

**整理した論点**:
- 元論文の設計思想は「報酬はタスクに紐づき、形態には紐づかない」（論文 Appendix で "unified forward-progress reward" を全環境で使い回している）。形態を探索するのが Leader の仕事であり、報酬まで形態ごとに作り直すのは本来の co-design の前提に反する
- `compute_reach_bonus()` は現状 **2リンク決め打ち**（`bodies[1]`, `bodies[-1]` のみ参照、半径帯の式 `[|L1-L2|, L1+L2]` も2リンク専用）。3軸・4軸への拡張は可能（N本リンクの到達可能半径帯は `reach_max=ΣLi`, `reach_min=max(0, 2*max(Li)-ΣLi)` に一般化できる、角度項もベクトル和をN本に増やすだけ）が、現状のコードは未対応
- 拡張順序として: (1) まず同じ2自由度内でリンク比率が異なる別アームに対して**コード変更なし**で同じ報酬が機能するかを検証 → (2) その結果を見てからN本リンクへの一般化（リファクタリング）に投資するかを判断、という段階的アプローチを採用

| | rrbot_arm（元） | rrbot_arm2（新） |
|---|---|---|
| link1（"1"、肩→肘） | 0.30 | **0.15**（短い） |
| link2（"11"、肘→先端） | 0.25 | **0.45**（長い） |
| デフォルト合計リーチ | 0.55m | 0.60m（ほぼ同等） |

合計リーチはほぼ揃えつつ、「どちらの関節が主に動く必要があるか」という運動学的な性質を大きく変えている。body名（"0","1","11"）・joint名（"1_joint","11_joint"）は `rrbot_arm.xml` と同一構造を維持し、`compute_reach_bonus()` 等の既存コードを**一切変更せず**にそのまま使える（2リンク決め打ちのコードと完全互換）。

起動コマンド: `xml_name=rrbot_arm2` 以外は v5-4 と全く同じ設定（`hydra.run.dir=single_run/rrbot_arm2_cnoid_v1`）。

**狙い**: 同じ報酬コードが、追加のチューニングなしでリンク比率の異なるアームでも cube を押す方向に学習を進められるか観察する。v5-4（元のrrbot_arm）と比較し、両方とも同程度に進む／詰まるなら「この問題は報酬よりタスク設定（cube距離・arm_safe_init等）由来」、片方だけ進むなら「報酬がアーム形状に依存している」ことの手がかりになる。

## 11. 重大バグ発見・修正: ロボット自身の関節が damping・joint_range ともに無視されていた（2026-07-01）

### きっかけ

「v5-4 が515エポック動かして一度もcubeに触れない」事象についてユーザーから「本当にコードのバグではないと言い切れるか」と問われ、`mean_action`方策の制御出力と`qpos`の推移を毎ステップ細かく確認したところ、`epoch=300`チェックポイントで以下を発見:

```
step  20: qpos1=0.209  tip_dist=0.83633  ctrl1≈0.00000
step 100: qpos1=1.630  tip_dist=0.83633  ctrl1≈0.00001
step 195: qpos1=3.317  tip_dist=0.83633  ctrl1≈0.00000
```

肘関節`qpos1`が制御出力ほぼゼロのまま毎ステップ律儀に増え続けるのに、cubeとの距離`tip_dist`は完全に不変——`qpos`が実体と乖離している疑いが生じた。

### 制御実験での確認

`PusherEnv`を直接構築し、肩関節に8ステップだけ`ctrl=1.0`のトルクパルスを与え、以後`ctrl=0`で215ステップ放置。`qpos0`・`qvel0`に加えて、`body_xmat`（実際の物理シミュレーション結果のリンク回転行列）から独立に計算した`real_angle`も比較した。

```
step  0: ctrl=1.00  qpos0=0.219   qvel0=5.94   real_angle=0.219
step  7: ctrl=1.00  qpos0=7.96    qvel0=47.86  real_angle=1.678
step  8: ctrl=0.00  qpos0=9.88    qvel0=47.86  real_angle=-2.691
step215: ctrl=0.00  qpos0=406.16  qvel0=47.86  real_angle=-2.243   ← 65回転以上
```

`real_angle`は`qpos0`を2πで割った余りと正確に一致——**`qpos`の異常ではなく、実際に物理シミュレーション上でアームが永久に回転し続けていた**。関節可動域`range="-90 90"`（±π/2 ≈ ±1.5708）は完全に無視されていた。

### 原因

2つの独立したバグが重なっていた。

**バグ1: hinge joint用のYAML生成時に`joint_damping`が一切書き出されていなかった**

`khrylib/rl/envs/common/mujoco_env_choreonoid.py` の `mujoco_xml_to_body()` 内、`process_one_joint()` の hinge（revolute）分岐が `damping=` を渡していなかった（prismatic分岐は渡していた）:

```python
# 修正前（hinge）: damping を一切渡していない
elif jtype_mj == 'hinge':
    ...
    add_link(child_name, parent_name, 'revolute', jname,
             axis, rng_deg, translation, mass, com, inr, shape)   # damping 省略

# 修正前（prismatic、参考）: こちらは渡していた
elif jtype_mj in ('slide', 'prismatic'):
    damping = float(j_el.get('damping', default_damping))
    add_link(..., damping=damping)
```

`add_link()` は `damping=None` がデフォルトのため、YAML出力時（`if lk.get('damping') is not None:` のガード）でhinge jointの`joint_damping`行が**そもそも生成されていなかった**。これは6/25に見つけた「Choreonoid AISTがprismatic jointのdampingを無視するバグ」の修正時、cube（prismatic）だけに対処してhingeを見落としていたもの。

**バグ2: 既存の減衰回避ロジックが「ロボット本体」を明示的に除外していた**

同ファイル `ChoreonoidSimWorld._build_sim_body_entries()` で、YAMLから読んだdampingを使い `j.dq` を毎ティック減衰させる仕組みは存在したが、

```python
for entry_idx, (sb, _) in enumerate(entries[1:], start=1):  # ← [1:] でロボット本体(entries[0])を除外
```

`entries[1:]`（"extra body" = cube のみ）にしか適用されておらず、**ロボット自身の関節（肩・肘）には一度も適用されていなかった**。`_load_body_defs()` 側の damping パース処理にも `if i > 0:` という同根の除外があった。

**さらに、joint_range（関節可動域）はそもそも誰も読んでいなかった**: YAML には `joint_range: [-90, 90]` が正しく書き出されていたが、それを読んでシミュレーション中にクランプする処理が存在しなかった。Choreonoid AISTは（少なくとも本構成では）revolute jointのrangeを物理エンジン側で強制してくれない。

### 修正

3箇所を修正:

1. **hinge jointにもdampingを渡す**（`process_one_joint()`）
2. **`_load_body_defs()`**: `i > 0` 制限を撤廃し、ロボット本体を含む全bodyについて `joint_damping` と `joint_range` の両方をYAMLからパース（`joint_range` はrevolute jointのみ度→ラジアン変換。prismaticは元々メートル単位）
3. **`_build_sim_body_entries()` / `step()`**: 全bodyについて減衰係数とjoint limitを事前計算し、`step()`の各物理ティックで `j.dq *= factor` による減衰に加え、`j.q` が範囲外なら `[lower, upper]` にクランプして `j.dq=0`、クランプが発生したbodyだけ `calcForwardKinematics()` で実リンク姿勢を同期。

変数名も `_extra_joint_damping`/`_extra_joint_decay`（"extra"限定を示唆する誤解を招く名前）から `_joint_damping`/`_joint_decay`/`_joint_limits`/`_joint_limits_compiled` に変更。

### 検証（修正後、同じ制御実験を再実行）

```
step  0: ctrl=1.00  qpos0=0.097  qvel0=3.45   real_angle=0.097
step  5: ctrl=1.00  qpos0=1.431  qvel0=5.69   real_angle=1.431   ← 終端速度に収束（無限加速しない）
step  6: ctrl=1.00  qpos0=1.5708 qvel0=0.0000 real_angle=1.5708  ← ちょうど90°でクランプ
step215: ctrl=0.00  qpos0=1.5708 qvel0=-0.0000 real_angle=1.5708 ← 215ステップ後も正確に90°のまま
```

dampingにより速度が無限加速せず終端速度で頭打ちになり、joint limitで正確に±90°にクランプされ、`real_angle`（実物理）と`qpos`が常に一致することを確認した。

### 影響範囲

**これまでの全実験（v2〜v5-4、rrbot_arm2_cnoid_v1）は、ロボットの関節が無制限に回転し続ける壊れた物理の上で行われていた。** `[normalize_obs] extreme-value rows clamped` という警告がセッション中ずっと出続けていたのも、この無制限に増加する`qpos`が原因だった可能性が高い。「Followerが探索信号不足で止まる」という分析（9節）自体は誤りではない可能性があるが、その分析の土台になった観測・学習結果が壊れた物理下のものだったため、**修正後に再実験して確認し直す必要がある**。

v5-4・rrbot_arm2_cnoid_v1 は起動済みプロセスのため、この修正は反映されていない（次回再起動時から有効）。

## 12. バグ発見後の知見再評価方針（2026-07-01）

11節のバグ発見を受け、「これまでの知見・報酬改造は全部やり直しなのか」を検討した。結論は **「一括では判断しない・カテゴリ分けして段階的に確認する」**。

### 12.1 これまでの知見への影響評価

バグの性質（**関節に一度トルクが加わると、本来あるはずの減衰・可動域制限が効かず無制限に回転し続ける**）を踏まえて仕分けた。

**🟢 バグと無関係・そのまま有効と考えられるもの**

| 知見 | 理由 |
|---|---|
| cube damping/質量/drift系のバグ（6/25） | cube（prismatic joint）の話。今回のhinge joint（アーム関節）バグとは別物で、修正は既に正しくcube側に適用済み |
| config.py の FLAG マージバグ（reward_specs/env_specs） | Hydra設定読み込みの話で物理エンジンと無関係 |
| choreonoidの終了処理バグ（os._exit） | プロセス終了の話で物理シミュレーションと無関係 |
| run_2axis_mvp.sh のパイプライン検証 | ファイル生成・学習起動の確認で、学習結果の中身は見ていない |
| 初期姿勢penetration exploit・`arm_safe_init`導入の判断 | エピソード開始時（t=0、まだ1ステップも動いていない瞬間）の静的な幾何チェック。その後関節が回転し続けるかどうかとは無関係 |

**🟡 部分的に再解釈が必要なもの**

| 知見 | 何が変わるか |
|---|---|
| 野球バット戦術（v2、最初期） | 「報酬ハッキングで一撃を狙う戦術を学習した」と説明していたが、無制限回転バグ自体が「振り回す」動きを物理的に増幅していた可能性がある。速度報酬が速度最大化を誘導するという論理自体はバグと無関係に成立するが、挙動の激しさはバグの寄与もあったかもしれない |
| `compute_reach_bonus()`の形態評価（rest tip座標など） | `body.bone_offset`という設計パラメータのみから計算しており、シミュレーション中の実際の関節角度を一切参照していない。Leaderへの報酬信号自体はバグの影響を受けていなかったと考えられる（コード上はそう読めるが未検証） |

**🔴 再検証必須のもの**

| 知見 | 理由 |
|---|---|
| 「Followerが探索信号不足で止まる」という結論（9節） | `ctrl→0`に収束する学習自体は本物に見えるが、その後アームが正しく静止していたのか、観測されないところで暴走回転していたのかが今回まで分かっていなかった |
| v5-3で見た「design上は届くのに実行時は届かない」というギャップ分析 | `arm_safe_init`の影響だけでなく、無制限回転バグの影響も混ざっていた可能性 |
| `[normalize_obs] extreme-value rows clamped`警告の原因 | 今回これが直接原因と判明。これまで「正規化の話」程度の扱いだった箇所の意味が変わる |

### 12.2 報酬改造をやり直すべきか

ユーザーから提起された懸念: 「バグが原因でアームが動かなかったのに、それを"報酬設計のせい"だと誤解して色々改造したのではないか。バグが治って動くようになるなら、報酬を1から見直すべきでは」。

これも一括では判断せず、改造を2種類に仕分けた。

**A. 「動かないこと」自体への対症療法だった可能性が高いもの**

- `compute_reach_bonus()`（到達可能性ボーナス、9節）— 「Leaderがcube方向に形態を育てない」という症状への対策として導入。Followerがまともに学習できていれば、形態最適化への勾配（Followerの実際の成果を継承する仕組み）はもっと素直に機能していた可能性がある
- `arm_safe_init`回転をreach_bonusの角度項に反映する補正（9.8節）— reach_bonus自体の補正なので、reach_bonusが不要ならこれも道連れで不要になる
- 「探索シグナル不足」という診断そのもの（12.1の🔴参照）

**B. バグと無関係に、独立した報酬設計の問題として正当だったもの**

- 目標座標報酬（バット戦術対策、2〜3節）— 速度報酬が「最大速度で叩き出す」ことを最適化してしまう問題は、物理が正常でも論理的に成立する
- PBRS（静止搾取対策、5〜6節）— `1/(1+dist)`を毎ステップ加算する設計だと、Leaderが「形態だけ近くに置いて動かない」ことで報酬を稼げてしまう問題は、関節が暴走しようがしまいが成立する
- penetration exploit対策（cube_x_offset・arm_safe_initの導入、4節）— エピソード開始時点の静的な幾何の話

### 12.3 採用方針: 今すぐ巻き戻さず、段階的に判断する

A を今すぐ剥がすと、それが実は今でも必要だった場合に逆戻りしてしまうリスクがある。そのため:

1. **まず現状（フル装備＋物理バグ修正版）の v5-5・rrbot_arm2_cnoid_v2 を走らせて様子を見る**
2. **もし cube が動き始めたら**: A に分類した中で本当に不要な複雑さ（reach_bonusの角度項やarm_safe_init補正など）が残っていないか、後から一つずつ外すアブレーションをして身軽にする価値がある
3. **もしそれでも cube が動かなければ**: A の蓄積が「壊れた物理への対症療法の積み重ね」だった疑いが強まるので、PBRSだけ（reach_bonusなし）まで一旦戻して、そこから何が本当に必要かを再構築する

「今すぐ1から作り直す」のではなく、**今の実験結果を見てから、本当に必要だった改造とそうでなかった改造を仕分ける**という段階的アプローチを採用する。

## 13. 探索ギャップ仮説と対照実験（2026-07-01）

### 仮説: cube が遠すぎて「近づく=報酬」に気づかない

Route C（ゼロアクション健全性チェック）で物理層は正常と確認済み。それでも cube が動かない。物理バグ修正後の dense diagnostic では arm が cube 方向へ実際に接近（tip_dist 1.109→1.068）していたが、`fwd_contact=0` が epoch437 まで続いた。

> cube が遠すぎると、多少アームが長くなっても「先端が近くなったら報酬が入る」という事実に Follower が気づかない可能性がある。

PBRS の contact 報酬は `Δφ = 1/(1+d_new) - 1/(1+d_old)`。cube-tip 距離 d=1.07m のとき、0.01m 近づいてもΔφ ≈ 0.00023 と極めて微小。学習バッチ内でランダム探索が偶然 cube 方向に動く確率も低く、その微小シグナルが平均化されて埋もれる。一方 ctrl_cost は毎ステップ確実に引かれるため、**「動かない方がctrl_costゼロで得」という局所最適に先に収束してしまう**構造的問題。

## 14. 初期接触exploit（Bug 6）の発見と対策（2026-07-01）

### 経緯

curriculum_v1（cube_x_offset=0.1）で epoch11 に fwd_cube が初めて非ゼロになった。**しかしアームは1mmも動いていなかった**（デバッグ戦記 Bug 6 参照）。

調査の結果、Leader が「arm_safe_init +90° 回転後の arm tip が cube 左面にぴったり触れる形態」を学習しており、エピソード開始直後の物理エンジン接触解消インパルスで cube が飛んでいた。

### 解決策: `_check_initial_contact()` による episode rejection

`transit_execution()` 内の `reset_state()` 直後に以下のチェックを追加:

```python
def _check_initial_contact(self):
    # arm_safe_init (+90°) 補正後の実際の先端位置を計算
    tip_xy = shoulder_xy + rot(π/2) @ (bo1 + bo11)
    cube_xy = get_body_com("cube")[:2]
    # cube のバウンディングボックス + arm 半径 + マージン
    if |tip_xy[0] - cube_xy[0]| < 0.23 and |tip_xy[1] - cube_xy[1]| < 0.23:
        return True  # 初期接触 → エピソードを打ち切る
```

Leader は再設計を余儀なくされ、initial contact exploit を学習できなくなる。`+env_specs.check_init_contact=false` で無効化可能（デフォルト有効）。

### 現状と示唆

| 手法 | 結果 | 示唆 |
|---|---|---|
| reach_bonus のみ（PBRS + 角度補正） | 初期接触exploitに収束 | reach_bonus が「エピソード開始時に触れる」最適解を作ってしまう |
| 初期接触チェック追加（curriculum_v2） | 学習中（結果待ち） | exploit なしで本物の push を学べるか |
| 速度ベース報酬（routeB_v1） | epoch26 で exec_R_eps=3.55 急上昇 | **物理が正常なら速度報酬で bat tactic が自然発生する**（cube は動く） |

**routeB_v1 の exec_R_eps=3.55 は「物理エンジン修正後に cube が実際に動く」ことの実証**であり、「目標座標報酬 or 物理エンジンのせいで動かない」問題は解消されていると判断できる。残る問題は「目標座標報酬で exploit なしに本物の push を学べるか」（curriculum_v2 で検証中）。

### 根本的なミスマッチの発見: 元リポジトリは「移動ロボット」前提

元の pusher.xml を解析すると、Pusher タスクのロボットは:
```
<joint name="root" type="free"/>  ← 体全体が自由に移動できる！
<body name="1/2/3/4">  ← 4本の脚（Crawler）
cube サイズ: 1辺 1.0m（巨大）
```

論文に明記: *"Pusher reuses the Crawler agent configuration in a 3D setting"*

**具体的なミスマッチ**:

| 観点 | 元 Pusher（Crawler型） | 我々の rrbot_arm |
|---|---|---|
| ベース | free joint（体ごと移動） | 固定（原点固定） |
| 探索勾配 | 常に存在（1歩前進 = 報酬増） | アームが届く配置になって初めて発生 |
| cube サイズ | **1辺 1.0m**（どこに当たっても押せる） | 1辺 0.3m（精密な接触が必要） |
| 観測ベクトル | 位置・クォータニオン・速度（意味ある情報） | **11個のゼロ埋め**（固定根本では無意味） |
| 近接報酬の基準 | ルートボディ（移動するので勾配あり） | アーム先端（届かないと勾配ほぼなし） |
| PBRS の必要性 | 体が動くので静止搾取しにくい | 静止アームが最適解になりやすい→PBRS 必須→勾配が消える |

### 結論と次のアクション

**「PBRS が必要な理由」自体が、固定根本アームという我々の設定から来ている**。元リポジトリはこの設定を想定していないため、報酬設計の全体的な見直しが必要。

採用方針（セクション 16 参照）:
1. **cube を大きくする**（元 Pusher に近い 1 辺 0.5〜1.0m）
2. **速度ベース報酬に戻す**（PBRS なし、元の `Δx/dt` 形式）
3. 大きな cube ならアームが静止していても「cubeに触れる」搾取が成立しにくく、PBRS 不要になる

## セクション 16: 実験管理方針 — 早期打ち切り基準

### 基本的な考え方

RL の早期打ち切りには 2 つの層がある：

- **タスク固有層**（変わらない）: 「何が成功か」の物理的定義。今回なら `fwd_cube > X m/step` の X はタスクの物理から決まる
- **実験役割層**（変わる）: その実験が「何を示すための実験か」によって、何を見れば情報が得られたかが変わる

**実験ごとに「この実験で何が分かれば十分か」を事前に決めておき、その情報が得られた時点で止める。**

### 汎用的な打ち切り基準（3 条件の AND）

```
打ち切り条件（AND）:
  1. fwd_cube < 0.01         （最低ラインすら届かない）
  2. 直近 50 epoch の平均上昇率 < 0.0002/epoch  （傾きがほぼゼロ）
  3. 上記が ep100 以降も継続  （序盤の不安定期を除外）
```

**fwd_cube > 0.1** は「成功確認基準」として持つが、打ち切り判断には使わない。  
打ち切りは「絶対値が低い」ではなく **「上昇率の消滅」** で判断する。

### 実験役割による基準の違い（例）

| 実験の役割 | 適切な打ち切りタイミング |
|-----------|------------------------|
| 「学習できるか」の検証（例: 実験 B） | 改善が止まった時点（停滞判定） |
| 「〇〇がなければどうか」の対照実験（例: 実験 C） | **横ばいを確認できた時点で目的達成** → 早めに止めてよい |
| 報酬設計の比較 | 両者の差が明確になった時点 |

対照実験は「学習できないことを示す」のが目的の場合がある。その場合、横ばいの確認 = 実験成功であり、B と同じ 1000 epoch 走らせる必要はない。

### 過去の実験への適用（反省）

過去の失敗実験（curriculum_v3、ctrl_offset05 等）は epoch 数の問題ではなかった。`fwd_cube = 0.000x` でゼロに張り付いており、傾きも存在しなかった。これは「勾配がない」状態であり、何 epoch 追加しても改善しない。打ち切り判断は正しかった。

**epoch 数の問題 vs. 設定の問題の見分け方**:
- `fwd_cube` が 0.001 → 0.050 と上昇中 → epoch 数の問題（まだ走らせる）
- `fwd_cube` が 0.001 → 0.001 で flat 100 epoch → 設定の問題（止めてよい）

### 最終結果

| 指標 | 実験 B（co-design） | 実験 C（制御のみ） |
|------|-------------------|-------------------|
| 最終 fwd_cube | **0.10〜0.12**（ep50 以降プラトー） | 0.01〜0.02（全期間横ばい） |
| 最終 exec_R_eps | 90〜116 | 10〜15 |
| 成功基準（>0.1）| ✅ ep50 で到達 | ❌ 未達 |
| 学習の傾き | ep50 でプラトー、ep139 まで改善なし | ep0 から横ばい |

**B/C 比 ≈ 8〜10 倍**。形態最適化なしでは制御だけでは学習が進まないことが対照実験で示された。

### B から得られた結論

1. **big cube（1m）+ 速度ベース報酬 + co-design の組み合わせで cube を動かせることが確認された**
2. **co-design（形態最適化）は制御学習の足場として機能する** — C との比較で定量的に示された
3. **joint range ±90° の制約下では、バットタクティック（3 ステップで全可動域を振り切り、その後制御停止）が最適戦略として収束する** — これは設計の限界であり、改善には joint range 拡張が必要（実験 D へ）
4. **co-design が学んだ形態の特徴**: 上腕 0.3m→0.7m に伸長、折り畳み形（スイングで前腕が +x 方向に展開）

B が答えた問いは「co-design で cube を動かせるか」であり、答えは YES。「より良い制御（持続押し）を学べるか」は実験 D の問いになった。

### 学習された戦略の分析

動画（Bug 7 修正後）と数値診断から、実験 B が学んだ戦略を特定した：

**バットタクティック（スイング押し）**

```
Step 0: アーム静止、cube 静止（q0≈0, vx=0）
Step 1: 肩が回転開始（q0≈0.26, vx≈0）
Step 2: 前腕が cube 方向にスイング → 接触（q0≈0.96, vx≈10 m/s）
Step 3: q0=π/2（joint limit）到達 → アーム固定（vx≈15 m/s でピーク）
Step 4〜200: アーム静止、cube が惰性で飛翔しながら減速
```

**形態の特徴**: 上腕が左後方（bone_offset≈[-0.49,-0.50]）、前腕が右前方（[+0.50,+0.50]）の「折り畳み形」。肩を π/2 まで回転させると前腕が一気に +x 方向に振られて cube を打つ。

**評価**: 物理的には正当な相互作用（アームが動いて cube を押した）。fwd_cube > 0.1 の成功基準も達成。ただし joint limit に当たった後は制御が完全に止まり、残り 197 ステップで報酬ゼロになる。「持続的な制御」ではない。

### co-design が何を最適化したか

Leader（形態最適化）が学んだこと：
- 上腕リンクを長く伸ばして（0.3m → 0.7m）スイング時の接触面積を拡大
- 折り畳み形にすることで、肩の π/2 回転で前腕が確実に +x を向く
- joint limit にぶつかる前に最大速度で cube を打てる形態

Follower（制御）が学んだこと：
- 最初の数ステップで肩関節に最大トルクを出して全力スイング
- joint limit 到達後は出力ゼロ（当然 cube には届かない）

### 次のステップ：継続的な押し戦略を学習させるために

現状の問題: joint range が -90° 〜 +90°（π/2）しかなく、一度振り切ると戻れない。

対応方針（セクション 18 参照、実装済み）:
1. **joint range 拡張**（-180° 〜 +180°）: 限界に当たってから引いて再度押せる
2. **関節速度制限**（3.0 rad/s）: 高速スイングによる衝撃的 impulse を抑制し、持続的な接触を促す

## セクション 18: joint range 拡張 + 速度制限（実装済み）

**問題**: joint range が ±90° のため 3 ステップで上限に達し、残り 197 ステップで制御不能。  
**解決**: range 拡張 + 速度制限をセットで実装。

### 実装内容（コミット `01f2d08`）

```xml
<!-- rrbot_arm.xml 変更前 -->
<joint range="-90 90" type="hinge" />
<!-- rrbot_arm.xml 変更後 -->
<joint range="-180 180" type="hinge" velocity="3.0" />
```

- `mujoco_env_choreonoid.py` の `step()` 内で毎 tick 速度クランプを適用（`_joint_vel_limits_compiled` キャッシュ）
- 速度クランプは position クランプより先に適用（高速接触による impulse 防止が目的）

| 往復可能 | ✗（±90° 上限で固定） | ✓（±180° 範囲で継続運動可能） |

## セクション 19: 目標座標への cube 押しに向けたロードマップ

### 現在地（2026-07-03）

- **確認済み**: co-design あり（B）で cube を動かせる。バットタクティック（一撃離脱）が最適戦略として収束。
- **確認済み**: joint range ±180° + velocity cap 3.0 rad/s（実験 D）でバットタクティックを排除し、ゆっくりとした押しが可能。
- **未確認**: D が「押して引いて再度押す」往復戦略を学習できるか（ep8 時点では1回押しで停止）。

### 最終目標

> 指定した目標座標（例: x=3.0m）まで cube を押し込む。

---

### 超えるべき壁と想定問題

#### 壁1: 往復戦略の学習（最優先・実験D で確認中）

D の ep50〜100 時点で「押した後に引いて再度押す」動作が出るか。
出ない場合: joint range・velocity cap・報酬の再設計が必要。

#### 壁2: アームのリーチ制約

cube を1回押すと x≈1.82 に移動。そこからアームが届くかは co-design の形態進化次第。
B では ep50 でリンクが 0.7m+0.7m まで伸びた。D でも同様に伸びるか要確認。

#### 壁3: 目標座標報酬の搾取リスク

速度ベース報酬は「どこまで押すか」を指定できない。target 報酬（PBRS）に切り替えると静止搾取が再発する可能性がある（セクション5〜6 参照）。velocity cap + joint range 拡張で一撃離脱は防げているが、別形態の搾取が起きないか慎重に検証が必要。

#### 壁4: エピソード長不足

target=3.0m（初期 x=1.0 から +2.0m）まで押すには push 回数 ≈ 3 回以上必要。
1 cycle ≈ 75 step として最低 225 step → 現在の 200 step では不足する可能性。

#### 壁5: y 軸のずれ

現報酬は `Δx/dt - 0.1 * |Δy/dt|` で y ペナルティが軽い。目標座標への精密な押しでは y 軸制御も重要になる。

---

### 検証ルート（順序依存あり）

```
【現在】実験 D（ep8〜）学習中
        ↓ ep50〜100 を待つ
        
【検証1】D が往復戦略を学習したか確認
        YES → 次へ
        NO  → velocity cap・joint range・報酬の再調整

        ↓

        アームが 0.7m+ に伸びているか
        リーチが cube の移動後位置に届くか

        ↓

【実験E】episode 長延長（200 → 500 step）
        3 回以上の push cycle が出るか確認

        ↓

【実験F】target 報酬の導入（use_target_reward=true、target_x=1.5 など近め）
        静止搾取が再発しないか確認
        → 再発しない: target を段階的に延ばすカリキュラムへ

        ↓

【実験G】カリキュラム（target_x: 1.5 → 2.0 → 3.0）
        各段階でモデルを転移して学習

        ↓

        目標達成: 指定座標への cube 押し
```

### 並行作業の可否

**実験D の結果が出るまで（ep50〜100）は並行できる実験はない。**

E・F・G はいずれも D の結果（往復戦略の確認、形態のリーチ確認）を前提とする。D が失敗した場合は設定の再調整が先になるため、先行着手は無駄になるリスクが高い。

D の収束を待ちながらできること:
- 可視化スクリプトの改善
- スケッチ生成メッシュ → arm 形態への pipeline 整備（Level 0 とは独立して進められる）

## セクション 20: 実験 D/E 結果と次の方針

### 実験 D（rrbot_arm + velocity cap + range ±180°）最終結果

**設定**: joint range ±180°、velocity cap 3.0 rad/s、速度ベース報酬（Δx/dt）  
**停止**: ep66（fwd_cube≈0.085 でプラトー）

| ep | fwd_cube | 状態 |
|----|----------|------|
| 0 | 0.002 | 初期 |
| 8 | 0.013 | アーム伸び始め |
| 50 | 0.072 | プラトー気味 |
| 66 | **0.085** | 停止（プラトー確認） |

**形態**: 上腕 0.3m → 0.706m、前腕 0.25m → 0.707m（B と同様の伸長）

**確認できたこと**:
- velocity cap (3.0 rad/s) により、バットタクティックが「3 ステップで π/2 到達」から「10 ステップかけてゆっくり倒れる」に変化
- cube が 3.3m 移動（B の約同等）したが、それ以降はアームが止まる
- **往復戦略（引いて再押し）は ep66 まで出現しなかった**

**往復が出ない根本原因**:
1. **幾何学的制約**: 1 回の押しで cube が x=4.23m に到達 → アームの最大リーチ 1.41m を大きく超える → 物理的に re-approach 不可能
2. **報酬の勾配**: `fwd_contact = Δ(1/(1+dist))` は引き戻し時にマイナスになるため RL が引き戻し自体を学習しない
3. **速度ベース報酬の限界**: Δx/dt を最大化しようとするため、cube を「ちょうど目標まで」押す理由がない

**結論**: velocity cap は bat tactic のスピードを緩和したが、「1 shot で止まる」パターンを根本的には解決できなかった。報酬の設計変更が必要。

### 実験 E（tripo_arm）初期結果

**設定**: 7 リンク・6 関節スケッチ生成アームを Stackelberg PPO 環境に適応  
**停止**: ep4（初期動作確認のため）

| ep | exec_R_eps | fwd_cube | 備考 |
|----|-----------|----------|------|
| 0 | 235 | 0.285 | 初期から大きな報酬（rrbot_arm ep0 の 47 倍） |

**確認できたこと**:

- D と同様に「前方に倒れて cube を押し、止まる」1-shot 戦略
- `fwd_contact = nan` が発生 → 分岐構造（branch 先端体 "21111"）の `get_body_com()` 計算に問題がある可能性（要調査）

### 次の方針: target reward（PBRS）への切り替え

velocity reward（Δx/dt）から target PBRS に切り替える。

**目的**: 「cube をちょうど target 座標まで押す」行動が最適になるよう報酬を再設計。cube が target を通過しすぎると reward がマイナスになるため、arm が精密にコントロールする動機が生まれる。

```python
# 新しい報酬
φ = 1 / (1 + dist(cube, target))
reward_fwd_cube = φ(t+1) - φ(t)
# 静止 → 0、target に近づく → 正、通り過ぎ → 負
```

**target_x 設定（近い目標から始めるカリキュラム）**:
- rrbot_arm: target_x = 1.5m（初期 cube x=1.0 から 0.5m 先）
- tripo_arm: target_x = 1.1m（初期 cube x=0.6 から 0.5m 先）

実験 D2（rrbot_arm + velocity cap + target reward）、E2（tripo_arm + target reward）として起動予定。

---

### RL における探索と報酬設計の一般論

「往復を見つかるまで待つ」のではなく「往復する理由を報酬で与える」ことが一般的。待つことが有効なのは、望ましい行動への報酬勾配がゼロでない場合のみ。今回は幾何学的制約により勾配がゼロであるため、報酬設計が先決。

## セクション 21: sparse reward 問題と報酬設計の進化

### D2/E2（target PBRS のみ）：短命で廃止

**D2**（rrbot_arm + target PBRS、target_x=1.5）: ep4 で fwd_cube=0.0002 と極めて低調。  
**E2**（tripo_arm + target PBRS、target_x=1.1）: ep3 で fwd_cube≈0 で変化なし。

**原因**: target PBRS は sparse reward の典型であり、cube が target 方向に動いたときにしか勾配が発生しない。序盤の探索がほぼゼロになる。

### sparse reward 問題の本質

目標座標への押しタスクには避けられないトレードオフが存在する：

```
速度報酬（Δx/dt）: cube が動けばどこでも報酬 → 探索容易 / 目標位置に止まれない
目標報酬（PBRS）: 目標に近づいたときのみ報酬 → 精密 / 探索困難
```

これは RL の「疎な報酬（sparse reward）問題」の典型であり、程度の差はあれ原理的に解消できない。対策として報酬設計で補う（待っても勾配がゼロのため学習しない）。

### 対策2案の設計

**① 組み合わせ報酬（`cube_target_weight` パラメータ）**  
速度報酬（探索）+ target PBRS（精密化）の重み付き合算：

```python
reward = vel_reward + cube_target_weight * target_pbrs
# vel_reward: Δx/dt（探索確保）
# target_pbrs: Δφ(target)（目標誘導）
```

- スクラッチから学習可能
- α・β のバランスが設計のカギ
- 実験 D3：cube_target_weight=1.0, target_x=1.5m

**② カリキュラム転移（velocity → target）**  
Phase 1（D: 速度報酬で「押す能力」習得）→ Phase 2（target PBRS で「どこまで押すか」習得）

- Phase 1 の checkpoint（arm 形態 + 押す方策）を初期値に転移
- Phase 2 は「習慣の上書き」のみが課題
- 実験 D4：D の best checkpoint から `reset_epoch=true` + target PBRS
- D のep66 は Phase 1 として十分（arm が 1.41m まで成長、push 能力確立済み）

### D3/D4 ep0 結果

| 実験 | 手法 | ep0 fwd_cube | ep0 exec_R_eps | 初期状態 |
|------|------|------------|--------------|--------|
| D3（①）| 組み合わせ報酬 | 0.0000 | 0.01 | スクラッチ |
| D4（②）| カリキュラム転移 | **-0.0004** | **-0.57** | D の best から転移 |

D4 が ep0 でマイナスになっているのは正常：D の方策は「全力で押す」習慣があり、target PBRS ではそれが cube の通り過ぎ → マイナス報酬として現れる。これを上書きして収束できるか ep10〜20 で判断。

### 参考: E（tripo_arm）の驚くべき ep5 結果

E（ep5）で `fwd_cube=0.943` という突出したスコアが出た。速度報酬で測ると tripo_arm は cube を非常に速く動かせる模様（7 リンク・高自由度により多方向からの押しが可能）。ただし `fwd_contact=nan`（分岐構造の先端 body "21111" で `get_body_com()` 計算エラー）の問題が残っており、contact reward が機能していない。target reward 実験（E2）でも同様の問題があったため E2 は廃止。tripo_arm の nan 問題は別途対処が必要。
