# タスク設計と報酬関数：元リポジトリとの比較・改変記録

## 概要

元の StackelbergPPO の pusher タスクは「箱を前方に押す」という同じ目標でも、**報酬関数の設計思想が形態 co-design 向けに特化している**。  
本ページでは元設計の意図を考察した上で、固定根本アームへの移植で生じた問題と、独自改変（目標座標報酬）の設計根拠を記録する。

---

## 1. 元リポジトリの報酬設計と意図

### 報酬式

```python
# 元の pusher.py
reward_fwd_cube = (xposafter - xposbefore) / self.dt          # cube の x 方向速度
reward_fwd_cube -= 0.1 * abs(yposafter - yposbefore) / self.dt  # y 方向逸脱ペナルティ
reward_ctrl = -0.1 * np.sum(action ** 2)                       # アクションコスト
```

「**cube を x 方向に速く動かせ**」という速度ベースの報酬。到達目標座標は指定しない。

### なぜ速度報酬なのか

**（1）形態最適化との相性：どんな形態でも成立するスケール**

morphology co-design では学習中にアームのリンク長・構造が変化する。  
距離報酬 `1/(1 + dist(cube, target))` は事前に target を固定する必要があるが、

- 短い形態期：cube がせいぜい x=0.8 まで → 報酬の天井 `1/(1+0.7) ≈ 0.59` で頭打ち → 形態を伸ばす勾配が弱い  
- 速度報酬：押せた量に比例して報酬が出る → 形態の能力成長に常に勾配が生まれる

速度報酬は「どれだけ押せる形態か」という能力そのものを測定しており、**暗黙的なカリキュラム**として機能する。

**（2）Dense かつ常に有意なシグナル**

速度報酬はすべてのタイムステップで有意な値を返す。  
距離報酬では cube が遠い初期状態ほど勾配が薄く、学習初期のシグナルが乏しい。

**（3）タスク定義が「能力」であって「特定到達点」ではない**

研究目的は「より良い形態を co-design できるか」の実証。  
`target_x=1.5 に正確に置く` という特定の到達点よりも、「cube に大きな運動量を与える能力」を測定する方が形態比較として汎用的。

**（4）モバイルベースとの組み合わせ前提**

元実装のロボットは**自由関節付きモバイルベース**（位置が動ける）。

- ロボットが cube を少し押す → cube が転がる → ロボットが追従して再び押す、という**持続的押し**が成立する
- アクションコスト `−0.1*|ctrl|²` のため、一撃で大きな力を出すより**小さい力で何度も押し続ける**方が効率的
- 結果として「バット戦術」より「継続押し」が高報酬になる設計

## 2. 固定根本アームへの移植で生じた問題（野球バット戦術）

### rrbot_arm の構成

本プロジェクトでは元のモバイルベースロボットの代わりに **2 関節・固定根本アーム**を使用する。

```
world → body_0 (固定ベース, z=0.15) → body_1 (肩, ±60°) → body_11 (肘, ±90°)
```

### 何が起きたか

`eval_cnoid_visual.py` による可視化で、アームが**野球バット型スイング**を学習していることが判明した。

- 肘をほぼ 0°（直線伸ばし）に固定し、肩を高速回転させて cube を一撃する
- cube が x 方向に高速で吹き飛び、速度報酬が最大化される
- 持続的接触は行わない

### 根本原因

固定根本アームでは **cube を追従できない**。  
バット戦術で一撃したあと cube は転がっていくが、アームは届かない。  
→「持続押し」が物理的に不可能なため、速度を最大化する唯一の戦略がバットスイングになる。

さらに形態最適化も絡む：
- 接触報酬 `1/(1 + dist(cube, arm_tip))` によりアームリンクが cube に届くよう伸長される
- 長くなったアームをバットのように振る挙動が強化される → リンク長 × 角速度 = 先端速度を最大化

```
tip_velocity = ω × (L_shoulder + L_elbow)  # リンクを直線展開したときの先端速度
```

**元リポジトリではアクションコスト + ベース追従コストにより batting が不利だったが、固定根本アームにはその抑制がない。**

## 3. 独自改変：目標座標ベース報酬

### 設計方針

バット戦術を抑制するため、「cube を特定座標の近くに維持する」報酬に変更した。

```python
# design_opt/envs/pusher.py（use_target_reward: true のとき）
dist_to_target = np.linalg.norm(
    np.array([xposafter, yposafter]) - np.array([target_x, target_y]))
reward_fwd_cube = 1.0 / (1.0 + dist_to_target)
```

| 性質 | 説明 |
|------|------|
| 値域 | `(0, 1]` — 常に有界 |
| 最大 | 1.0（cube が target に一致） |
| バット抑制 | cube を target を超えて飛ばすと dist が大きくなり報酬が下がる |
| 手前止まり抑制 | target より手前で止まっても dist が大きく報酬が下がる |

### target 座標の設定根拠と変遷

`rrbot_arm.xml` の実際のキューブ幾何：

| パラメータ | 値 |
|---|---|
| cube body pos | x=1.0m（XML `pos="1.0 0 0.20"`） |
| cube half_size | 0.15m → **左面 x=0.85m**（qpos=0 時） |
| cube_x_offset=0.5 適用後 | cube 中心 x=1.5m、左面 x=1.35m |
| アーム bone_offset 上限 | x 成分最大 0.5m/段 → 2段で x 方向最大 1.0m |
| デフォルト先端リーチ | body_1(0.30m) + body_11(0.25m) = 0.55m（x 方向） |
| cube damping | 2.0（cube_fixed スライド関節の減衰） |

**v3/v4 の target_x=1.5 の問題**：`cube_x_offset=0.5` を加えると cube 中心が x=1.5m になり、`target_x=1.5` と一致する。cube はエピソード開始時点から目標位置にあるため、アームが何もしなくても最大報酬を得てしまった（v4 epoch 50 で確認）。

**v5 での修正**：`target_x=2.0` に変更。cube が x=1.5（中心）から x=2.0 まで 0.5m 押す必要がある。

### 速度報酬との比較

| 観点 | 速度報酬（元） | 目標座標報酬（改変） |
|------|-------------|-------------------|
| 形態汎用性 | 高：どんなサイズでも成立 | 低：target がリーチ内か依存 |
| バット抑制 | なし | あり（overshoot = 低報酬） |
| 学習スケール | 無制限（固定根本では 10000+/ep） | 有界（最大 ~1000/ep） |
| タスク定義 | 押す能力の最大化 | 特定座標への制御 |
| 固定根本アームとの相性 | 悪（batting を誘発） | 良（持続的押し・目標維持を誘発） |

### 設定方法

`choreonoid_train.py` 実行時に Hydra フラグで指定する（`pusher.yml` への記載は不要）：

```bash
choreonoid --no-window --python scripts/choreonoid_train.py \
    cfg=pusher xml_name=rrbot_arm \
    +reward_specs.use_target_reward=true \
    +reward_specs.target_x=1.5 \
    +reward_specs.target_y=0.0
```

デフォルト（`use_target_reward` 未指定）は元の速度報酬のまま動作する（後方互換）。

> **注意**: `design_opt/utils/config.py` の `reward_specs` は FLAG オーバーライドを明示的にマージする必要がある。  
> 修正済み（2026-06-24）: `self.reward_specs.update(FLAG.get('reward_specs', dict()) or {})`  
> 修正前は FLAG フラグが無視され、速度報酬のまま動いていた（→ `docs/進捗.md` 参照）。

## 4. penetration インパルス問題と cube 初期位置の修正（2026-06-24）

### 発見した問題：複数 run での退化戦略

v2・v3・v4 のすべてで物理エンジンの挙動を悪用する **Reward Hacking** が発生した。

#### v2（速度報酬・前腕 penetration）

v2 eval 動画で「アーム静止・cube が飛ぶ」現象を確認。骨格デバッグ print により以下が判明：

```
cube 幾何: center x=1.0m（body pos）, half_size=0.15m → 左面 x=0.85m, 右面 x=1.15m, y:[-0.15,+0.15]

v2 骨格最適化後:
  body_1  bone_offset: x=+0.497, y=−0.498  （斜め -45°）
  body_11 bone_offset: x=+0.494, y=−0.498

joints=0 の初期状態でのリンク位置:
  elbow: (0.830, −0.018)   ← cube 左面 x=0.85 の直前
  tip:   (1.323, −0.517)

前腕（elbow→tip 線分）が cube を横切る:
  t=0.04 → (0.851, −0.038)  ← cube 内部に入る
  t=0.27 → (0.963, −0.150)  ← cube 下端で出る
```

→ エピソード開始直後、物理エンジンが侵入解消インパルスを発生 → **cube 飛翔**。
先端ではなく**前腕中間部**が刺さっており、動画では step 1 時点でアームが跳ね返った後の状態が映っていた。

収束理由：
- `reward_fwd_contact = 1/(1+dist(cube, arm_tip))` がアームを cube 方向へ誘導
- 形態最適化が「初期姿勢で前腕が cube 内部を横切るリンク方向・長さ」に収束
- 毎エピソード確実に impulse → 高い cube 速度 → 速度報酬を回収

#### v3（目標座標報酬・類似の penetration 疑惑）

v3 eval 動画で「アーム静止・cube がジリジリ進む」を確認。v3 のボーン詳細データはないが、v2 と類似の penetration インパルス退化が起きた可能性が高い。

#### v4（cube が最初から target にある）

`cube_x_offset=0.5` で cube 中心を x=1.5m に移動したが、`target_x=1.5` と偶然一致した。  
アームは何もしなくても `reward_target = 1/(1+0) = 1.0/step` を得られるため、アームが縮んで静止する退化戦略に収束（epoch 50 で確認、exec_R_eps≈785 が高止まり）。

### 対処：cube_x_offset + target_x の修正

`design_opt/envs/pusher.py` の `reset_state` に `cube_x_offset` / `cube_x_noise` を追加。

```python
# env_specs で設定可能:
#   cube_x_offset=0.5  → cube 左面 = 0.85 + 0.5 = 1.35m（デフォルト arm reach 0.55m を大幅に超える）
#   cube_x_noise=0.2   → ±0.2m ランダム化 → 固定位置戦略を防止
cube_x_idx = self.model.nq - 2   # [j1, j11, cube_x, cube_y] の cube_x
qpos[cube_x_idx] = init_cube_x + cube_x_offset + noise
```

**offset=0.5 の根拠**：
- cube 中心 = 1.0 + 0.5 = **1.5m**、左面 = **1.35m**
- デフォルト arm 先端 = 0.55m → **0.80m のギャップ → 初期 penetration 不可能**
- arm を積極的に伸ばして動かさないと cube に接触できない

**v4→v5 の追加修正**：`target_x` を cube 初期位置（1.5m）より遠い **2.0m** に設定。cube を 0.5m 積極的に押す必要がある。

## 5. 第3の報酬ハッキング：形態最適化による静止近接搾取とポテンシャルシェーピング修正（2026-06-26）

### 発見した問題

v2-2・v5 の学習中（epoch 460 前後）、動画では cube もアームも静止しているのに `exec_R_eps` が上昇し続けた。

**原因**: `reward_fwd_contact = 1/(1 + dist(arm_tip, cube))` は、**arm_tip が cube に近ければ毎ステップ定数として蓄積される**。
形態最適化（Leader）が bone_offset を調整して初期姿勢で arm_tip を cube 付近に常駐させることで、制御ポリシー（Follower）が何もしなくても高報酬を得られる。

```
初期状態（arm_safe_init, shoulder=π/2, cube at x=1.5m）:
  arm_tip ≈ (0, 0.45) → dist ≈ 1.57m → reward_fwd_contact ≈ 0.39/step
  → 1000 step episode = 390/ep の静止ベースライン

v2-2 epoch 287: exec_R_eps=587, arm-cube 距離 ≈ 0.70m（縮んでいる証拠）
v5 epoch 170:   exec_R_eps=1240, cube-target reward 0.667 + contact 0.573
```

Leader が「arm_tip を cube に向けたまま固定」できる骨格を学習 → cube 変位はほぼゼロ。

### 元の設計思想との整合

元の `reward_fwd_contact` の意図は：
- **Follower 誘導**: arm_tip を cube に近づける探索シグナル
- **Leader 誘導**: arm が cube に届く形態を育てる

これは「形態が cube に届く能力を持つか」を測定するという観点では正しい。  
問題は「**静止状態でも毎ステップ蓄積される**」ことであり、報酬の方向性ではなく**積算形式**が搾取の原因。

### 修正：ポテンシャルベース報酬シェーピング（PBRS）

Ng et al. (1999) の理論に基づき、ポテンシャル関数 φ(s) を定義して**その差分のみを報酬とする**。

```python
# 旧（静止搾取される）
reward_fwd_contact = 1.0 / (1.0 + dist(arm_tip, cube))   # ∈ [0, 1]

# 新（potential shaping: φ(t+1) - φ(t)）
φ(s) = 1.0 / (1.0 + dist(arm_tip, cube))                 # ポテンシャル関数（形は同じ）
reward_fwd_contact = φ(s_{t+1}) - φ(s_t)                 # ∈ [-1, +1]
```

| ケース | 旧報酬 | 新報酬 |
|---|---|---|
| arm_tip が cube 付近で静止 | +0.39〜/step（累積） | **0** |
| arm_tip が cube に近づく | 小さな正の変化 | **+**（近づいた分だけ） |
| arm_tip が cube から遠ざかる | 小さな負の変化 | **−** |
| 形態変化で arm_tip が一歩近い位置に | 静止報酬として蓄積 | **その 1 歩分のみ** |

### 設計思想が維持される理由

Stackelberg 構造で考えると：
- **Follower** の最適行動が「arm を swing して cube に近づける」になる → 動かさないと reward = 0
- **Leader** は「Follower が速く大きく swing できる骨格」で高報酬 → **到達・接触能力の高い形態**を育てる

PBRS は最適ポリシーを変えないことが数学的に証明されており（Ng 1999）、報酬の方向性は元設計と同一。**「近づける能力」への評価はそのまま残り、「静止近接の搾取」だけが塞がれる**。

### 実装（`design_opt/envs/pusher.py`）

```python
# __init__: ポテンシャルキャッシュを初期化
self.prev_contact_potential = None

# transit_execution(): エピソード開始時の φ を記録
arm_ref_body = self.robot.bodies[-1].name if self.is_fixed_base else "0"
dist0 = np.linalg.norm(self.get_body_com("cube") - self.get_body_com(arm_ref_body))
self.prev_contact_potential = 1.0 / (1.0 + dist0)

# step() 実行フェーズ: Δφ を報酬として使う
curr_dist = np.linalg.norm(self.get_body_com("cube") - self.get_body_com(arm_ref_body))
curr_contact_potential = 1.0 / (1.0 + curr_dist)
contact_weight = self.cfg.reward_specs.get('contact_weight', 1.0)  # デフォルト 1.0
reward_fwd_contact = contact_weight * (curr_contact_potential - self.prev_contact_potential)
self.prev_contact_potential = curr_contact_potential
```

`contact_weight` を `reward_specs` から設定可能（デフォルト 1.0、変更不要）。

## 6. PBRS 適用後の実験結果と残存問題（2026-06-27 確認）

### v2-3・v5 の学習結果（epoch ~728 時点）

| run | reward_fwd_cube | reward_fwd_contact | exec_R_eps 推移 | 動作 |
|---|---|---|---|---|
| v2-3（速度） | `Δx/dt`（静止=0） | PBRS Δφ（静止=0） | ≈ 0 で横ばい | arm が cube を向いていない。cube 変位なし |
| v5（目標座標） | `1/(1+dist_target)`（**静止=0.667/step**） | PBRS Δφ（静止=0） | ≈ 667 で横ばい | cube 変位なし。静止ベースラインで頭打ち |

### v2-3 の形態（epoch best）

```
body_1  bone_length: 0.300 → 0.330m (+0.030)
body_11 bone_length: 0.250 → 0.622m (+0.372)
rest arm tip: (-0.176, -0.511)  ← cube (1.5, 0) と反対方向
```

arm が -x/-y 方向（cube と逆）に伸びている。PBRS 適用で静止ベースラインがゼロになった結果、**探索シグナルが完全にゼロ**のまま学習が進まない。

### v5 の形態（epoch best）

```
body_1  bone_length: 0.300 → 0.603m (+0.303)  ← 上限近く
body_11 bone_length: 0.250 → 0.700m (+0.450)  ← 上限近く
rest arm tip: (1.309, -0.603)  ← joints=0 でほぼ cube 方向（cube 左面 x=1.35m）
```

arm は joints=0 で最大限 cube 方向に伸びている。しかし **arm_safe_init（shoulder=π/2）によりエピソード開始時は +y 向き**に回転されるため、実際の初期 tip 位置は cube から離れる。

### PBRS 適用の誤解と実際の状況

「PBRS を適用した」のは `reward_fwd_contact` のみ。`reward_fwd_cube` は変更していない。

```python
# reward_fwd_cube (v5) ← PBRS 未適用、静止でも 0.667/step
reward_fwd_cube = 1.0 / (1.0 + dist_to_target)           # ← 静止ベースライン残存

# reward_fwd_contact ← こちらだけ PBRS 適用済み
reward_fwd_contact = contact_weight * (curr_contact_potential - self.prev_contact_potential)
```

これが v5 が exec_R_eps≈667 で全く改善しない直接原因。cube が動かなくても `1/(1+0.5)×1000=667` が毎エピソード保証される。

### 残存する問題まとめ

| 問題 | 対象 | 影響 |
|---|---|---|
| `reward_fwd_cube` の静止ベースライン | v5 | exec_R_eps=667 で収束・cube 動かず |
| 探索シグナル不足 | v2-3 | exec_R≈0 が続き、学習が進まない |
| arm_safe_init と形態最適化の方向不一致 | v5 | joints=0 で cube 方向に育ったアームが、init 時には +y 向きに強制回転される |

### 対策の方向性

`reward_fwd_cube` にも PBRS を適用することで、cube が動いたときだけ報酬が入る設計にする：

```python
# v5: cube-target 距離にも PBRS 適用
if use_target:
    curr_cube_potential = 1.0 / (1.0 + dist_to_target)
    if self.prev_cube_potential is None:
        self.prev_cube_potential = curr_cube_potential
    reward_fwd_cube = curr_cube_potential - self.prev_cube_potential
    self.prev_cube_potential = curr_cube_potential
```

ただし両報酬とも PBRS になると静止ベースラインが完全にゼロになる（v2-3 と同じ状況）。探索シグナルを補うために `alive_bonus` や `reward_approach_cube` の追加を検討する。

## 8. Leader/Follower の報酬「分離」 vs 「役割別の追加報酬」— 元論文との整合性検討（2026-06-29）

### 背景

7節までの対策（PBRS）で探索シグナルが消えた問題への対案として、「世間一般の2軸pusher報酬（dense distance, PBRSなし）をFollowerに使い、Leaderには形態だけから計算する到達可能性スコアを別途与える」という案を検討した。これは元論文（Dai et al., "Efficient Morphology-Control Co-Design via Stackelberg Proximal Policy Optimization", ICLR 2026, https://openreview.net/pdf?id=sJ0vOOkclw）の設計思想に反しないか確認した。

### 1. そもそも Leader と Follower は何を最適化しているか

- **Leader**（形態エージェント）: エピソード最初の `T` ステップだけ動き、bone_offset（リンク長・伸びる方向）を編集する。
- **Follower**（制御方策）: `T` ステップ後、確定した形態のもとで実際にアームを動かして cube を押す。

両者は別々のネットワークだが、評価される「成果」の一部は共有されている。これが論文の式（eq.5, eq.6）の意味。

### 2. Leader の成績はどう計算されるか（核心）

論文 eq.5 を日本語で言うと:

```
Leaderの成績 = (自分の設計フェーズ中の報酬 R^L) + (Followerがその後実際に稼ぐ報酬 R^F を、そのままコピーしたもの)
```

```
J^L = E[ Σ_{t=0}^{T-1} γ^t R^L(s_t^L, a_t^L)              ← Leader 自身の設計フェーズ報酬
        + Σ_{t=T}^∞    γ^{t-T} R^F(s_t^F, a_t^F; s_T^L) ]  ← Follower の長期リターンをそのまま継承

J^F = E[ Σ_{t=T}^∞ γ^{t-T} R^F(s_t^F, a_t^F; s_T^L) ]       ← Follower は後者のみ
```

Leader は自分では何もしていない実行フェーズについても、「自分が作った形態で Follower がどれだけうまくやれたか」で評価される。比喩で言うと「設計者の評価は、自分の設計コストだけでなく、その設計図を使って職人が実際にどれだけうまく作れたかにも連動する」。これにより Leader は「Follower がこの形態をどう使うか」を**先読み**して形態を決めるようになる——これが論文タイトルの "Stackelberg"（先導者が追従者の最適反応を見越して動く）の核心。

### 3. なぜ「完全分離」はNGか

検討した対案（「Leader には軌跡を一切見ない幾何的 reachability score だけ与える」）は、上の式の**後半項（Follower の報酬の継承）をまるごと削除して別物に置き換える**ことに相当する。そうすると Leader は「Follower が実際にどれだけ上手く押せたか」を一切見なくなり、ただ「理屈上届く形をしているか」だけで評価される。これは Follower の実際の挙動と無関係に Leader を評価することになり、「Follower の応答を見越して設計する」という前提（= Stackelberg である意味そのもの）を壊す。

### 4. なぜ「追加で上乗せ」はOKか

論文付録 E.7（Multi-objective and role-specific reward design）に明記あり:

> "the leader–follower decomposition allows reward terms to be **assigned selectively** to the structure-design or control-learning stages. For example, complexity or material-cost penalties can be applied **only to the leader** (structure) updates, **without interfering with controller learning**. This role-specific reward routing provides a high degree of flexibility for real-world design requirements."

つまり論文も「Leader 専用の追加報酬項」を **Follower の `R^F` 継承に上乗せする形で** 使うことを想定済み（例: 材料コスト・複雑さペナルティ）。式で見ると違いがはっきりする:

```
NG（分離）:  Leader成績 = Leader自身の報酬 ＋ 幾何reachability score（Followerの成果を見ない）
OK（上乗せ）: Leader成績 = Leader自身の報酬 ＋ 到達可能性ボーナス ＋ Followerの報酬（そのまま継承・変更なし）
```

「上乗せ」版では Leader は依然として「Follower が実際に上手くやれたか」を見ており、そこに「形態自体の特性（届きそうかどうか）」という別の評価軸を**追加情報として足す**だけ。Follower の報酬設計（押せたかどうかを測る部分）には一切触れないので、Follower の学習・探索勾配を壊さない。

### 5. 結論・採用方針

今回困っている現象（PBRS 適用後に cube へ届く勾配が消えた）は **Follower の報酬設計（reward_fwd_contact/cube）側の問題**。これを Leader 用の到達可能性ボーナスで置き換えるのではなく:

- Follower の報酬（R^F）は触らずそのまま残す（or 別途改善する場合も Leader 側の変更とは独立に検討する）
- **Leader の設計フェーズ報酬（R^L）に「到達可能性ボーナス」を追加項として乗せる**

という形にすれば、論文の枠組み内（role-specific reward routing）で「設計段階からどれくらい動けそうかを考慮する」という発想を、Stackelberg の前提（Leader が Follower の応答を見越す）を壊さずに実装できる。

### 次にやること（具体化が必要な項目）

1. 到達可能性ボーナスの具体式を決める（リンク長・`arm_safe_init` の初期回転を考慮した cube までの幾何的距離、もしくは到達可能半径との比較）
2. このボーナスを `R^L`（Leader の設計フェーズ報酬）のどこに・どう加算するか、コード上の挿入点を特定する（`design_opt/envs/pusher.py` の形態編集フェーズ報酬計算部分）
3. Follower 側の探索シグナル不足（5節・進捗.md 知見）への対策は本件と独立に検討を続ける

## 9. 到達可能性ボーナス `compute_reach_bonus()` の実装（2026-06-29）

8節の方針（「Follower の `R^F` 継承はそのまま、Leader の設計フェーズ報酬 `R^L` に到達可能性ボーナスを追加項として乗せる」）を実装した。**これは報酬の「分離」ではなく「追加」であり、元論文の Stackelberg 構造（Leader が Follower の実際の応答を見越して評価される、という前提）を壊さない設計変更である**ことを改めて明記しておく。

### 9.1 採用した式（案A: 半径のみ）

ユーザーと協議の上、まず簡易版（リンクの向き・関節可動域 ±90° を無視し、距離だけで届くかを判定する「半径のみ」案）から始めることにした。角度まで考慮する案B（より正確だが逆キネマティクス計算が必要）は次の拡張候補として保留。

```python
reach_min, reach_max = abs(L1 - L2), L1 + L2          # 2リンクアームの到達可能半径帯
excess = max(0.0, reach_min - d, d - reach_max)        # 帯の中なら 0
reach_bonus = reach_bonus_scale * exp(-reach_bonus_k * excess)
```

- `d` が `[|L1-L2|, L1+L2]` の中に入っていれば `excess=0` → ボーナス満額（`reach_bonus_scale`）
- 範囲外に出るほど指数的に滑らかに減衰（`reach_bonus_k` が減衰の鋭さ）
- 連続関数なので学習を阻害する不連続なジャンプがない

### 9.2 挿入箇所

`design_opt/envs/pusher.py` の `step()` 内、`attribute_transform` ステージが終わり `execution` に遷移する直前（旧: `reward = 0.0` 固定だった箇所）。論文 eq.5 の `R^L`（Leader 自身の設計フェーズ報酬、このリポジトリでは従来常に 0 だった）に相当する場所。

```python
elif self.stage == 'attribute_transform':
    ...
    reward = 0.0
    if self.cur_t == self.cfg.skel_transform_nsteps + 1:
        succ = self.transit_execution()
        ...
        reward = self.compute_reach_bonus()   # ← ここだけ変更。execution フェーズの reward 計算には一切触れていない
    ...
```

`execution` ステージ（Follower の `reward_fwd_cube` / `reward_fwd_contact` 計算）は **1行も変更していない**。

### 9.3 実装中に見つけたバグ: `get_body_com` は「サブツリー全体のCOM」を返す

最初の実装は `get_body_com(body名)` の差分でリンク長を求めようとした（`L1 = ||get_body_com(bodies[1]) − get_body_com(bodies[0])||` 等）。デバッグ出力を仕込んで1エポックのスモークテストを行ったところ、**全サンプルで `L1` が厳密に `0.000`** になるという不自然な結果が出た。

原因を調査した結果、`get_body_com()` は対象ボディ単体の重心ではなく **そのボディを根とする部分木（自分＋すべての子孫）の重心**を返す実装だった（既存コードの `arm_ref_body` 系のロジックもこれを前提にしている）。`bodies[0]`（root, 非アクチュエータの固定マウント）はこのリポジトリの初期形態で `bone_offset=0`・質量もほぼ無いため、`get_body_com(bodies[0])`（＝部分木{0,1,11}全体の重心）と `get_body_com(bodies[1])`（＝部分木{1,11}の重心）がほぼ一致してしまい、差分が常に0になっていた。

**修正**: リンク長は `body.bone_offset`（設計パラメータそのもの、`khrylib/robot/xml_robot.py` で `bone_end - bone_start` として保持される値）を直接読む方式に変更。肩のピボット位置（ワールド座標）は `get_body_com` ではなく `self.data.body_xpos[self.model._body_name2id[name]]`（ボディの**フレーム原点**、部分木重心ではない）を使用。既存コードの `get_sim_obs()` 内 (`design_opt/envs/pusher.py` の `root_offset` 計算箇所) で同じ `body_xpos` パターンが使われており、これに倣った。

```python
def compute_reach_bonus(self):
    scale = self.cfg.reward_specs.get('reach_bonus_scale', 0.0)
    if scale == 0.0 or not self.is_fixed_base or len(self.robot.bodies) < 3:
        return 0.0
    bodies = self.robot.bodies
    L1 = float(np.linalg.norm(np.asarray(bodies[1].bone_offset, dtype=float)))
    L2 = float(np.linalg.norm(np.asarray(bodies[-1].bone_offset, dtype=float)))
    shoulder_xy = self.data.body_xpos[self.model._body_name2id[bodies[1].name]][:2]
    cube_xy = self.get_body_com("cube")[:2]
    d = np.linalg.norm(cube_xy - shoulder_xy)
    reach_min, reach_max = abs(L1 - L2), L1 + L2
    excess = max(0.0, reach_min - d, d - reach_max)
    k = self.cfg.reward_specs.get('reach_bonus_k', 3.0)
    return scale * np.exp(-k * excess)
```

修正後、同じスモークテストで `L1`/`L2` がエピソードごとに 0.1〜0.6m 程度で妥当に変動し、ボーナスも 0.07〜0.43 の範囲で滑らかに変化することを確認した。

### 9.4 設定キーと後方互換性

| キー | デフォルト | 説明 |
|---|---|---|
| `reward_specs.reach_bonus_scale` | `0.0`（無効） | ボーナス満額時の値。明示的にオプトインしないと既存runの挙動は変わらない |
| `reward_specs.reach_bonus_k` | `3.0` | 範囲外での減衰の鋭さ |

`is_fixed_base=False`（移動ベース型ロボット）または `robot.bodies` が3未満の場合は常に `0.0` を返し、pusher 以外の環境・形態には影響しない。

### 9.5 既知の限界（次の拡張候補）

- 関節可動域（±90°×2関節）による角度方向の制約を無視している（9.1 の案B）。「半径的には届くが逆方向を向いている」ケース（v2-3・v5-2 で実際に発生した問題）を直接は検出できない可能性がある
- `cube_x_noise` によるエピソードごとのランダム化は考慮しているが（実際にサンプリングされた cube 位置を使用）、複数 noise サンプルにわたる「ロバストな到達可能性」の平均は取っていない
- `reach_bonus_scale` / `reach_bonus_k` の値は未チューニング（v2-4・v5-3 で動かしながら調整する）

### 9.6 v2-4・v5-3 中間結果（epoch139、2026-06-29）

`reach_bonus_scale=2.0, reach_bonus_k=3.0` で v2-3/v5-2 の後継として起動（`docs/進捗.md` 参照）。epoch139 時点で `eval_cnoid_visual.py` の動画・rest arm tip 座標を確認した。

| run | 報酬構成 | rest arm tip（epoch139） | train_R_eps の挙動 | 観察 |
|---|---|---|---|---|
| v2-4 | 速度ベース + PBRS(contact) + reach_bonus | x=-0.80, y=-1.19（cube 逆方向） | 0.15〜0.41 の間で不安定、時々大きな負スパイク（-3.49, -12.73, -23.91 等） | v2-3 と同じ問題（cube と逆方向に成長）が reach_bonus 追加後も継続 |
| v5-3 | 目標座標 + PBRS(contact+cube) + reach_bonus | x=1.02, y=0.17（cube 方向 = +x、**v5-2 から方向が反転して改善**） | 0.3 前後→epoch139 で 1.19〜1.36 に上昇・安定 | 方向は正しくなったが arm の総リーチ（L1+L2）がまだ cube までの距離に対して不足。cube 変位は両 run ともまだゼロ |

**速度ベース報酬 (v2-4) と目標座標報酬 (v5-3) で reach_bonus への応答が明確に分かれた**: v5-3 は Leader の学習信号（train_R_eps）が時間とともに滑らかに上昇し、形態が正しい方向に成長したのに対し、v2-4 は信号が不安定なまま方向の改善が見られない。原因はまだ特定していないが、仮説としては:

- v2-3/v2-4 系の速度ベース報酬（`reward_fwd_cube = Δx/dt`）は、cube に触れてもいないと一切信号が出ない（接触するまで完全に 0）。reach_bonus だけでは Follower 側の探索シグナル不足（5節・進捗.md 既知の問題）を補えていない可能性がある
- v5-2/v5-3 系の目標座標報酬は PBRS 化されていてもなお `reward_fwd_cube` の構造上、reach_bonus と相性が良い可能性がある（要検証）

**次の確認ポイント**: epoch300〜500 で再度動画確認。v5-3 の arm が実際に cube に届く長さまで伸びるか、v2-4 が逆方向のまま停滞し続けるかを見る。v2-4 が改善しない場合、速度ベース報酬系列そのものの探索シグナル不足を別途対策する必要がある。
